# 25. Definition of Done - правило готовности этапа

Версия: v1.0  
Статус: рабочий стандарт проекта  
Проект: New Era AI Platform

---

## 1. Назначение документа

Definition of Done - это правило, по которому определяется, можно ли считать задачу, этап, багфикс, документ, миграцию или новую функцию завершёнными.

Codex не должен писать "готово", если выполнена только часть работы. Задача считается завершённой только после прохождения обязательных проверок из этого документа.

Документ применяется ко всем частям проекта:

- MVP;
- Prompt Arena;
- Code Arena;
- Multi Model Battle;
- AI Team Mode;
- Judge Mode;
- Leaderboard;
- API;
- Supabase PostgreSQL;
- OpenRouter-интеграция;
- UI;
- документация;
- деплой.

---

## 2. Главное правило

Готово означает не "код написан", а "код написан, проверен, не ломает старое, задокументирован и зафиксирован в Git".

Codex не имеет права считать задачу завершённой только по визуальной оценке или по факту изменения файлов.

Перед отчётом Codex должен:

- выполнить проверки, которые применимы к задаче;
- честно указать проверки, которые не запускались;
- объяснить причину, если проверка не выполнена;
- указать, что нужно проверить человеку вручную.

Сначала проверки - потом отчёт. Нельзя писать финальный отчёт до проверки результата.

---

## 3. Уровни готовности

Чтобы DoD не превращался в лишнюю бюрократию, для проекта используются три уровня.

### 3.1. Базовый DoD

Применяется к маленьким изменениям:

- правка текста;
- небольшая правка UI;
- изменение документации;
- безопасный багфикс без изменения API и БД.

Минимум:

- TypeScript проверен, если затронут код;
- lint проверен, если затронут код;
- build проверен, если изменение может повлиять на сборку;
- документация обновлена, если изменилась логика;
- commit создан;
- краткий отчёт написан.

### 3.2. Расширенный DoD

Применяется, если затронуты:

- API;
- база данных;
- OpenRouter;
- Supabase;
- авторизация;
- RLS;
- сохранение пользовательских данных;
- важные пользовательские сценарии.

Дополнительно нужны:

- smoke-проверка;
- проверка ошибок;
- проверка безопасности;
- проверка миграций, если затронута БД;
- тесты, если затронута критическая логика.

### 3.3. Полный DoD

Применяется к большим функциям:

- новый режим проекта;
- крупный экран интерфейса;
- новая таблица БД;
- новая AI-логика;
- production-деплой;
- изменение архитектуры.

Дополнительно нужны:

- Pull Request;
- human review;
- план отката;
- расширенный отчёт;
- проверка production/staging, если был деплой;
- обновление документации и changelog.

## 4. Базовые команды проверки

Команды нужно запускать из корня проекта.

```bash
npm run typecheck # проверяет TypeScript без сборки проекта
npm run lint # проверяет код через ESLint
npm run build # проверяет production-сборку Next.js
npm test # запускает автоматические тесты, если они настроены
npm run smoke # запускает быстрые smoke-проверки, если они настроены
```

Если команда отсутствует в `package.json`, Codex должен написать это в отчёте и не делать вид, что проверка прошла.

## 5. Обязательный чек-лист для любой задачи

Перед завершением любой задачи Codex должен проверить:

- [ ] Изменения действительно решают поставленную задачу.
- [ ] Не добавлена лишняя функциональность сверх задачи.
- [ ] Старые рабочие функции не сломаны.
- [ ] Код соответствует текущей архитектуре проекта.
- [ ] Нет хаотичных временных решений.
- [ ] Нет лишних `console.log`, `debugger`, `alert`, тестовых UI-блоков и мусорного кода.
- [ ] Нет секретных ключей, токенов, паролей и приватных URL в коде.
- [ ] `.env.local` не попал в Git.
- [ ] Новые переменные окружения описаны в `.env.example` или документации.
- [ ] TypeScript проверен или указана причина, почему не проверен.
- [ ] lint проверен или указана причина, почему не проверен.
- [ ] build проверен или указана причина, почему не проверен.
- [ ] Документация обновлена, если изменилась логика, API, БД или поведение проекта.
- [ ] Commit создан с понятным названием.
- [ ] Написан короткий отчёт о выполненной работе.

## 6. Smoke-проверка

Smoke-проверка - это быстрый минимальный тест основного сценария после изменения.

Примеры smoke-проверок:

- открыть главную страницу;
- отправить prompt;
- получить ответ от одной или нескольких моделей;
- проверить, что ошибка API отображается пользователю понятным текстом;
- проверить, что страница не падает при пустом ответе;
- проверить, что в консоли браузера нет критических ошибок.

Smoke-проверка не заменяет полноценные тесты, но помогает быстро понять, что основная функция не сломана.

## 7. Definition of Done для тестирования

Если задача затрагивает критическую бизнес-логику, API, БД, AI-интеграции или сохранение данных, Codex должен добавить или обновить тесты.

Минимальные требования:

- [ ] Unit-тесты добавлены или обновлены для новой критической логики.
- [ ] Integration-тесты добавлены или обновлены для важных API-route или server actions.
- [ ] E2E-тест добавлен или обновлён для ключевого пользовательского сценария, если задача крупная.
- [ ] `npm test` проходит локально или указана причина, почему тесты не запускались.
- [ ] Если тесты не нужны, Codex объяснил почему.

Порог покрытия не вводится жёстко на этапе MVP. Для critical path целевой ориентир - не ниже 80%, но обязательный порог можно включить позже, когда тестовая база проекта станет стабильной.

Формат в отчёте:

```markdown
- Unit: пройдено / не требовалось / не запускалось
- Integration: пройдено / не требовалось / не запускалось
- E2E: пройдено / не требовалось / не запускалось
- Coverage: X% / не измерялось
```

## 8. Definition of Done для автоматических проверок

Codex не должен заменять автоматические проверки собственными словами.

Для проекта желательно постепенно подключить:

- pre-commit hooks;
- lint-staged;
- GitHub Actions;
- Vercel preview checks;
- автоматический запуск тестов на Pull Request.

Минимальная цель CI:

```bash
npm run typecheck # проверяет TypeScript
npm run lint # проверяет стиль и ошибки кода
npm run build # проверяет production-сборку
npm test # запускает тесты
```

Правило:

- если pipeline зелёный - можно рассматривать PR дальше;
- если pipeline красный - задача не считается готовой;
- если CI ещё не настроен - Codex обязан запускать проверки локально и писать результат в отчёте.

## 9. Definition of Done для Frontend

Если задача затрагивает интерфейс, нужно проверить:

- [ ] Страница открывается без ошибок.
- [ ] Основной пользовательский сценарий работает.
- [ ] Нет визуально сломанных блоков.
- [ ] Нет критических ошибок в консоли браузера.
- [ ] Состояние загрузки обработано.
- [ ] Состояние ошибки обработано.
- [ ] Состояние пустых данных обработано.
- [ ] Кнопки, формы и поля работают ожидаемо.
- [ ] UI не ломается при длинном тексте.
- [ ] UI не ломается при пустом ответе API.
- [ ] UI не показывает пользователю технические debug-данные.
- [ ] Интерфейс не ломается на мобильных разрешениях от 320px.
- [ ] Тач-элементы достаточно крупные для мобильного использования.
- [ ] Скролл работает корректно.

## 10. Definition of Done для API

Если задача затрагивает API-route, server action или backend-логику, нужно проверить:

- [ ] Endpoint отвечает корректно.
- [ ] Метод запроса соответствует задаче: GET, POST, PATCH, DELETE.
- [ ] Все входные данные валидируются через Zod или другую утверждённую схему.
- [ ] Некорректные данные обрабатываются безопасно.
- [ ] Успешный сценарий проверен.
- [ ] Ошибочный сценарий проверен.
- [ ] Ошибки возвращаются в понятном формате.
- [ ] API не возвращает stack trace пользователю.
- [ ] API не раскрывает секреты, ключи, технические токены и приватные URL.
- [ ] Frontend корректно обрабатывает ответ API.
- [ ] Выполнена smoke-проверка API.
- [ ] Для публичных endpoint, которые вызывают OpenRouter или дорогие операции, предусмотрен rate limiting или понятный план его добавления.

## 11. Definition of Done для Supabase PostgreSQL

Если задача затрагивает таблицы, миграции, seed-данные, RLS или SQL-логику, нужно проверить:

- [ ] Миграция создана.
- [ ] Миграция применима без ошибок.
- [ ] Названия таблиц и колонок соответствуют документации проекта.
- [ ] Не создано дублирование сущностей.
- [ ] Foreign key настроены корректно.
- [ ] Индексы добавлены там, где они нужны.
- [ ] RLS-политики проверены.
- [ ] Пользователь не получает доступ к чужим данным.
- [ ] Чувствительные колонки не возвращаются в общих SELECT-запросах без необходимости.
- [ ] Seed-данные не содержат мусорных или битых строк.
- [ ] Миграция не удаляет существующие данные без явного предупреждения.
- [ ] После миграции старый код не ломается.
- [ ] Если используются сгенерированные типы Supabase, они обновлены.
- [ ] Перед опасной миграцией описан backup или план отката.
- [ ] Документация по БД обновлена.

## 12. Definition of Done для AI-интеграций

Если задача затрагивает OpenRouter, список моделей, отправку prompt, получение ответов или сравнение моделей, нужно проверить:

- [ ] API-ключ не попал в код.
- [ ] Модели берутся из правильного источника.
- [ ] Codex не придумал несуществующую модель, endpoint или параметр API.
- [ ] Ошибка отсутствия API-ключа обработана.
- [ ] Ошибка лимита или отсутствия кредитов обработана.
- [ ] Ошибка конкретной модели обработана.
- [ ] Таймаут запроса обработан понятным сообщением.
- [ ] Одна упавшая модель не ломает весь экран.
- [ ] Нет бесконечной загрузки.
- [ ] Ответы моделей сохраняются корректно, если это предусмотрено задачей.
- [ ] Пользователь видит понятное сообщение при ошибке.
- [ ] Поведение соответствует выбранному режиму проекта.
- [ ] Для формы отправки prompt предусмотрена защита от случайных повторных отправок.

## 13. Definition of Done для безопасности

Безопасность обязательна для всех изменений, которые затрагивают API, БД, AI, авторизацию или пользовательские данные.

Проверить:

- [ ] Секреты не добавлены в код.
- [ ] `.env.local` не отслеживается Git.
- [ ] Новые переменные окружения описаны в `.env.example` или документации.
- [ ] Входные данные API валидируются через Zod или утверждённую схему.
- [ ] Ошибки API не раскрывают stack trace.
- [ ] Публичные endpoint защищены от частых запросов, если они могут тратить деньги или ресурсы.
- [ ] RLS не открывает чужие данные.
- [ ] Новые зависимости проверены на необходимость.
- [ ] Новые зависимости не добавлены без согласования.
- [ ] `npm audit` запущен для крупных изменений или указана причина, почему не запускался.

Если ключ случайно попал в репозиторий, задача не считается готовой. Нужно выполнить ротацию ключа и удалить секрет из истории Git или создать отдельную задачу на очистку истории.

## 14. Definition of Done для Performance

Для MVP performance-проверки не должны блокировать маленькие задачи, но обязательны для крупных экранов, API и AI-сценариев.

Проверить:

- [ ] Нет очевидно лишних запросов к Supabase.
- [ ] Нет N+1 запросов там, где нужна одна выборка или join.
- [ ] Большие списки используют пагинацию или лимиты.
- [ ] Интерфейс не зависает при ожидании AI-ответа.
- [ ] Кнопка отправки prompt не запускает случайно несколько одинаковых запросов.
- [ ] Картинки оптимизированы через `next/image`, если применимо.
- [ ] Тяжёлые вычисления не выполняются на каждом рендере без необходимости.

Для будущих этапов можно добавить строгие пороги:

- Lighthouse Performance не ниже 85;
- Lighthouse Accessibility не ниже 95;
- контроль размера bundle;
- отдельный мониторинг P95 для обычных API.

Важно: AI-endpoint через OpenRouter может отвечать дольше обычного API, поэтому общий лимит 500ms нельзя применять к генерации ответов моделей.

## 15. Definition of Done для доступности

Если задача затрагивает UI, нужно проверить базовую доступность:

- [ ] Интерактивные элементы имеют понятный текст или `aria-label`.
- [ ] Поля форм имеют label или понятную подпись.
- [ ] Управление с клавиатуры не сломано.
- [ ] Focus state виден.
- [ ] Контраст текста достаточный для чтения.
- [ ] Ошибки формы понятны без чтения кода.
- [ ] Интерфейсом можно пользоваться на мобильном устройстве.

Axe-core и автоматические a11y-проверки можно добавить позже в CI.

## 16. Definition of Done для Code Review

Для важных изменений Codex не должен напрямую пушить в `main`.

Через Pull Request должны проходить:

- новые функции;
- изменения БД;
- изменения RLS;
- изменения OpenRouter-интеграции;
- изменения авторизации;
- изменения архитектуры;
- удаление файлов;
- добавление новых зависимостей;
- production-related изменения.

Минимум для PR:

- [ ] Есть понятное название PR.
- [ ] Есть описание, что сделано.
- [ ] Есть список изменённых файлов или зон.
- [ ] Есть инструкция, как проверить.
- [ ] Есть отчёт Codex по DoD.
- [ ] Есть human review или явное разрешение владельца проекта.
- [ ] Merge не выполняется без approval.

## 17. Definition of Done для production и отката

Если задача затрагивает production, деплой, переменные окружения, Supabase или Vercel, нужно проверить:

- [ ] Production build проходит.
- [ ] Все нужные переменные окружения добавлены в Vercel.
- [ ] `.env.local` используется только локально.
- [ ] Сайт открывается после деплоя.
- [ ] Основной сценарий MVP работает после деплоя.
- [ ] Ошибки деплоя проверены.
- [ ] Нет зависимости от локальных файлов, которых нет на Vercel.
- [ ] Supabase подключён к нужному окружению.
- [ ] Для опасных изменений есть план отката.
- [ ] Перед опасной миграцией есть backup или понятный план восстановления.
- [ ] Для крупных новых функций предусмотрен feature flag или другой способ быстро отключить функцию.
- [ ] Ошибки логируются через доступный инструмент проекта.

Для MVP достаточно ручной проверки главной страницы и основного сценария. Для следующих этапов нужно добавить Sentry, Logflare или Vercel Analytics.

## 18. Definition of Done для документации

Если задача затрагивает документацию, нужно проверить:

- [ ] Документ имеет понятное название.
- [ ] Документ не противоречит другим файлам проекта.
- [ ] Термины используются одинаково во всех документах.
- [ ] Названия таблиц, API-route и сущностей совпадают с кодом.
- [ ] Нет устаревших решений, если они уже заменены.
- [ ] Добавлены только нужные изменения, без хаоса.
- [ ] Документ помогает Codex и человеку понять следующий шаг.
- [ ] Значимые изменения отражены в changelog, если это требуется.

## 19. Специальные правила для Codex

Codex - AI-агент, поэтому для него действуют дополнительные ограничения.

Codex должен:

- [ ] Перед началом задачи читать актуальную документацию проекта.
- [ ] Перед началом задачи учитывать этот DoD.
- [ ] Проверять, что каждый import реально существует.
- [ ] Проверять, что новый npm-пакет действительно добавлен в `package.json`.
- [ ] Не придумывать методы Supabase, OpenRouter, Next.js или других библиотек.
- [ ] Делать изменения маленькими логическими шагами.
- [ ] Писать commit так, чтобы изменение можно было откатить через `git revert`.
- [ ] Честно писать, если задача выполнена частично.
- [ ] Честно писать, если проверка не запускалась.

Codex не должен генерировать разный подход без причины, если задача уже имеет утверждённую архитектуру в документации проекта.

## 20. Codex запрещено

Codex не имеет права:

- [ ] Добавлять новые зависимости без согласования.
- [ ] Менять структуру папок без явной задачи.
- [ ] Удалять файлы без явной инструкции.
- [ ] Делать merge без approval.
- [ ] Деплоить production без явного подтверждения человека.
- [ ] Скрывать ошибки проверок.
- [ ] Писать "готово", если build не проверен и причина не указана.
- [ ] Писать "готово", если API изменён, но smoke-проверка не выполнена и причина не указана.
- [ ] Писать "готово", если БД изменена, но миграция не проверена и причина не указана.
- [ ] Придумывать несуществующие API, импорты, модели или методы.
- [ ] Вставлять секретные ключи в код, README, issue, PR или commit message.
- [ ] Оставлять debug-код.

## 21. Частичная готовность

Если задача выполнена не полностью, Codex обязан прямо написать:

```markdown
Задача выполнена частично.

Что сделано:
- ...

Что не сделано:
- ...

Причина:
- ...

Что нужно сделать дальше:
- ...

Какие проверки не выполнены:
- ...
```

Частичная готовность лучше, чем ложное "готово".

### Что сделано

- [x] ...

### Что НЕ сделано
- [ ] ...
- Причина: ...

## 23. Commit-правила

Каждый важный этап должен завершаться commit.

Commit должен быть понятным и отражать смысл изменения.

Хорошие примеры:

- `docs: add definition of done`
- `feat: add models api`
- `fix: handle openrouter errors`
- `db: add votes table`
- `ui: improve prompt arena page`
- `test: add prompt arena api tests`
- `chore: add ci checks`

Плохие примеры:

- `update`
- `fix`
- `test`
- `changes`
- `final`
- `123`

Если используется issue-трекер, желательно добавлять номер задачи в commit или PR description.

## 24. Специфические критерии для режимов проекта

### Prompt Arena

- [ ] Prompt отправляется корректно.
- [ ] Ответы моделей отображаются отдельно.
- [ ] Ошибка одной модели не ломает весь результат.
- [ ] История задачи сохраняется, если это предусмотрено этапом.

### Code Arena

- [ ] Кодовые ответы отображаются читаемо.
- [ ] Форматирование кода не ломается.
- [ ] Потенциально опасный код не выполняется автоматически.
- [ ] Есть предупреждение, если ответ модели требует ручной проверки.

### Multi Model Battle

- [ ] Все выбранные модели получают запрос.
- [ ] Модель с ошибкой отображается как ошибка, а не ломает общий экран.
- [ ] Результаты можно сравнить.
- [ ] Пользователь понимает, какая модель дала какой ответ.

### AI Team Mode

- [ ] Роли моделей отображаются понятно.
- [ ] Ответы разных ролей не смешиваются.
- [ ] Итоговый результат не скрывает индивидуальные ответы, если они нужны для анализа.

### Judge Mode

- [ ] Критерии оценки понятны.
- [ ] Оценка сохраняется, если это предусмотрено задачей.
- [ ] Судья не оценивает пустой или ошибочный ответ как полноценный.

### Leaderboard

- [ ] Голоса или оценки сохраняются корректно.
- [ ] Пользователь не может голосовать бесконечно, если это запрещено логикой этапа.
- [ ] Результаты не искажаются из-за ошибок одной модели.

## 25. Глоссарий

### Definition of Done

Правило, по которому задача считается завершённой.

### Smoke-проверка

Быстрый минимальный тест основного сценария, чтобы убедиться, что функция не сломана полностью.

### RLS

Row Level Security в Supabase PostgreSQL. Используется для ограничения доступа к строкам таблиц.

### Seed-данные

Начальные тестовые или системные данные, которые добавляются в базу.

### Critical path

Критически важный сценарий, без которого продукт теряет основной смысл. Для проекта это, например, отправка prompt и получение ответов моделей.

### Feature flag

Переключатель, который позволяет включить или выключить функцию без полного отката кода.

### Rollback

Откат изменения, если после релиза обнаружена ошибка.

## 26. Итоговое правило

Definition of Done важнее слов Codex.

Если Codex написал "готово", но чек-лист не выполнен, задача не считается завершённой.

Готово означает:

- работает;
- проверено;
- не ломает старое;
- безопасно;
- задокументировано;
- сохранено в Git;
- можно переходить к следующему этапу.
