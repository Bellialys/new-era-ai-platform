# 02 - План проекта

## Назначение файла

Этот файл описывает общий поэтапный план разработки проекта **Новая эпоха**.

Главная задача файла - показать логику движения от простой рабочей версии к большой AI-платформе. Подробный порядок версий и задач находится в файле `14-roadmap.md`. Если между этим файлом и `14-roadmap.md` появляется расхождение, главным считается `14-roadmap.md`.

## Главный принцип разработки

Проект разрабатывается без хаоса и без попытки сразу создать большую сложную систему.

Основные правила:

1. Сначала создаётся простая рабочая основа.
2. Потом добавляется минимальный MVP режима Prompt Arena.
3. После этого подключаются OpenRouter API, Supabase, голосование и история.
4. Только после стабильного Prompt Arena добавляются Code Arena, Multi Model Battle, Judge Mode и Leaderboard.
5. Сложные функции вроде Code Arena Runner и AI Team Mode добавляются только после стабильной базы, лимитов и контроля безопасности.
6. Каждый этап должен давать рабочий результат.
7. После каждого важного этапа нужно делать Git commit.
8. Новая функция не должна ломать уже работающую часть проекта.

## Что считается MVP

В проекте есть два уровня MVP.

### Первый UI MVP

Первый UI MVP - это версия, где интерфейс уже показывает основную идею проекта, но ещё может работать без реальных AI-запросов и базы данных.

В него входит:

- главная страница;
- поле ввода задачи;
- выбор нескольких моделей;
- кнопка отправки;
- карточки ответов моделей;
- тестовые mock-ответы;
- базовая структура будущего Prompt Arena.

Цель первого UI MVP - быстро увидеть, как будет выглядеть основная логика платформы.

### Рабочий MVP

Рабочий MVP - это стабильная версия Prompt Arena, где пользователь может отправить задачу нескольким AI-моделям и сравнить ответы.

В него входит:

- реальная отправка запроса через OpenRouter API;
- получение ответов от нескольких моделей;
- отображение ответов рядом;
- сохранение задачи и ответов в Supabase;
- голосование за лучший ответ;
- история запросов;
- базовая обработка ошибок;
- базовый деплой на Vercel.

Рабочий MVP завершается на версии `v1.0 Stable Prompt Arena`.

## Что не входит в первый MVP

В первый MVP не добавляем:

- регистрацию пользователей;
- оплату;
- сложный Leaderboard;
- AI Team Mode;
- запуск чужого кода;
- полноценный sandbox;
- сложную админ-панель;
- расширенную аналитику;
- командную работу моделей;
- автоматическое тестирование кода;
- сложные тарифы и подписки.

Эти функции добавляются позже, когда базовая платформа уже работает стабильно.

## Контрольные точки проверки

После каждого важного этапа нужно проверять основные сценарии.

### После создания Next.js

Проверить:

- сайт запускается локально;
- ошибок в терминале нет;
- главная страница открывается;

### После UI MVP

Проверить:

- поле ввода работает;
- модели можно выбрать;
- кнопка отправки реагирует;
- карточки ответов отображаются;
- интерфейс не ломается на мобильном экране.

### После подключения OpenRouter

Проверить:

- API-ключ хранится только в `.env.local`;
- ключ не попал в GitHub;
- запрос отправляется с backend-стороны;
- ответ приходит;
- ошибка модели показывается понятно;
- дорогие модели не используются без необходимости.

### После подключения Supabase

Проверить:

- подключение работает;
- данные сохраняются;
- данные можно прочитать;
- таблицы соответствуют `08-database.md`;
- лишние данные не открыты публично;
- RLS не создаёт случайную дыру в безопасности.

### После деплоя

Проверить:

- сайт открывается по публичной ссылке;
- переменные окружения добавлены в Vercel;
- OpenRouter работает в production;
- Supabase работает в production;
- нет ошибок сборки;
- основные сценарии работают после обновления страницы.

### Перед добавлением нового режима

Проверить:

- старый Prompt Arena работает;
- история не сломалась;
- голосование не сломалось;
- база данных не потеряла совместимость;
- API routes не конфликтуют;
- новые изменения зафиксированы отдельным commit.

## Риски проекта

### Слишком сложный старт

Если сразу делать все режимы, проект станет слишком большим и запутается.

Решение: сначала делаем UI MVP и Stable Prompt Arena.

### Ошибки с API-ключами

Нельзя вставлять секретные ключи прямо в код.

Решение: хранить ключи только в `.env.local` и Vercel Environment Variables.

### Большие расходы на AI API

Если не поставить лимиты, запросы к моделям могут стать дорогими.

Решение: добавить лимиты количества моделей, лимиты длины prompt, логирование и контроль дорогих моделей.

### Сложная база данных

Если сразу сделать много таблиц, можно запутаться и сломать MVP.

Решение: сначала использовать минимальные таблицы `models`, `tasks`, `model_responses`, `votes`, а остальные добавлять позже.

### Ранний запуск чужого кода

Code Arena Runner может создать серьёзные риски безопасности.

Решение: сначала сделать Code Arena Lite без запуска кода, а Runner добавить только после аккаунтов, лимитов, логирования и sandbox.

### Поломка старых функций

Новая функция может сломать старую.

Решение: перед добавлением каждого нового режима проверять основные сценарии Prompt Arena.
