# 15 - Changelog проекта

## Назначение файла

Этот файл хранит историю важных изменений проекта **Новая эпоха**.

Changelog нужен, чтобы быстро понимать:

- что уже изменено;
- что добавлено;
- что исправлено;
- что удалено;
- какие решения повлияли на проект;
- какие ограничения ещё остаются;
- какая версия проекта сейчас готовится.

Changelog не заменяет Git history. Git показывает технические изменения в файлах, а changelog объясняет смысл этих изменений.

## Главное правило

`14-roadmap.md` является главным источником порядка версий.

`15-changelog.md` не должен придумывать другой порядок разработки. Он только фиксирует, что уже сделано, что изменено и какие версии готовятся по roadmap.

```text
git commit
# фиксирует техническое изменение в репозитории

changelog
# объясняет смысл изменения для проекта
```

## Важное исправление порядка

Code Arena Runner должен быть только на `v1.7`.

Нельзя указывать Code Arena Runner как `v1.5`, потому что до безопасного запуска кода проекту нужны:

- аккаунты;
- лимиты;
- контроль расходов;
- админ-панель;
- feature flags;
- логирование;
- security review;
- отдельная sandbox-архитектура.

Правильный порядок:

```text
v1.1 - Code Arena Lite
# модели пишут и объясняют код, но код не запускается

v1.5 - Accounts and Profiles
# появляются пользователи, история, лимиты и доступы

v1.6 - Admin Panel and Limits
# появляется управление моделями, режимами и расходами

v1.7 - Code Arena Runner
# только теперь можно добавлять sandbox-запуск кода
```

## Категории изменений

### Added

Используется для новых функций, файлов, страниц, таблиц, API routes и возможностей.

Примеры:

```text
Добавлена страница /arena.
# новая UI-страница

Добавлен endpoint /api/compare.
# новый backend endpoint

Добавлена таблица tasks.
# новая таблица базы данных
```

### Changed

Используется для изменения существующей логики, структуры или правил.

Примеры:

```text
Изменён порядок версий по roadmap.
# проект стал более последовательным

Обновлена схема таблицы models.
# база синхронизирована с AI-моделями
```

### Fixed

Используется для исправления ошибок и противоречий.

Примеры:

```text
Исправлена версия Code Arena Runner с v1.5 на v1.7.
# устранён конфликт между документами

Исправлена обработка пустого prompt.
# пользовательская ошибка больше не ломает запрос
```

### Removed

Используется для удаления устаревших решений, полей или функций.

Примеры:

```text
Удалены поля is_free и is_paid из модели данных.
# заменены на price_label

Удалён запуск кода из ранней Code Arena.
# Runner перенесён на v1.7
```

### Security

Используется для изменений, связанных с безопасностью, ключами, лимитами, RLS, sandbox и защитой расходов.

Примеры:

```text
OpenRouter API key используется только server-side.
# ключ не попадает в браузер

Добавлена проверка allowlist моделей.
# пользователь не может вызвать любую модель напрямую

Code Runner запрещён до v1.7.
# защита от небезопасного запуска чужого кода
```

### Known Issues

Используется для честного описания известных ограничений.

Примеры:

```text
История гостя может быть ограниченной до появления аккаунтов.
# полноценная приватная история требует Supabase Auth

Leaderboard пока не имеет достаточного количества данных.
# нужны votes и judge results
```

# Правила обновления changelog

## Когда обновлять

Changelog нужно обновлять:

```text
после добавления новой функции;
# Added

после изменения логики;
# Changed

после исправления ошибки;
# Fixed

после удаления функции;
# Removed

после улучшения безопасности;
# Security

перед release;
# версия должна иметь итоговое описание
```

## Когда можно не обновлять

Changelog можно не обновлять для мелких внутренних изменений:

```text
исправлена опечатка в комментарии;
# не влияет на проект

переставлен пробел;
# незначительное изменение

локальная временная правка без commit;
# не попала в историю
```

Но если изменение влияет на пользователя, архитектуру, API, безопасность, данные или roadmap - его лучше записать.

# Связь changelog с Git commit

## Пример commit

```bash
git status
# показывает изменённые файлы

git add docs/15-changelog.md
# добавляет changelog в индекс Git

git commit -m "Update project changelog"
# фиксирует обновление changelog проекта
```

## Пример записи changelog

```text
## Fixed

- Исправлена версия Code Arena Runner с v1.5 на v1.7.
  # устранён конфликт между roadmap, decisions и Code Arena spec
```

## Правило

```text
Один commit может иметь одну или несколько записей changelog.
# если изменения важные

Одна версия changelog может включать много commit.
# release summary
```

# Release checklist

Перед созданием новой версии нужно проверить:

```text
все важные изменения внесены в changelog;
# история актуальна

версия имеет статус и дату;
# понятно, выпущена она или ещё планируется

Known Issues описаны;
# ограничения честно указаны

Security изменения выделены;
# безопасность не теряется

Git commit сделан;
# состояние сохранено

npm run build проходит;
# если уже есть код

production проверен;
# если был deploy
```

# Итог

`15-changelog.md` должен быть главным журналом изменений проекта.

Он помогает:

```text
понимать историю проекта;
# что и зачем менялось

готовить release;
# версии становятся понятными

фиксировать безопасность;
# важные security изменения не теряются

видеть ограничения;
# Known Issues

связывать документацию и код;
# changelog объясняет смысл commit
```

Главное правило:

**после каждого важного этапа обновлять changelog и делать Git commit.**
