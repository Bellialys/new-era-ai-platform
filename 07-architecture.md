# 07 - Архитектура проекта

## Назначение файла

Этот файл описывает архитектуру проекта **Новая эпоха**.

Архитектура нужна, чтобы проект развивался поэтапно, без хаоса и без конфликтов между файлами документации.

Главные задачи файла:

- показать, из каких частей состоит система;
- объяснить связь между frontend, backend, OpenRouter, Supabase и Vercel;
- закрепить правило, что секретные ключи не попадают в браузер;
- показать архитектуру MVP и будущих режимов;
- объяснить, какие части проекта можно делать сейчас, а какие только позже;
- сохранить совместимость с `14-roadmap.md`, `08-database.md`, `09-api-structure.md` и `12-security-and-env.md`.

## Краткое описание проекта

**Новая эпоха** - это AI-платформа для сравнения, оценки и совместной работы нескольких AI-моделей.

Пользователь вводит задачу, выбирает режим, после чего система отправляет запрос нескольким AI-моделям через OpenRouter. Ответы можно сравнивать, оценивать, сохранять и использовать в разных режимах.

Основные режимы проекта:

- Prompt Arena;
- Code Arena;
- Image Arena / Visual Arena;
- Multi Model Battle;
- Judge Mode;
- Leaderboard;
- AI Team Mode.

Для MVP главным режимом является:

```text
Prompt Arena
# первый основной рабочий режим проекта
```

## Главный архитектурный принцип

Frontend не должен напрямую обращаться к OpenRouter API.

Правильная схема:

```text
User Browser -> Next.js Frontend -> Next.js API Routes -> OpenRouter API
# OpenRouter API key остаётся только на сервере
```

Неправильная схема:

```text
User Browser -> OpenRouter API
# так делать нельзя, потому что секретный ключ может попасть в браузер
```

Все секреты должны храниться только здесь:

```text
.env.local
# локальные секреты на компьютере разработчика

Vercel Environment Variables
# production и preview секреты на Vercel
```

## Архитектурные слои

Проект делится на несколько слоёв.

```text
UI Layer
# страницы, формы, карточки, состояние загрузки, отображение ошибок
```

# 1. Frontend

Frontend - это часть проекта, которую видит пользователь.

В MVP frontend отвечает за:

```text
Ввод задачи.
# пользователь пишет prompt или задачу

Выбор моделей.
# пользователь выбирает 2-3 модели для сравнения

Отправку запроса.
# frontend вызывает свой backend route, а не OpenRouter напрямую

Показ загрузки.
# пользователь видит, что модели отвечают

Показ ответов.
# ответы отображаются в отдельных карточках

Голосование.
# пользователь выбирает лучший ответ

Просмотр истории.
# после v0.7 пользователь видит сохранённые сравнения
```

Frontend не должен:

```text
Хранить OPENROUTER_API_KEY.
# секретный ключ нельзя отдавать в браузер

Вызывать OpenRouter напрямую.
# все AI-запросы идут через backend

Доверять только скрытым кнопкам.
# пользователь может обойти UI, поэтому проверка нужна на сервере

Самостоятельно решать лимиты.
# лимиты должны проверяться на backend
```

# 2. Backend внутри Next.js

Backend в проекте - это серверные маршруты Next.js.

Backend отвечает за:

```text
Приём запросов от frontend.
# frontend отправляет данные только на свои API routes

Валидацию данных.
# проверка prompt, model IDs, режима и длины запроса

Проверку лимитов.
# защита от лишних расходов и спама

Работу с OpenRouter.
# backend отправляет запросы к AI-моделям

Работу с Supabase.
# backend сохраняет задачи, ответы, голоса и историю

Обработку ошибок.
# frontend получает понятный JSON, а не сырой технический сбой
```

## Главные backend routes для MVP

Подробно маршруты описаны в `09-api-structure.md`.

```text
GET /api/models
# вернуть список доступных моделей

POST /api/compare
# отправить задачу нескольким моделям и вернуть ответы

POST /api/vote
# сохранить голос пользователя за лучший ответ

GET /api/history
# вернуть историю сравнений

GET /api/history/[taskId]
# вернуть одно сохранённое сравнение
```

## Что backend не должен делать в MVP

```text
Запускать пользовательский код.
# это запрещено до Code Arena Runner v1.7

Открывать service role key на frontend.
# service role key только на сервере

Разрешать любые model ID из запроса пользователя.
# backend должен сверять model ID с allowlist

Игнорировать ошибки моделей.
# частичный успех должен обрабатываться корректно
```

# 3. Service Layer

Service Layer - это внутренняя логика backend. Она помогает не превращать API routes в хаос.

Рекомендуемые сервисы:

```text
openrouter service
# отправка запросов к OpenRouter и нормализация ответов

models service
# allowlist моделей, роли моделей, активность, порядок вывода

supabase service
# работа с таблицами Supabase

validation service
# проверка входных данных

rate limit service
# ограничения запросов и защита от лишних расходов

error service
# единый формат ошибок
```

## Пример логики `POST /api/compare`

```text
1. Frontend отправляет prompt и список model IDs.
# пользователь нажал кнопку сравнения

2. Backend проверяет prompt.
# запрос не должен быть пустым или слишком длинным

3. Backend проверяет выбранные модели.
# модель должна быть активна и разрешена в таблице models

4. Backend создаёт запись в tasks.
# задача сохраняется в Supabase

5. Backend отправляет запросы к OpenRouter.
# каждая модель получает один и тот же prompt

6. Backend сохраняет ответы в model_responses.
# успешные ответы и ошибки сохраняются отдельно

7. Backend возвращает результат frontend.
# пользователь видит карточки ответов
```

```text

# правильная схема вызова моделей
```

## Правила работы с моделями

```text
Использовать allowlist моделей.
# пользователь не должен отправлять произвольный model ID

Хранить модели в таблице models.
# frontend получает только разрешённые модели

Проверять model ID перед production.
# точные OpenRouter IDs могут меняться

Разделять модели по ролям.
# prompt, code, judge, team, dev

Ограничивать количество выбранных моделей.
# защита от лишних расходов
```

## Роли моделей

```text
prompt
# обычное сравнение ответов в Prompt Arena

code
# генерация и анализ кода в Code Arena Lite

judge
# оценка ответов в Judge Mode

team
# роль в AI Team Mode

dev
# дешёвые или бесплатные модели для разработки
```

# 8. Безопасность архитектуры

## Главные правила

```text
OpenRouter key только на сервере.
# ключ нельзя показывать пользователю

Service role key только на сервере.
# этот ключ обходит RLS и опасен на frontend

RLS включается для пользовательских таблиц.
# защита данных в Supabase

Все входные данные проверяются на backend.
# frontend-проверки недостаточно

Модели выбираются только из allowlist.
# пользователь не может вызвать любую дорогую модель

Количество моделей ограничено.
# защита бюджета

Длина prompt ограничена.
# защита от слишком дорогих запросов

Runner запрещён до v1.7.
# запуск чужого кода требует отдельной sandbox-архитектуры
```

## Что нельзя делать

```text
Нельзя хранить API key в React component.
# код компонента уходит в браузер

Нельзя писать ключи в GitHub.
# секреты могут быть украдены

Нельзя запускать пользовательский код через обычный Next.js route.
# это опасно для сервера и данных

Нельзя доверять model ID из frontend без проверки.
# пользователь может подменить запрос

Нельзя отправлять service role key в браузер.
# это полный доступ к базе
```

## v1.1 - Code Arena Lite

Code Arena Lite сравнивает код, но не запускает его.

```text
Разрешено:
# генерировать код, объяснять код, сравнивать решения

Запрещено:
# запускать пользовательский код, выполнять тесты, использовать sandbox
```

Архитектурно это похоже на Prompt Arena, но с другим типом задачи:

```text

# модели генерируют или анализируют код без запуска
```

# 12. Поток данных Prompt Arena

## Без Supabase - v0.4

```text
1. Пользователь вводит задачу.
# frontend собирает prompt

2. Пользователь выбирает модели.
# frontend передаёт model IDs

3. Frontend вызывает POST /api/compare.
# запрос идёт в backend

4. Backend вызывает OpenRouter.
# ключ хранится на сервере

5. Backend возвращает ответы.
# frontend показывает карточки
```

## С Supabase - v0.5 и дальше

```text
1. Пользователь вводит задачу.
# frontend собирает prompt

2. Backend создаёт task.
# задача сохраняется в Supabase

3. Backend вызывает OpenRouter.
# модели генерируют ответы

4. Backend сохраняет model_responses.
# ответы сохраняются в базе

5. Frontend показывает результат.
# пользователь видит сравнение

6. Пользователь голосует.
# vote сохраняется в Supabase

7. Пользователь открывает history.
# frontend получает сохранённые tasks и responses
```

# 13. Обработка ошибок

Проект должен возвращать понятные ошибки.

## Пример ошибок

```text
VALIDATION_ERROR
# неправильные данные от пользователя

MODEL_NOT_ALLOWED
# выбранная модель не разрешена

OPENROUTER_ERROR
# ошибка внешнего AI-сервиса

SUPABASE_ERROR
# ошибка базы данных

RATE_LIMITED
# пользователь превысил лимит

PARTIAL_MODEL_FAILURE
# часть моделей ответила, часть нет
```

## Частичный успех

Если одна модель не ответила, но другие ответили, весь запрос не должен полностью ломаться.

```text
Модель A ответила успешно.
# показываем ответ

Модель B вернула ошибку.
# показываем понятное сообщение об ошибке в карточке этой модели

Модель C ответила успешно.
# показываем ответ
```

Это важно для стабильности Prompt Arena.

# 14. Контроль расходов

Так как OpenRouter может использовать платные модели, архитектура должна учитывать бюджет.

Минимальные меры для MVP:

```text
Ограничить длину prompt.
# меньше риск дорогого запроса

Ограничить количество моделей за один запуск.
# например 2-3 модели в MVP

Ограничить max tokens.
# backend задаёт лимит ответа

Использовать allowlist моделей.
# пользователь не может вызвать случайную дорогую модель

Логировать ошибки и расходы.
# хотя бы базово на backend
```

После аккаунтов можно добавить:

```text
Дневные лимиты пользователя.
# защита бюджета

Лимиты по ролям.
# разные возможности для обычного пользователя и админа

Admin Panel.
# управление моделями и лимитами без изменения кода
```

# 18. Критерии правильной архитектуры

Архитектура считается правильной, если:

```text
Frontend не знает секретные ключи.
# OpenRouter key и service role key не попадают в браузер

Все AI-запросы идут через backend.
# frontend вызывает только свои API routes

Модели выбираются из allowlist.
# нет произвольного доступа к дорогим моделям

Supabase используется постепенно.
# сначала MVP-таблицы, потом будущие таблицы

Prompt Arena работает до сложных режимов.
# сначала стабильный MVP

Runner не запускается до v1.7.
# безопасность важнее скорости

AI Team Mode не начинается до v2.0.
# сначала нужна стабильная база проекта

Каждый важный этап фиксируется Git commit.
# история проекта остаётся понятной
```

```text
Сначала документация.

Потом voting и history.
```

```text
# каждый этап должен давать рабочий результат
```

# Актуализация архитектуры v0.5

Текущий backend слой находится здесь:

```text
src/app/api/models/route.ts
# отдаёт список разрешённых моделей

src/app/api/compare/route.ts
# принимает prompt, modelIds и modeSlug

src/lib/server/models.ts
# hardcoded fallback allowlist моделей

src/lib/server/model-catalog.ts
# Supabase-first catalog и fallback resolution

src/lib/server/openrouter.ts
# вызовы OpenRouter API

src/lib/server/arena-persistence.ts
# best-effort сохранение Prompt Arena в Supabase

src/lib/server/supabase.ts
# server-only Supabase client

src/lib/server/utils.ts
# ApiError, validation, safe logging
```

Текущий поток данных:

```text
Browser /arena
# пользователь вводит prompt и выбирает модели

GET /api/models
# frontend получает Supabase models.id или fallback selection ids

POST /api/compare
# frontend отправляет prompt, modelIds, modeSlug

Backend validation and model catalog
# проверка prompt, modelIds, modeSlug; resolution selectionId -> model_key

OpenRouter API
# backend получает ответы моделей

Supabase PostgreSQL
# backend best-effort сохраняет task и model_responses

Browser /arena
# frontend показывает карточки ответов и taskId, если сохранение прошло
```

Текущий fallback:

```text
Если Supabase не настроен или models query недоступен, Prompt Arena использует hardcoded allowlist.
# это нужно для устойчивости MVP и локальной разработки
```

# Будущий поток Image Arena / Visual Arena

Image Arena не входит в первый MVP и не должна реализовываться до стабильной Prompt Arena.

Будущий поток данных:

```text
Пользовательская визуальная идея
# один prompt/idea для генерации изображения

Next.js frontend
# страница /image-arena отправляет идею и выбранные image-модели только в backend

Next.js API route
# будущий POST /api/image-arena/generate валидирует идею, модели, лимиты и пользователя

OpenRouter image-capable models
# backend вызывает только модели с image output capability

Supabase Storage
# бинарные файлы изображений сохраняются в Storage bucket

Supabase PostgreSQL metadata
# в базе хранится metadata, model_id, task_id, storage_path, размеры, статус и ошибки

UI сравнения
# пользователь видит сетку изображений и выбирает победителя
```

Ключевое правило: PostgreSQL не должен хранить бинарные изображения; в базе хранится только metadata и путь к файлу в Supabase Storage.
