# 14 - Roadmap проекта

## Назначение файла

Этот файл является главным источником порядка разработки проекта **Новая эпоха**.

Если в других файлах есть расхождение по версиям, приоритет имеет этот файл.

Roadmap нужен, чтобы проект развивался поэтапно:

- сначала рабочий простой MVP;
- потом стабильная Prompt Arena;
- потом дополнительные режимы;
- только после этого сложные функции вроде Code Arena Runner и AI Team Mode.

Главное правило: **одна версия - один понятный рабочий результат**.

---

## Главный принцип разработки

Проект нельзя делать как один большой сложный продукт сразу.

Правильный подход:

```text
Сначала рабочий минимум.
# проект должен запускаться и показывать основной сценарий

Потом подключение реальных AI-моделей.
# ответы должны приходить через backend, а не из браузера

Потом база данных.
# задачи, ответы и голоса должны сохраняться

Потом стабильность.
# ошибки, лимиты, деплой и проверка безопасности

Потом новые режимы.
# Code Arena, Battle, Judge, Leaderboard

Потом аккаунты и админ-контроль.
# лимиты, история пользователя, управление моделями

Только потом запуск чужого кода и Team Mode.
# это самые сложные и рискованные части проекта
```

## Что считается MVP

В проекте есть два уровня MVP.

### UI MVP

UI MVP - это версия `v0.3`.

Она нужна, чтобы увидеть интерфейс и проверить основной пользовательский сценарий без реального AI.

Входит:

- главная страница;
- форма ввода задачи;
- выбор нескольких моделей;
- кнопка запуска сравнения;
- карточки ответов;
- mock-ответы вместо реального OpenRouter;
- базовое состояние loading/error/empty.

Не входит:

- OpenRouter;
- Supabase;
- голосование;
- история;
- авторизация;
- Runner;
- Team Mode.

### Stable Prompt Arena MVP

Настоящий рабочий MVP - это версия `v1.0`.

Входит:

- реальные ответы моделей через OpenRouter;
- сохранение задач и ответов в Supabase;
- голосование за лучший ответ;
- история сравнений;
- деплой на Vercel;
- базовые лимиты и обработка ошибок;
- безопасное хранение ключей;
- стабильная Prompt Arena.

Не входит:

- Code Arena Runner;
- AI Team Mode;
- полноценная админ-панель;
- платёжная система;
- сложная аналитика;
- автоматический запуск пользовательского кода.

# v0.4 - OpenRouter Integration

## Цель

Подключить реальные AI-модели через OpenRouter, но только через backend.

## Что входит

- переменная `OPENROUTER_API_KEY` в `.env.local`;
- API route `POST /api/compare`;
- server-side запросы к OpenRouter;
- allowlist моделей;
- обработка ошибок моделей;
- timeout;
- частичный успех, если одна модель ответила, а другая нет.

## Важные правила безопасности

```text
OPENROUTER_API_KEY хранится только в .env.local.
# ключ нельзя вставлять в frontend

Ключ нельзя называть NEXT_PUBLIC_OPENROUTER_API_KEY.
# всё с NEXT_PUBLIC попадёт в браузер

Запросы к OpenRouter идут только через /api/compare.
# браузер не должен обращаться к OpenRouter напрямую

Модели выбираются только из server-side allowlist.
# пользователь не должен отправлять произвольный model id
```

## Минимальный API

```text
POST /api/compare
# принимает prompt и список modelIds

GET /api/models
# возвращает разрешённые модели для интерфейса
```

## Критерии готовности

- пользователь получает реальные ответы от минимум 2 моделей;
- ключ OpenRouter не виден в браузере;
- неизвестная модель отклоняется;
- ошибка одной модели не ломает весь результат;
- есть понятное сообщение об ошибке.

## Git commit

```bash

# добавить backend-интеграцию OpenRouter

git commit -m "feat: integrate openrouter compare api"
# сохранить подключение OpenRouter
```

# v1.2 - Multi Model Battle

## Цель

Сделать режим формального боя моделей по одной задаче.

## Что входит

- страница `/battle`;
- выбор моделей-участников;
- единый prompt;
- ответы всех моделей;
- сравнение результатов;
- выбор победителя пользователем;
- сохранение battle result.

## Отличие от Prompt Arena

Prompt Arena - это обычное сравнение ответов.

Multi Model Battle - это более формализованный режим соревнования с понятными правилами, участниками и победителем.

## Критерии готовности

- можно создать battle;
- можно выбрать 2 или больше моделей;
- ответы отображаются рядом;
- победитель сохраняется.

# v1.3 - Judge Mode

## Цель

Добавить модель-судью, которая оценивает ответы других моделей.

## Что входит

- страница `/judge`;
- выбор моделей-участников;
- выбор judge-модели;
- критерии оценки;
- генерация оценок;
- объяснение решения судьи;
- сохранение judge result.

## Важные правила

```text
Judge model не должна оценивать свой же ответ.
# иначе оценка может быть нечестной

Критерии оценки должны быть видны пользователю.
# оценка не должна быть "магической"

Оценка судьи не заменяет голос пользователя.
# это отдельный сигнал качества
```

## Критерии готовности

- judge-модель получает ответы участников;
- оценка имеет понятные критерии;
- пользователь видит объяснение;
- результат сохраняется.

# v1.4 - Leaderboard

## Цель

Сделать рейтинг моделей по результатам голосования и Judge Mode.

## Что входит

- страница `/leaderboard`;
- рейтинг моделей;
- фильтр по режимам;
- количество участий;
- количество побед;
- win rate;
- средняя judge-оценка, если есть;
- защита от вывода рейтинга по слишком малому числу данных.

## Важное правило

Leaderboard нельзя считать объективной истиной.

Он показывает статистику внутри проекта, а не абсолютное качество моделей во всём мире.

## Критерии готовности

- рейтинг строится из сохранённых данных;
- модели с малым числом участий помечаются осторожно;
- есть фильтр по режимам;
- данные не считаются вручную в коде.

# v1.6 - Admin Panel and Limits

## Цель

Добавить управление моделями, режимами, лимитами и стоимостью.

## Что входит

- страница `/admin`;
- включение/отключение моделей;
- управление `price_label`;
- управление `role_tags`;
- лимиты по пользователям;
- лимиты по моделям;
- просмотр ошибок;
- базовая статистика расходов;
- подготовка к Code Arena Runner.

## Почему это нужно до Runner

Runner требует строгого контроля:

- кто может запускать код;
- сколько запусков можно делать;
- какие языки разрешены;
- сколько времени работает код;
- сколько памяти доступно;
- как логируются ошибки;
- как отключить опасную функцию.

## Критерии готовности

- администратор может отключить модель;
- администратор может ограничить дорогие модели;
- лимиты применяются на backend;
- обычный пользователь не может попасть в `/admin`;
- есть базовый audit trail для важных действий.

# v1.7 - Code Arena Runner

## Цель

Добавить безопасный запуск кода и проверку тест-кейсов.

## Это опасная версия

Code Arena Runner нельзя делать раньше `v1.7`.

Причина: запуск чужого кода - это риск безопасности, расходов и стабильности.

## Что входит

- sandbox execution;
- ограничение времени выполнения;
- ограничение памяти;
- список разрешённых языков;
- тест-кейсы;
- сохранение результатов запуска;
- отдельная таблица `code_results`;
- логирование ошибок;
- отключение Runner через admin panel.

## Что нужно проверить перед стартом v1.7

```text
Есть аккаунты.
# нужно понимать, кто запускает код

Есть лимиты.
# нельзя дать бесконечные запуски

Есть админ-панель.
# Runner нужно уметь отключить

Есть логирование.
# ошибки и злоупотребления должны фиксироваться

Есть отдельный security review.
# запуск кода нельзя добавлять как обычную функцию
```

## Что запрещено

```text
Запускать код в обычном Next.js API route без sandbox.
# это опасно

Давать доступ к файловой системе сервера.
# код пользователя не должен видеть окружение

Передавать секреты в среду выполнения кода.
# API keys не должны быть доступны пользовательскому коду

Давать сетевой доступ по умолчанию.
# пользовательский код не должен ходить куда угодно
```

## Критерии готовности

- код запускается только в изолированной среде;
- timeout работает;
- memory limit работает;
- секреты недоступны коду;
- результаты тестов сохраняются;
- Runner можно выключить администратором;
- есть fallback, если Runner недоступен.

# v2.0 - AI Team Mode

## Цель

Добавить режим командной работы нескольких AI-моделей с ролями.

## Что входит

- страница `/team`;
- роли моделей;
- несколько шагов рассуждения;
- team run;
- team steps;
- итоговый ответ;
- возможность сравнить вклад участников;
- сохранение результата.

## Возможные роли

```text
Planner
# разбивает задачу на шаги

Developer
# предлагает техническое решение

Critic
# ищет слабые места

Security Reviewer
# проверяет риски безопасности

Judge
# оценивает варианты

Summarizer
# собирает финальный ответ
```

## Почему Team Mode только v2.0

AI Team Mode требует:

- стабильной Prompt Arena;
- нормальной базы данных;
- моделей с разными ролями;
- контроля расходов;
- логирования шагов;
- ограничений по пользователям;
- хорошего UI для многошагового процесса;
- защиты от бесконечных циклов.

Это слишком сложно для MVP.

## Критерии готовности

- можно создать team run;
- каждая модель выполняет свою роль;
- шаги сохраняются;
- пользователь видит процесс и финальный ответ;
- есть лимит количества шагов;
- Team Mode можно выключить через настройки проекта.

## Итоговая логика развития

Проект развивается так:

```text
Сначала документация.
# понимаем, что строим

Потом базовый Next.js.
# получаем рабочую основу

Потом Prompt Arena UI.
# видим будущий продукт

Потом OpenRouter.
# получаем реальные ответы моделей

Потом Supabase.
# сохраняем данные

Потом голосование и история.
# появляется ценность сравнения

Потом деплой и стабилизация.
# проект можно показать и использовать

Потом Stable Prompt Arena.
# первая настоящая версия

Потом новые режимы.
# расширяем продукт без ломания основы

Потом аккаунты и админка.
# добавляем контроль и лимиты

Потом Runner.
# безопасно запускаем код

Потом Team Mode.
# строим сложную командную AI-систему
```

Главный ориентир: **каждый этап должен давать работающий результат, который можно проверить, сохранить в Git и развивать дальше**.
