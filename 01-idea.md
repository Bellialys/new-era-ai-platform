# 01 - Идея проекта

## Название проекта

**Новая эпоха** - AI-платформа для сравнения, оценки и совместной работы нескольких AI-моделей.

## Краткое описание

**Новая эпоха** - это веб-платформа, где пользователь вводит задачу, выбирает режим работы, а система отправляет запрос нескольким AI-моделям через backend.

После получения ответов пользователь может:

- сравнить ответы разных моделей рядом;
- выбрать лучший вариант;
- проголосовать за лучший ответ;
- сохранить историю сравнения;
- позже использовать отдельные режимы для кода, баттлов, оценки, рейтингов и командной работы моделей.

Главный смысл проекта - не просто получить один ответ от одной модели, а увидеть несколько вариантов и понять, какая модель лучше подходит под конкретную задачу.

## Главная идея

Обычный AI-чат работает по схеме:

```text
Пользователь -> одна AI-модель -> один ответ
```

Проект **Новая эпоха** работает по другой схеме:

```text
Пользователь -> несколько AI-моделей -> несколько ответов -> сравнение -> оценка -> лучший результат
```

Это превращает AI из одиночного помощника в систему сравнения, анализа и выбора лучшего ответа.

## Какую проблему решает проект

Пользователю часто сложно понять, какая AI-модель лучше подходит для конкретной задачи.

Например:

- одна модель лучше пишет код;
- другая модель лучше объясняет сложные темы;
- третья модель лучше анализирует документы;
- четвёртая модель лучше работает как критик или судья;
- пятая модель даёт более короткие и практичные ответы.

Без сравнения пользователь видит только один ответ и не понимает, был ли он лучшим.

Проект решает эту проблему через единый интерфейс, где несколько моделей отвечают на одну задачу, а пользователь может быстро сравнить качество ответов.

## Для кого создаётся проект

Платформа может быть полезна для:

- студентов;
- программистов;
- тестировщиков;
- авторов текстов;
- исследователей;
- команд разработки;
- людей, которые часто работают с AI;
- пользователей, которым нужно выбрать лучшую модель под конкретную задачу.

## Основная ценность проекта

Проект должен давать пользователю не просто генерацию текста, а инструмент выбора.

Ключевые ценности:

- сравнение моделей на одной задаче;
- прозрачность качества ответов;
- возможность голосовать за лучший вариант;
- накопление истории сравнений;
- будущий рейтинг моделей;
- будущая командная работа нескольких моделей с разными ролями.

## Основной сценарий MVP

Первый рабочий сценарий проекта должен быть простым.

Пользователь:

1. открывает страницу Prompt Arena;
2. вводит задачу или вопрос;
3. выбирает несколько AI-моделей;
4. нажимает кнопку запуска;
5. получает ответы моделей рядом;
6. сравнивает ответы;
7. выбирает лучший ответ;
8. сохраняет результат в истории.

Именно этот сценарий является основой MVP.

## Что является MVP

MVP проекта - это **Stable Prompt Arena**.

MVP не должен включать все идеи проекта сразу. Он должен доказать главную ценность: сравнение нескольких моделей на одной задаче.

MVP считается готовым, когда работает следующий минимум:

- есть главный экран Prompt Arena;
- пользователь может ввести запрос;
- пользователь может выбрать несколько моделей;
- backend отправляет запросы через OpenRouter API;
- ответы моделей отображаются рядом;
- можно выбрать лучший ответ;
- задача, ответы и голос сохраняются в Supabase;
- историю сравнений можно открыть повторно;
- проект можно развернуть на Vercel.

## Границы MVP

В MVP входит:

- Prompt Arena;
- OpenRouter API;
- Supabase PostgreSQL;
- базовая история;
- простое голосование;
- базовая защита ключей через backend;
- деплой на Vercel.

В MVP не входит:

- полноценная регистрация пользователей;
- платёжная система;
- публичные профили;
- полноценная админ-панель;
- запуск пользовательского кода;
- сложная аналитика;
- полноценный Leaderboard;
- AI Team Mode;
- автоматическая командная работа моделей.

## Режимы проекта

Проект развивается поэтапно. Режимы описывают итоговое направление, но не все они реализуются сразу.

### 1. Prompt Arena

Базовый режим проекта.

Пользователь вводит один запрос, выбирает несколько моделей и получает несколько ответов рядом.

Этот режим является основой MVP и реализуется первым.

### 2. Code Arena

Режим для задач по программированию.

Он делится на два этапа:

- **Code Arena Lite** - модели пишут, объясняют, исправляют и сравнивают код без запуска;
- **Code Arena Runner** - в будущем безопасно запускает код в sandbox и показывает результат тестов.

Code Arena Lite можно добавлять после Stable Prompt Arena.

Code Arena Runner нельзя добавлять рано, потому что запуск чужого кода требует отдельной безопасности, лимитов, логирования и изоляции.

### 3. Multi Model Battle

Режим соревнования моделей.

Несколько моделей отвечают на одну задачу, а пользователь или модель-судья выбирает лучший ответ.

Этот режим расширяет идею Prompt Arena и добавляет более явный формат баттла.

### 4. Judge Mode

Режим оценки ответов.

Одна модель или пользователь оценивает ответы по критериям:

- точность;
- полнота;
- логика;
- полезность;
- стиль;
- безопасность;
- качество кода.

Judge Mode нужен для более строгого сравнения моделей.

### 5. Leaderboard

Рейтинг моделей по результатам голосований, оценок и задач.

В первой версии Leaderboard должен быть простым. Позже его можно расширить категориями, фильтрами, типами задач и статистикой качества.

### 6. AI Team Mode

Самый сложный режим проекта.

В этом режиме несколько моделей работают как команда с ролями:

- Planner;
- Architect;
- Developer;
- Tester;
- Critic;
- Security Reviewer;
- Editor;
- Judge.

AI Team Mode не входит в MVP и должен появляться только после стабильной базы, истории, ролей, лимитов, логирования и опыта с предыдущими режимами.

## Будущий режим Image Arena / Visual Arena

Image Arena / Visual Arena - будущий визуальный режим проекта.

Идея режима: пользователь вводит одну визуальную идею, выбирает 2-3 image-capable модели, а backend отправляет запросы через OpenRouter только с server-side ключами. Разные image-модели генерируют изображения, система сохраняет файлы в Supabase Storage, записывает metadata и storage path в Supabase PostgreSQL, а пользователь сравнивает результаты в сетке и выбирает лучший вариант.

Этот режим не входит в первый MVP. Он откладывается до стабильной Prompt Arena, настроенного Storage, лимитов стоимости и safety-контролей.

Если другой файл противоречит этому порядку, главным считается `14-roadmap.md`.

## Чем проект отличается от обычного AI-чата

Обычный AI-чат:

- даёт ответ одной модели;
- не показывает альтернативы;
- не сравнивает качество;
- не сохраняет статистику сравнений;
- не формирует рейтинг моделей;
- не даёт использовать модели как команду.

Проект **Новая эпоха**:

- отправляет одну задачу нескольким моделям;
- показывает ответы рядом;
- помогает выбрать лучший ответ;
- сохраняет историю сравнений;
- позволяет голосовать;
- в будущем добавляет модель-судью;
- в будущем формирует Leaderboard;
- в будущем позволяет использовать модели как команду.

## Технологическая основа

Предварительно выбранный стек:

- **Next.js** - frontend и backend API routes;
- **TypeScript** - типизация проекта;
- **Vercel** - размещение сайта;
- **Supabase PostgreSQL** - база данных;
- **OpenRouter API** - подключение AI-моделей;
- **GitHub** - хранение кода;
- **Visual Studio Code** - редактор кода.

## Главные технические правила

1. Не вставлять секретные API-ключи в frontend.
2. Не хранить ключи в GitHub.
3. Использовать `.env.local` для локальной разработки.
4. Использовать Vercel Environment Variables для production.
5. Все запросы к OpenRouter выполнять только через backend.
6. Сначала делать простой рабочий MVP.
7. Не добавлять сложные режимы до стабильной Prompt Arena.
8. Не запускать пользовательский код до отдельной безопасной версии Runner.
9. Перед новой функцией проверять, что старая часть проекта не сломалась.
10. Важные этапы фиксировать через Git commit.

## Главный принцип разработки

Проект нужно делать без хаоса.

Сначала создаётся простой рабочий результат. Потом он стабилизируется. Только после этого добавляется следующая функция.

Правильный подход:

```text
Сделал маленькую функцию -> проверил -> закоммитил -> перешёл дальше
```

Неправильный подход:

```text
Сразу делать все режимы, аккаунты, оплату, Runner и Team Mode
```

## Итоговая цель

Итоговая цель проекта - создать платформу, где пользователь может сравнивать AI-модели, оценивать качество ответов, выбирать лучший результат и постепенно использовать несколько моделей как полноценную рабочую команду.

Проект должен развиваться от простого MVP к сильной AI-платформе, но каждый этап должен давать рабочий результат.
