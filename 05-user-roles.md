# 05 - User Roles

## Назначение файла

Этот файл описывает роли пользователей в проекте **Новая эпоха**.

Роли нужны, чтобы заранее понимать:

- кто может пользоваться платформой;
- какие функции доступны разным типам пользователей;
- что входит в MVP;
- что добавляется после MVP;
- какие ограничения нужны для защиты бюджета;
- как роли связаны с Supabase Auth, RLS и будущей админ-панелью;
- какие функции нельзя открывать раньше времени.

Главное правило: **роли не должны усложнять первый MVP**.

Сначала нужно сделать рабочую Prompt Arena. Потом можно добавлять аккаунты, профили, админ-панель, лимиты и расширенные права.

## Главный принцип ролей

В проекте используется постепенное развитие ролей.

Нельзя сразу строить сложную систему пользователей, если ещё нет стабильной Prompt Arena.

Правильная логика:

1. Сначала гость может открыть сайт и сравнить ответы моделей.
2. Потом появляется сохранение истории.
3. Потом добавляется авторизация.
4. Потом добавляются профили и личные лимиты.
5. Потом появляется админ-панель.
6. Потом открываются более сложные режимы с повышенным контролем.

## Минимальный набор для MVP

Для MVP не нужна полноценная система ролей.

В MVP достаточно трёх сущностей:

| Сущность | Назначение | Нужна в MVP |
| --- | --- | --- |
| Гость | Обычный пользователь без входа | Да |
| Система | Backend, который вызывает OpenRouter и пишет данные в Supabase | Да |
| Администратор-разработчик | Владелец проекта, который вручную меняет настройки | Да, но без отдельной админ-панели |

Полноценные аккаунты, роли и админ-панель добавляются позже.

## Роль Guest

Guest - это пользователь без регистрации.

Это основная роль для MVP, потому что платформа должна быстро показывать главную идею: сравнение ответов нескольких AI-моделей.

### Что Guest может делать в MVP

Guest может:

- открыть главную страницу;
- открыть Prompt Arena;
- ввести задачу;
- выбрать разрешённые модели;
- отправить запрос;
- получить ответы моделей;
- сравнить ответы рядом;
- выбрать лучший ответ;
- увидеть понятную ошибку, если модель не ответила;
- посмотреть ограниченную историю, если эта функция уже добавлена.

### Что Guest не может делать

Guest не может:

- видеть чужие приватные запросы;
- управлять моделями;
- менять настройки проекта;
- видеть API-ключи;
- использовать дорогие модели без лимитов;
- запускать чужой код;
- открывать админ-панель;
- управлять Leaderboard;
- обходить лимиты через frontend.

### Ограничения Guest

Для Guest нужны ограничения:

- максимум 2-3 модели за один запрос на раннем этапе;
- максимальная длина prompt;
- ограничение частоты запросов;
- ограничение дорогих моделей;
- защита от пустых и слишком больших запросов;
- запрет доступа к админским API;
- запрет на Code Arena Runner до `v1.7`.

## Роль User

User - это зарегистрированный пользователь.

Эта роль не нужна для первого MVP. Она появляется в версии `v1.5 - Accounts and Profiles`.

### Что User сможет делать

User сможет:

- входить в аккаунт;
- видеть свою историю;
- сохранять избранные ответы;
- получать персональные лимиты;
- использовать больше моделей за запрос;
- голосовать с защитой от накрутки;
- пользоваться будущими режимами с контролем доступа;
- работать с Code Arena Lite;
- использовать Judge Mode в рамках лимитов.

### Что User не должен делать

User не должен:

- видеть чужие приватные задачи;
- менять чужие ответы;
- управлять моделями;
- менять лимиты других пользователей;
- получать доступ к service role key;
- получать доступ к OpenRouter API key;
- запускать код без разрешённого Runner-механизма;
- менять системные настройки.

## Роль Moderator

Moderator нужен только тогда, когда появится публичный контент.

Например:

- публичные сравнения;
- публичный Leaderboard;
- комментарии;
- жалобы;
- спорные голосования;
- пользовательские публикации.

В MVP модератор не нужен.

### Что Moderator может делать

Moderator может:

- скрывать неподходящий публичный контент;
- проверять жалобы;
- отмечать спам;
- проверять спорные элементы Leaderboard;
- помогать поддерживать качество публичной части проекта.

### Что Moderator не может делать

Moderator не может:

- менять список моделей;
- управлять API-ключами;
- менять роли администраторов;
- менять системные лимиты;
- удалять базу данных;
- видеть секреты;
- выполнять действия владельца проекта.

## Роль Admin

Admin - это администратор платформы.

Полноценный Admin появляется в версии `v1.6 - Admin Panel and Limits`.

До этого администратор существует только как разработчик, который вручную меняет конфигурацию проекта.

### Что Admin может делать

Admin может:

- включать и отключать модели;
- менять `sort_order` моделей;
- менять `price_label` моделей;
- ограничивать дорогие модели;
- смотреть технические ошибки;
- смотреть статистику использования;
- управлять лимитами;
- блокировать проблемные функции;
- проверять работу OpenRouter;
- проверять работу Supabase;
- управлять настройками будущего Leaderboard.

### Что Admin не должен делать через интерфейс

Admin не должен видеть секретные ключи в интерфейсе.

Секреты должны храниться только здесь:

```text
.env.local
# локальные секреты разработчика

Vercel Environment Variables
# production-секреты на стороне Vercel
```

Даже если пользователь имеет роль Admin, backend не должен отдавать ему значения секретных переменных.

## Роль Owner

Owner - это владелец проекта с самым высоким уровнем прав.

Эта роль нужна позже, когда появятся:

- несколько администраторов;
- платные функции;
- сложные лимиты;
- публичные данные;
- разные уровни доступа.

### Что Owner может делать

Owner может:

- назначать администраторов;
- удалять администраторов;
- менять глобальные настройки;
- включать и отключать платные функции;
- управлять стратегией лимитов;
- принимать финальные решения по публичным данным;
- просматривать системную статистику.

### Отличие Owner от Admin

Admin управляет рабочими настройками платформы.

Owner управляет самой системой прав и глобальными решениями проекта.

## Роль Paid User

Paid User нужен только после проверки идеи и появления монетизации.

В MVP эта роль не нужна.

### Возможные преимущества Paid User

Paid User может получить:

- больше запросов в день;
- больше моделей за один запрос;
- доступ к дорогим моделям;
- расширенную историю;
- экспорт результатов;
- расширенный Judge Mode;
- расширенный AI Team Mode;
- приоритетную обработку запросов.

### Почему нельзя добавлять оплату рано

Оплата усложняет проект.

Для оплаты нужны:

- аккаунты;
- биллинг;
- лимиты;
- статусы подписки;
- обработка ошибок платежей;
- юридические условия;
- поддержка пользователей.

Поэтому сначала нужно сделать стабильную Prompt Arena и проверить ценность продукта.

## Роль API User

API User - это внешний пользователь или сервис, который обращается к платформе через API.

Эта роль не нужна в MVP.

### Что потребуется для API User

Для API User нужны:

- отдельные API-ключи проекта;
- лимиты на ключ;
- логи запросов;
- отключение ключей;
- контроль расходов;
- защита от злоупотреблений;
- документация внешнего API.

До появления стабильного продукта эту роль лучше не добавлять.

## Системная роль AI Judge

AI Judge - это не пользователь-человек.

Это модель, которая используется в Judge Mode для оценки ответов других моделей.

AI Judge появляется в версии `v1.3 - Judge Mode`.

### Что делает AI Judge

AI Judge может:

- сравнивать ответы моделей;
- оценивать качество;
- выбирать победителя;
- объяснять решение;
- выставлять оценки по критериям;
- помогать формировать Leaderboard.

### Ограничение

AI Judge не должен иметь пользовательских прав.

Это системная роль внутри конкретного AI-запроса, а не аккаунт в Supabase Auth.

## Системная роль AI Team Agent

AI Team Agent - это модель с определённой ролью внутри AI Team Mode.

Например:

- Planner;
- Developer;
- Reviewer;
- Security Reviewer;
- Final Answer Writer.

AI Team Agent появляется только в `v2.0 - AI Team Mode`.

### Ограничение

AI Team Agent не является пользователем проекта.

Он не должен иметь доступ к:

- пользовательским данным вне конкретной задачи;
- API-ключам;
- админским настройкам;
- базе данных напрямую;
- чужим приватным запросам.

## Роли и режимы проекта

### Prompt Arena

Prompt Arena доступна первой.

Доступ:

- Guest - да, с лимитами;
- User - да, после `v1.5`;
- Admin - да, для тестирования;
- Owner - да.

### Code Arena Lite

Code Arena Lite появляется в `v1.1`.

Она не запускает код.

Доступ:

- Guest - возможно ограниченно;
- User - да, после появления аккаунтов;
- Admin - да;
- Owner - да.

### Code Arena Runner

Code Arena Runner появляется только в `v1.7`.

Он запускает код, поэтому требует строгой безопасности.

До `v1.7` запрещено:

- запускать пользовательский код;
- выполнять команды пользователя на сервере;
- подключать Docker Runner без security review;
- сохранять результаты выполнения кода в production;
- давать Runner гостям без лимитов.

Доступ после `v1.7`:

- Guest - нет или только очень ограниченно;
- User - ограниченно;
- Paid User - расширенно, если появится монетизация;
- Admin - да для тестирования;
- Owner - да.

### Multi Model Battle

Multi Model Battle появляется в `v1.2`.

Доступ:

- Guest - возможно с лимитами;
- User - да;
- Admin - да;
- Owner - да.

### Judge Mode

Judge Mode появляется в `v1.3`.

Доступ:

- Guest - ограниченно, потому что режим может быть дорогим;
- User - да, с лимитами;
- Paid User - расширенно;
- Admin - да;
- Owner - да.

### Leaderboard

Leaderboard появляется в `v1.4`.

Доступ:

- Guest - читать публичный рейтинг;
- User - читать и голосовать;
- Moderator - проверять спорные данные;
- Admin - управлять настройками;
- Owner - управлять глобальными правилами.

### AI Team Mode

AI Team Mode появляется только в `v2.0`.

Это дорогой и сложный режим.

Доступ:

- Guest - нет или демо-режим;
- User - ограниченно;
- Paid User - расширенно;
- Admin - да для тестирования;
- Owner - да.

## Пользовательские состояния

Роль и состояние аккаунта - это разные вещи.

Роль отвечает на вопрос: **что пользователь имеет право делать**.

Состояние отвечает на вопрос: **аккаунт активен или ограничен**.

Возможные состояния:

```text
anonymous
# пользователь без регистрации

active
# активный зарегистрированный пользователь

limited
# пользователь временно ограничен

blocked
# пользователь заблокирован

pending
# аккаунт создан, но не подтверждён
```

Для MVP достаточно:

```text
anonymous
# гость без регистрации
```

После `v1.5` можно добавить:

```text
active
# обычный активный пользователь

limited
# пользователь с ограничениями

blocked
# заблокированный пользователь
```

## Как хранить роли в базе данных

В MVP полноценная таблица профилей не обязательна.

После `v1.5` можно добавить таблицу `profiles`.

### Таблица profiles

Рекомендуемые поля:

```text
id
# уникальный идентификатор профиля

user_id
# связь с пользователем Supabase Auth

role
# роль пользователя: user, moderator, admin, owner

status
# состояние аккаунта: active, limited, blocked, pending

display_name
# имя, которое показывается в интерфейсе

created_at
# дата создания профиля

updated_at
# дата последнего обновления профиля
```

### Возможные значения role

```text
user
# обычный зарегистрированный пользователь

moderator
# модератор публичного контента

admin
# администратор проекта

owner
# владелец проекта
```

Guest можно не хранить как роль в `profiles`, потому что Guest не имеет аккаунта.

Для Guest лучше использовать временную сессию.

## Гостевые действия

Для гостя можно использовать временный идентификатор.

Возможные поля в таблицах `tasks` и `votes`:

```text
anonymous_session_id
# временный идентификатор гостевой сессии

user_id
# идентификатор пользователя Supabase Auth, если пользователь вошёл
```

Правило:

- если пользователь не вошёл - используется `anonymous_session_id`;
- если пользователь вошёл - используется `user_id`;
- нельзя показывать пользователю чужие записи по чужому session id;
- для публичных данных нужен отдельный флаг публичности.

## Связь ролей с таблицами

| Таблица | Кто создаёт | Кто читает | Кто изменяет |
| --- | --- | --- | --- |
| `models` | Admin или разработчик | Все через API | Admin |
| `tasks` | Guest или User через API | Владелец задачи | Система |
| `model_responses` | Система | Владелец задачи | Система |
| `votes` | Guest или User через API | Система и публичная статистика | Ограниченно |
| `profiles` | Система после регистрации | Владелец профиля | Владелец или Admin |
| `usage_logs` | Система | Admin | Система |
| `leaderboard_scores` | Система | Все, если данные публичные | Система или Admin |
| `code_results` | Runner после `v1.7` | Владелец задачи | Система |
| `team_runs` | Система после `v2.0` | Владелец запуска | Система |

## Row Level Security

После появления аккаунтов нужно использовать Supabase RLS.

RLS нужен, чтобы пользователь не мог читать или менять чужие данные.

Основная логика:

```text
Пользователь видит только свои приватные данные.
# защита истории и личных запросов

Публичные данные видят все.
# например, публичный Leaderboard

Администратор имеет расширенный доступ.
# только к тем данным, которые нужны для управления проектом

Секреты не видит никто через UI.
# ключи остаются только в environment variables
```

## Проверка прав на backend

Frontend может скрывать кнопки, но это не защита.

Настоящая проверка должна быть на backend.

Backend должен проверять:

- кто отправляет запрос;
- какая роль у пользователя;
- активен ли аккаунт;
- не заблокирован ли пользователь;
- не превышен ли лимит;
- разрешена ли выбранная модель;
- разрешён ли выбранный режим;
- можно ли читать конкретную задачу;
- можно ли голосовать;
- можно ли использовать дорогую модель;
- можно ли выполнять Runner-функции.

Принцип:

```text
Frontend показывает удобный интерфейс.
# это не безопасность

Backend принимает финальное решение.
# это настоящая защита
```

## Лимиты по ролям

Лимиты нужны для защиты от лишних расходов на OpenRouter и будущие Runner-функции.

### MVP-лимиты

Для MVP можно использовать простые ограничения:

| Роль | Запросы | Моделей за запрос | Доступ к дорогим моделям |
| --- | --- | --- | --- |
| Guest | Низкий лимит | 2-3 | Нет или ограниченно |
| Admin-разработчик | Технический лимит | По необходимости | Да, вручную |

### Будущие лимиты

| Роль | Запросов в день | Моделей за запрос | Дорогие модели | Runner |
| --- | --- | --- | --- | --- |
| Guest | Низкий лимит | 2-3 | Нет | Нет |
| User | Средний лимит | 3-5 | Ограниченно | Ограниченно после `v1.7` |
| Paid User | Высокий лимит | 5-10 | Да | Да, с лимитами |
| Admin | Технический лимит | По необходимости | Да | Да, для тестов |
| Owner | Глобальный доступ | По необходимости | Да | Да |

Точные цифры нужно выбирать после тестов стоимости.

## Авторизация

Авторизация не нужна для первого MVP.

Она появляется в `v1.5 - Accounts and Profiles`.

Рекомендуемый вариант - Supabase Auth.

Возможные способы входа:

| Способ | Когда использовать |
| --- | --- |
| Email + password | Самый понятный стартовый вариант |
| Magic link | Удобный вход без пароля |
| Google OAuth | Удобно для широкой аудитории |
| GitHub OAuth | Полезно для Code Arena и разработчиков |

Для первой реализации аккаунтов лучше выбрать один простой способ, а не добавлять всё сразу.

## Админ-панель

Полноценная админ-панель появляется в `v1.6 - Admin Panel and Limits`.

До `v1.6` не нужно строить сложную админку.

### Что можно делать до админ-панели

До `v1.6` можно использовать:

```text
Файл конфигурации моделей.
# простой способ задать разрешённые модели

Таблицу models в Supabase.
# более гибкий способ управлять списком моделей

Ручные изменения через Supabase Dashboard.
# достаточно для ранней версии
```

### Что должна уметь админ-панель после `v1.6`

Админ-панель должна уметь:

- включать и отключать модели;
- менять порядок моделей;
- менять лимиты;
- смотреть ошибки;
- смотреть статистику расходов;
- видеть проблемные запросы без раскрытия секретов;
- отключать режимы;
- управлять доступом к дорогим функциям;
- управлять пользователями.

## Ошибки доступа

Пользователь должен получать понятные ошибки.

Примеры:

```text
У вас нет доступа к этой функции.
# роль не позволяет выполнить действие

Войдите в аккаунт, чтобы сохранить историю.
# действие требует авторизации

Достигнут лимит запросов.
# пользователь превысил лимит

Эта модель недоступна для вашей роли.
# модель ограничена

Этот режим пока недоступен.
# функция ещё не включена по roadmap

Запуск кода будет доступен только после включения безопасного Runner.
# Code Arena Runner ещё не готов
```

## Что нельзя делать в MVP

В MVP нельзя добавлять:

- полноценную регистрацию, если она тормозит запуск Prompt Arena;
- сложные роли;
- платные подписки;
- публичные профили;
- сложную модерацию;
- запуск пользовательского кода;
- доступ к дорогим моделям без лимитов;
- админ-панель со слишком широкими правами;
- отображение секретов в UI;
- прямой вызов OpenRouter из браузера.

## Что сделать в первом MVP

В первом MVP нужно сделать так:

1. Guest открывает Prompt Arena.
2. Guest вводит задачу.
3. Guest выбирает разрешённые модели.
4. Backend проверяет prompt и выбранные модели.
5. Backend вызывает OpenRouter.
6. Backend сохраняет задачу и ответы в Supabase.
7. Guest видит ответы.
8. Guest выбирает лучший ответ.
9. Backend сохраняет vote.
10. История работает только в рамках разрешённой логики.
11. Секреты остаются только на backend.
12. Лимиты защищают бюджет.

## Проверка перед добавлением новой роли

Перед добавлением новой роли нужно ответить:

- зачем эта роль нужна сейчас;
- какую проблему она решает;
- какая версия roadmap требует эту роль;
- какие API она может вызывать;
- какие таблицы она может читать;
- какие таблицы она может изменять;
- какие лимиты у роли;
- как роль проверяется на backend;
- как роль проверяется через RLS;
- что будет при попытке обойти frontend;
- не ломает ли новая роль старый MVP.

## Итог

Для проекта **Новая эпоха** не нужно сразу делать сложную систему ролей.

Правильный путь:

1. В MVP использовать Guest, System и Admin-разработчика.
2. После стабильной Prompt Arena добавить User через Supabase Auth.
3. После аккаунтов добавить Admin Panel and Limits.
4. После лимитов открывать более дорогие и опасные функции.
5. Code Arena Runner разрешать только в `v1.7`.
6. AI Team Mode разрешать только в `v2.0`.

Главная цель первого этапа - не роли, а рабочая Prompt Arena, где пользователь может сравнить ответы нескольких AI-моделей и выбрать лучший результат.
