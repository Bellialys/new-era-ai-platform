# 2. Что было найдено на момент аудита

На момент аудита были найдены такие проблемы:

```text
02-project-plan.md частично использовал старую нумерацию версий.
# конфликт с 14-roadmap.md

09-api-structure.md конфликтовал с 08-database.md по vote_type.
# best не входил в разрешённые значения базы

09-api-structure.md местами использовал snake_case в API JSON.
# frontend/API должны использовать camelCase

04-mvp-scope.md указывал лимит prompt 4000.
# остальные документы и env использовали 8000

Нужно было разделить UI-выбор победителя и сохранённый vote.
# v0.3 UI-only, v0.6 saved vote

16-decisions.md говорил про docs/.
# фактически документация лежала в корне репозитория

12-security-and-env.md был написан не в общем стиле проекта.
# позже документ был обновлён
```

---

# 3. Что было исправлено после аудита

После этого аудита были выполнены последующие синхронизации.

Исправлено:

```text
02-project-plan.md синхронизирован с 14-roadmap.md.
# порядок версий приведён к канону

09-api-structure.md приведён к camelCase API JSON.
# taskId, responseId, modelIds, modeSlug

vote_type для пользовательского голоса закреплён как user на backend.
# best удалён из frontend API

04-mvp-scope.md приведён к MAX_PROMPT_LENGTH=8000.
# лимит совпадает с env и базой

16-decisions.md больше не требует обязательную папку docs/.
# документация хранится в корне репозитория

12-security-and-env.md обновлён.
# структура и правила окружения актуализированы
```

Дополнительные исправления кода после более глубокого аудита:

```text
ArenaApiResponse и ArenaResponseView разделены.
# API и UI больше не смешиваются

Mock-ответы используют answerText.
# совпадает с будущим /api/compare

modelRole добавляется на клиенте.
# не дублируется в API-like response

MAX_PROMPT_LENGTH=8000 применён в UI.
# textarea и validateForm используют лимит

Старые responses сбрасываются.
# UI не показывает устаревшие данные
```
