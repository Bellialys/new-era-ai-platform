# ADR-002: Двойная бухгалтерия для биллинга (Append-Only Ledger)

**Status:** Accepted
**Date:** 2026-07
**Deciders:** Platform Team

---

## Context

При проектировании биллинговой системы (v2.3) нужно было выбрать модель хранения финансовых данных.

Простейший подход: `users.balance_usd` — одно поле, которое инкрементируем/декрементируем.

---

## Decision

Использовать **двойную бухгалтерию** через `billing_events` — append-only ledger записей транзакций. Кэшированный `balance_usd` в `user_settings` — производный, не источник истины.

```sql
-- Источник истины
INSERT INTO billing_events (user_id, type, amount_usd, stripe_event_id)
VALUES ($userId, 'USAGE_CHARGE', -0.05, $eventId);

-- Кэш для быстрого чтения (не источник истины)
UPDATE user_settings SET balance_usd = balance_usd - 0.05 WHERE user_id = $userId;
```

---

## Rationale

**Почему не просто UPDATE balance:**
- Потеря аудиторского следа — невозможно воспроизвести историю
- Нет защиты от race conditions (несколько списаний одновременно)
- При баге в коде баланс может уйти в минус без возможности расследования
- Невозможно пройти финансовый аудит (SOC 2, enterprise требования)

**Почему Append-Only Ledger:**
- Полная история каждой транзакции
- `UNIQUE(stripe_event_id)` — идемпотентность: одно Stripe событие = одна запись
- При расхождении `SUM(billing_events) ≠ balance_usd` — reconciliation job обнаруживает и алертит
- Аудиторы могут верифицировать каждую операцию

---

## Consequences

**Позитивные:**
- Полный аудиторский след
- Идемпотентность через UNIQUE constraint
- Возможность reconciliation
- Основа для enterprise финансовой отчётности

**Негативные:**
- Таблица растёт бесконечно (смягчается: финансовые записи не удаляются по закону)
- Чтение баланса через SUM медленно на больших объёмах (смягчается кэшем в `user_settings`)

**Критическое правило:**
Reconciliation никогда не автокорректирует `balance_usd`. Только алерт администратору.
Автокоррекция скрыла бы баги в транзакционной логике. (см. ADR-003)
