# ADR-003: Reconciliation только алертинг (не автокоррекция)

**Status:** Accepted
**Date:** 2026-07
**Deciders:** Platform Team

---

## Context

После внедрения двойной бухгалтерии (ADR-002) нужна была reconciliation процедура:
ежедневная проверка что `SUM(billing_events)` совпадает с `user_settings.balance_usd`.

Вопрос: при обнаружении расхождения — автоматически исправить или только алертить?

---

## Decision

Reconciliation **только алертит** о расхождениях. Никогда не автоматически корректирует `balance_usd`.

```sql
-- При расхождении: только INSERT, никакого UPDATE balance_usd
INSERT INTO billing_events (type, metadata)
VALUES ('RECONCILIATION_ALERT', jsonb_build_object(
  'discrepancy', discrepancy,
  'ledger_sum', ledger_sum,
  'cached_balance', cached_balance
));
-- + уведомление всем admin
```

---

## Rationale

**Почему не автокоррекция:**
- Расхождение = симптом бага в транзакционной логике
- Автокоррекция скрывает баг: баланс "правильный", но причина не найдена
- Баг повторяется: следующая транзакция снова создаст расхождение
- При автокоррекции теряется информация о природе проблемы
- Финансовый аудит: автоматические "исправления" баланса — красный флаг

**Почему только алертинг:**
- Заставляет разобраться в причине расхождения
- Создаёт запись в ledger (`RECONCILIATION_ALERT`) — аудиторский след
- Администратор принимает осознанное решение: это ожидаемо или это баг?
- Соответствует принципу "fail loudly, not silently"

---

## Consequences

**Позитивные:**
- Баги в биллинге обнаруживаются и исправляются, а не маскируются
- Чистый аудиторский след
- Соответствие enterprise и SOC 2 требованиям

**Негативные:**
- Требует ручного вмешательства при каждом алерте
- Временное расхождение может вызывать беспокойство у пользователей (смягчается: показываем кэш)

**Операционное правило:**
При получении RECONCILIATION_ALERT: расследовать причину → исправить баг →
вручную скорректировать `balance_usd` через admin panel с записью в `admin_audit_log`.
