# ADR-001: HMAC-SHA256 вместо bcrypt для API-ключей

**Status:** Accepted
**Date:** 2026-07
**Deciders:** Platform Team

---

## Context

При реализации системы API-ключей (v2.5) встал вопрос: как хранить ключи безопасно?

Первый инстинкт — использовать bcrypt, как для паролей. Но API-ключи и пароли — разные сущности.

---

## Decision

Использовать **HMAC-SHA256** с серверным секретом (`API_KEY_SECRET`) вместо bcrypt.

```typescript
const keyHash = crypto.createHmac('sha256', process.env.API_KEY_SECRET)
  .update(rawKey).digest('hex');
```

---

## Rationale

**Почему не bcrypt:**
- bcrypt намеренно медленный (~100ms на верификацию) — это нужно для паролей, чтобы защититься от брутфорса
- API-ключ верифицируется при каждом HTTP-запросе (тысячи раз в секунду)
- 100ms × N req/s = неприемлемый overhead
- API-ключи имеют высокую энтропию (256+ бит) — брутфорс против HMAC-SHA256 нереален

**Почему HMAC-SHA256:**
- Верификация: микросекунды вместо ~100ms
- Высокая энтропия API-ключей делает rainbow tables бессмысленными
- Индустриальный стандарт для API-ключей (GitHub, Stripe, Twilio используют аналогичный подход)
- HMAC с серверным секретом: даже при утечке БД ключи не восстановимы без `API_KEY_SECRET`

---

## Consequences

**Позитивные:**
- Верификация за O(1), нет CPU overhead
- Простая реализация

**Негативные:**
- Компрометация `API_KEY_SECRET` = компрометация всех ключей (требует ротации всех ключей)
- В отличие от bcrypt, нет salt per-key (смягчается высокой энтропией самих ключей)

**Смягчение рисков:**
- `API_KEY_SECRET` хранится только в Vercel Environment Variables, не в коде
- Регулярная ротация `API_KEY_SECRET` (при компрометации — batch ротация всех ключей)
