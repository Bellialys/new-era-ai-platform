## Что входит в v2.7

### 1. SSO — SAML 2.0 / OIDC

```sql
CREATE TABLE organization_sso_configs (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id                UUID NOT NULL UNIQUE REFERENCES organizations(id),
  protocol              TEXT NOT NULL CHECK (protocol IN ('saml', 'oidc')),
  status                TEXT NOT NULL DEFAULT 'draft'
                        CHECK (status IN ('draft', 'testing', 'active')),

  -- SAML
  idp_entity_id         TEXT,
  idp_sso_url           TEXT,
  signing_certificates  JSONB DEFAULT '[]',  -- массив: [{cert, valid_until, active}]
  sp_entity_id          TEXT,
  acs_url               TEXT,
  clock_skew_tolerance  INTEGER DEFAULT 300, -- секунды

  -- OIDC
  discovery_url         TEXT,
  client_id             TEXT,
  client_secret_hash    TEXT,

  -- Общее
  role_mappings         JSONB DEFAULT '{}',
    -- {"okta-admins": "admin", "okta-users": "member"}
  jit_provisioning      BOOLEAN DEFAULT TRUE,
  test_user_email       TEXT,   -- для режима 'testing'

  created_at            TIMESTAMPTZ DEFAULT now(),
  updated_at            TIMESTAMPTZ DEFAULT now()
);

-- История изменений конфигурации
CREATE TABLE organization_sso_configs_history (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL,
  old_config  JSONB,
  new_config  JSONB,
  changed_by  UUID NOT NULL,
  changed_at  TIMESTAMPTZ DEFAULT now(),
  reason      TEXT
);
```

**Тестовый режим:**
`status = 'testing'` — только пользователь с `test_user_email` может войти через IdP.
`status = 'active'` — SSO включён для всех пользователей с `allowed_email_domains`.
При `status = 'active'` парольный вход для этих доменов отключается.

**Account Linking:**
При первом SSO-логине:
1. Ищем пользователя с тем же email в нашей БД
2. Если найден → связываем IdP Subject ID с существующим `user_id`
   (таблица `idp_subject_links`: `user_id`, `org_id`, `idp_subject_id`, `protocol`)
3. Вся история `usage_events`, `billing_events`, API-ключи сохраняются
4. Если не найден → JIT Provisioning: создаём аккаунт + добавляем в организацию

**Защита от SAML replay-атак:**
Таблица `saml_used_assertions` (`assertion_id TEXT`, `expires_at TIMESTAMPTZ`).
При каждом login: проверяем что `assertion_id` не использован ранее.
Также проверяем `InResponseTo` совпадает с нашим AuthnRequest ID.
Записи очищаются cron-задачей после `NotOnOrAfter`.

**Clock Skew:**
При валидации `NotBefore` / `NotOnOrAfter` добавляем ±`clock_skew_tolerance` секунд.

**Zero-downtime ротация сертификатов SAML:**
`signing_certificates` — массив объектов `{cert, valid_until, active}`.
При верификации подписи проверяем все `active: true` сертификаты.
Администратор добавляет новый сертификат, оба работают параллельно,
затем деактивирует старый.

### 2. SCIM 2.0 Provisioning

```sql
CREATE TABLE scim_tokens (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL REFERENCES organizations(id),
  token_hash  TEXT NOT NULL UNIQUE,
  name        TEXT NOT NULL,
  expires_at  TIMESTAMPTZ NOT NULL,
  last_used_at TIMESTAMPTZ,
  created_by  UUID NOT NULL,
  revoked_at  TIMESTAMPTZ,
  created_at  TIMESTAMPTZ DEFAULT now()
);
```

**Эндпоинты (RFC 7643/7644):**
```
GET  /scim/v2/Users?startIndex=1&count=100&filter=userName eq "user@co.com"
POST /scim/v2/Users
GET  /scim/v2/Users/{id}
PUT  /scim/v2/Users/{id}
PATCH /scim/v2/Users/{id}
DELETE /scim/v2/Users/{id}

GET  /scim/v2/Groups
POST /scim/v2/Groups
PATCH /scim/v2/Groups/{id}    -- маппинг групп на роли
DELETE /scim/v2/Groups/{id}
```

**Пагинация и фильтрация:**
Поддержка `startIndex`, `count`, `filter` — обязательное требование Okta/Azure AD.
Без этого интеграция не проходит их сертификационную валидацию.

**Soft Delete — SCIM deprovisioning:**
`DELETE /scim/v2/Users/{id}` или `PATCH active: false`:
- Устанавливает `user_settings.plan_status = 'suspended'`
- Отзывает все API-ключи (`revoked_at = now()`)
- Завершает все Supabase сессии
- НЕ удаляет запись из БД — сохраняем `billing_events`, `usage_events`, audit log
- Запись в `organization_audit_log (action='member_suspended_by_scim')`

**Group-Based Access Control:**
SCIM Groups маппируются на роли через `role_mappings` в `organization_sso_configs`.
Добавление пользователя в SCIM Group → автоматическое назначение роли в организации.

**SCIM очередь через Inngest:**
Все SCIM-запросы обрабатываются асинхронно через Inngest с idempotency key
(`scim_event_id` из заголовка или генерируется нами).
Повторная доставка не создаёт дубликат.

**Валидация схемы:**
Строгая проверка входящих SCIM-пакетов на соответствие RFC 7643.
При ошибке — детальный SCIM error response:
```json
{ "schemas": ["urn:ietf:params:scim:api:messages:2.0:Error"],
  "scimType": "invalidValue", "detail": "...", "status": 400 }
```

**Уведомления об истечении:**
За 14 дней до `expires_at` SCIM-токена → email admin + баннер в UI.
За 30 дней до `valid_until` SAML-сертификата → аналогично.

### 3. SSO Audit Log

```sql
CREATE TABLE sso_audit_log (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL,
  user_email  TEXT NOT NULL,
  protocol    TEXT NOT NULL,  -- 'saml', 'oidc', 'scim'
  action      TEXT NOT NULL,
    -- 'login_success', 'login_failed', 'jit_provisioned',
    -- 'account_linked', 'scim_created', 'scim_suspended', 'scim_updated'
  failure_reason TEXT,         -- 'invalid_signature', 'expired_assertion', 'replay_detected'
  ip_address  TEXT,
  idp_subject_id TEXT,
  created_at  TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_sso_audit_org_created ON sso_audit_log(org_id, created_at DESC);
```

Используется для: расследования инцидентов, доказательства контроля доступа (SOC 2).

### 4. Session Policies

```sql
CREATE TABLE session_policies (
  org_id                    UUID PRIMARY KEY REFERENCES organizations(id),
  session_timeout_hours     INTEGER DEFAULT 168,  -- 7 дней
  require_mfa_for           TEXT[] DEFAULT '{}',
    -- 'audit_export', 'balance_transfer', 'org_api_key_create'
  allowed_countries         TEXT[] DEFAULT '{}',  -- пусто = все страны
  updated_by                UUID,
  updated_at                TIMESTAMPTZ DEFAULT now()
);
```

**Step-Up MFA:**
Для операций в `require_mfa_for` — принудительный ввод TOTP кода
даже при активной сессии. Применяется независимо от SSO.

**Allowed Countries:**
При входе проверяем геолокацию по IP.
Вход из заблокированной страны → 403 + запись в `sso_audit_log`.

**Graceful Fallback при недоступности IdP:**
Если IdP не отвечает (timeout 10 сек), `owner` и `admin` могут войти
через one-time code на email. Обычные `member` получают страницу
«Сервис авторизации временно недоступен».

### 5. Принудительное завершение сессий

`POST /api/org/{id}/sessions/terminate` (только owner/admin):
- Инвалидирует все Supabase refresh tokens через `admin.auth.signOut(userId)` для каждого члена
- Запись в `organization_audit_log`
- Возвращает количество завершённых сессий

Индивидуально: `POST /api/org/{id}/members/{userId}/sessions/terminate`.

UI в `/admin/org/{id}/members` — список активных сессий с устройством, IP, временем.

### 6. Трансфер баланса org ↔ user

`POST /api/org/{id}/balance/transfer`:
```json
{ "direction": "org_to_user", "amount_usd": 50.00, "comment": "Возврат за unused credits" }
```

Запись в `billing_events` с типом `INTERNAL_LEDGER_TRANSFER`.
**Не попадает** в Stripe Tax / Invoices — помечен `taxable: false`.

**Maker-Checker для трансферов > $100:**
1. Maker (owner/admin) инициирует → статус `pending_approval`
2. Email уведомление второму admin
3. Checker подтверждает через отдельный endpoint (`POST .../approve`)
4. Только после подтверждения — атомарное обновление балансов

Все шаги в `organization_audit_log`.

### 7. Append-Only Organization Audit Log

```sql
-- Запрет UPDATE и DELETE на уровне БД
CREATE OR REPLACE FUNCTION prevent_audit_modification()
RETURNS TRIGGER AS $$
BEGIN
  RAISE EXCEPTION 'organization_audit_log is append-only';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER no_update_audit
  BEFORE UPDATE OR DELETE ON organization_audit_log
  FOR EACH ROW EXECUTE FUNCTION prevent_audit_modification();
```

Только `INSERT` разрешён. Соответствие требованиям SOC 2 Type II и ISO 27001.

**Streaming CSV Export:**
`GET /api/org/{id}/audit?export=csv&from=...&to=...` — серверные курсоры,
батчи по 1000 строк, Stream Response. Нет OOM при миллионах записей.

### 8. Мониторинг аномалий SSO

Inngest job каждые 5 минут анализирует `sso_audit_log`:
- Резкий рост `login_failed` (>20 за 5 мин для одной org) → алерт admin
- Вход из страны не в `allowed_countries` → 403 + немедленный алерт
- Массовое SCIM `scim_suspended` (>10 за час) → алерт (возможна атака)
- `replay_detected` любой → немедленный алерт

Алерты через `notifications` (in-app) + email через Resend.

### 9. SCIM Sync Dashboard

В `/admin/org/{id}/sso`:
- Статус SSO конфигурации (draft/testing/active)
- Последняя SCIM синхронизация: время, количество created/updated/suspended/errors
- Список ошибок с деталями
- Кнопка «Sync Now» (принудительная ручная синхронизация)
- Статус сертификатов SAML с датой истечения
