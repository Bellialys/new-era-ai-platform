## Что входит в v2.2

### 1. User cost budgets — месячные лимиты

Пользователь устанавливает лимит в USD на месяц через настройки профиля.

Новые поля в `user_settings`:
```
monthly_budget_usd   NUMERIC(10,4)  DEFAULT NULL   -- NULL = без лимита
current_month_cost   NUMERIC(10,4)  DEFAULT 0      -- кэш текущих расходов
billing_month        DATE           DEFAULT NULL   -- первый день текущего месяца
plan_type            TEXT           DEFAULT 'free' -- задел на v2.3 тарифы
```

**Проверка бюджета — O(1):**
Читаем `current_month_cost` из `user_settings`, не делаем `SUM()` по `usage_events`.
Обновление атомарно в той же транзакции, где пишется `usage_events`:
```sql
UPDATE user_settings
SET current_month_cost = current_month_cost + $cost
WHERE user_id = $userId;
```

**Атомарная блокировка (без гонок):**
```sql
SELECT monthly_budget_usd, current_month_cost
FROM user_settings
WHERE user_id = $userId
FOR UPDATE;
-- если current_month_cost + $estimated_cost > monthly_budget_usd → 429
```

При достижении 100% → 429 BUDGET_EXCEEDED. Запрос в `usage_events` пишется
со статусом `blocked` и `error_code = 'BUDGET_EXCEEDED'` — это аналитика
того, как часто пользователи упираются в лимит.

**Авто-сброс при смене месяца:**
При каждой проверке бюджета сравниваем `billing_month` с `date_trunc('month', now())`.
Если месяц сменился — атомарно обнуляем `current_month_cost` и обновляем
`billing_month`. Никакого cron-сброса не нужно — сброс происходит при первом
запросе нового месяца.

### 2. Cost alerts — уведомления в UI

При пересечении порогов 50% / 80% / 100% бюджета — запись в таблицу `notifications`.

Новая таблица:
```sql
CREATE TABLE notifications (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES auth.users(id),
  type            TEXT NOT NULL,             -- 'budget_50', 'budget_80', 'budget_100'
  threshold_key   TEXT NOT NULL,
  billing_month   DATE NOT NULL,
  message         TEXT NOT NULL,
  read            BOOLEAN DEFAULT FALSE,
  created_at      TIMESTAMPTZ DEFAULT now(),

  UNIQUE (user_id, threshold_key, billing_month)  -- идемпотентность
);
```

`UNIQUE(user_id, threshold_key, billing_month)` гарантирует: один алерт на
порог за месяц, даже если 1000 параллельных запросов пересекают порог одновременно.

UI: баннер с прогресс-баром бюджета (жёлтый 50%, оранжевый 80%, красный 100%).
Email в v2.2 нет — только in-app. Email (через Resend) — v2.3.

### 3. Admin dashboard — /admin

Доступен только пользователям с `user_metadata.role === 'admin'`.

**Middleware (server-side):**
```typescript
// middleware.ts
if (pathname.startsWith('/admin')) {
  const role = session?.user?.user_metadata?.role;
  if (role !== 'admin') return NextResponse.redirect('/403');
}
```
Проверка на сервере, не только на клиенте.

**Три раздела (простые таблицы, серверная пагинация):**

- **Обзор:** DAU за последние 7 дней, запросов за сегодня, суммарная стоимость
  за месяц, топ-5 моделей по использованию. Данные из `usage_events`.
- **Пользователи:** список с `email`, `plan_type`, `current_month_cost`,
  `monthly_budget_usd`, `created_at`. Пагинация 50/страницу, фильтр по email.
- **Модели:** использование за 30 дней, средняя стоимость запроса, win rate
  из `leaderboard_snapshots`.

Серверная пагинация через `limit/offset`, все запросы через `service_role`
клиент в server component — не через API.

**Аудит действий администратора:**
```sql
CREATE TABLE admin_audit_log (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  admin_id      UUID NOT NULL,
  action        TEXT NOT NULL,   -- 'view_user', 'set_budget', 'change_plan'
  target_entity TEXT,            -- 'user:uuid', 'global'
  metadata      JSONB,
  created_at    TIMESTAMPTZ DEFAULT now()
);
```
Любое действие администратора (просмотр профиля, изменение бюджета) пишется
в `admin_audit_log`. Только `service_role` имеет доступ на запись.

### 4. CSV export

`GET /api/usage?export=csv&from=2026-06-01&to=2026-07-01`

Auth-only. Лимит: 90 дней и 10 000 строк за один запрос.
Поля: `created_at`, `model_key`, `tokens_input`, `tokens_output`, `cost_usd`,
`latency_ms`, `mode_slug`, `status`.
`raw_response` и `prompt_text` не экспортируются.

Формат JSON: `?export=json` — те же данные, массив объектов.

При превышении лимита 10 000 строк → 400 с подсказкой сузить диапазон дат.

### 5. Rate limit hardening

Единый конфиг в `src/lib/server/rate-limit-config.ts`:
```typescript
export const RATE_LIMITS = {
  arena:       { authed: 60, guest: 10,  window: 60_000 },
  imageArena:  { authed: 30, guest: 5,   window: 60_000 },
  usage:       { authed: 30, guest: 0,   window: 60_000 },
  teamRun:     { authed: 20, guest: 0,   window: 60_000 },
  admin:       { authed: 60, guest: 0,   window: 60_000 },
};
```

Добавляем rate limit на `/api/arena` и `/api/image-arena` (сейчас без лимита).
Лимит применяется по `user_id` для авторизованных, по IP — только для гостей.
