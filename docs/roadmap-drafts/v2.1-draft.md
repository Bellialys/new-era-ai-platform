## Цель версии

В v2.0 созданы 8 таблиц DB v2 Foundation — они пустые.
v2.1 наполняет их данными: каждый AI-вызов пишет метрики, Team Mode мигрирует
на выделенные таблицы, Code Runner пишет историю запусков, лидерборд
обновляется инкрементально при каждом голосовании, пользователь получает
аналитику через /api/usage.

## Что входит в v2.1

### 1. usage_events — телеметрия каждого AI-вызова

Каждый вызов OpenRouter через наш backend пишет строку:

```
user_id | model_key | tokens_input | tokens_output | cost_usd | latency_ms | mode_slug | task_id | status
```

**Архитектурное решение:** запись асинхронная (fire-and-forget).
Если запись в БД упала — пользователь получает ответ, ошибка логируется,
счётчик инкрементируется. Сбой телеметрии не даёт 500.

Паттерн:
```typescript
// После отправки ответа клиенту
void recordUsageEvent(payload).catch(err => {
  console.error("[usage_events] write failed:", err.message);
  telemetryErrors.inc(); // Prometheus или аналог
});
```

Где вызывать: `src/lib/server/openrouter.ts` после получения ответа от OpenRouter.

### 2. team_runs + team_run_steps — Team Mode на выделенных таблицах

Сейчас Team Mode пишет в `tasks` и `model_responses` — это временное решение.
В v2.1 Team Mode пишет в `team_runs` и `team_run_steps`.

Каждый запуск:
```
team_runs: id | user_id | prompt | status | total_tokens | total_cost | created_at
team_run_steps: id | run_id | step_name | model_key | input | output | tokens | latency_ms
```

Шаги: Planner → Researcher → Critic → Finalizer.

Обратная совместимость: старые записи в `tasks` с mode_slug='ai-team-mode'
остаются. Новые запросы идут в `team_runs`.

### 3. code_runs — история выполнения Code Runner

Каждый запуск кода (Code Arena Lite) пишет строку:
```
user_id | language | status | exit_code | latency_ms | created_at
```

Payload (код и stdout) в v2.1 хранится в БД как TEXT.
Лимит: код ≤ 10 КБ, stdout ≤ 50 КБ. Если превышает — обрезаем с пометкой `truncated=true`.
(S3 для больших payloads — откладываем до v2.3, когда увидим реальные размеры.)

### 4. leaderboard_snapshots — инкрементальный ELO

**Текущая проблема:** Vercel Cron пересчитывает ELO раз в сутки по всем голосам.
При росте данных это таймаут (Vercel Cron лимит — 60 сек).

**v2.1 подход:**
- При каждом голосовании (`POST /api/judge`) — обновляем ELO двух моделей
  атомарно (Elo rating update formula).
- Vercel Cron раз в сутки — делает snapshot в `leaderboard_snapshots` для истории.
- `/api/leaderboard` читает текущий ELO из `models.elo_rating`, а не пересчитывает.

ELO формула (K=32):
```
expected_a = 1 / (1 + 10^((elo_b - elo_a) / 400))
new_elo_a = elo_a + K * (score - expected_a)
```

Стартовый ELO всех моделей: 1200.

### 5. GET /api/usage — аналитика для пользователя

Новый endpoint. Auth-only (401 для гостей).

Параметры:
```
GET /api/usage?from=2026-06-01&to=2026-07-01&group_by=model
```

Ответ:
```json
{
  "totalRequests": 142,
  "totalTokens": 284000,
  "totalCostUsd": 0.43,
  "byModel": [
    { "modelKey": "gpt-4o", "requests": 67, "tokens": 140000, "costUsd": 0.28 },
    { "modelKey": "claude-3-5-sonnet", "requests": 75, "tokens": 144000, "costUsd": 0.15 }
  ],
  "period": { "from": "2026-06-01", "to": "2026-07-01" }
}
```

Защита:
- Rate limit: 30 req/min per user (наш стандартный паттерн).
- Фильтрация по from/to обязательна (без дат — ошибка 400, не весь срез).
- Cache: `Cache-Control: private, max-age=300` (5 минут, только для конкретного пользователя).

## Что НЕ входит в v2.1

- Billing / лимиты стоимости — v2.2
- Admin dashboard — v2.2
- S3 для code payloads — v2.3
- Table partitioning — v3.0+ (нужно при миллионах строк)
- Outbox Pattern / PagerDuty / Dead Man's Snitch — L3 enterprise, не для alpha
- Разбивка токенов cache_read/cache_creation — нужна для точного billing (v2.2)
