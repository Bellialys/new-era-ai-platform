## Ключевые архитектурные решения

### Global Control Plane / Regional Data Plane

Концепция «Auth глобальный, данные региональные» нарушает GDPR (Schrems II):
email-адреса EU-пользователей оказываются на US-серверах.

**Решение: двухуровневая архитектура:**

```
Global Control Plane (US, нет PII):
  - global_user_id
  - organization_id
  - data_region ('eu' | 'us')
  - account_status

Regional Data Plane EU (Frankfurt):
  - user_settings (email, профиль)
  - usage_events
  - billing_events
  - saved_prompts
  - organization_audit_log

Regional Data Plane US (Virginia):
  - то же, для US-организаций
```

При логине JWT содержит `data_region`. Next.js Middleware читает регион из JWT
(без запроса к БД) и роутит на `eu.api.domain.com` или `us.api.domain.com`.

```sql
-- Global Control Plane
CREATE TABLE global_accounts (
  global_user_id  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL,
  data_region     TEXT NOT NULL CHECK (data_region IN ('eu', 'us')),
  account_status  TEXT NOT NULL DEFAULT 'active',
  created_at      TIMESTAMPTZ DEFAULT now()
  -- НИКАКИХ email, имён или PII
);
```

### Региональные KMS мастер-ключи

Отдельный мастер-ключ KMS для EU и для US.
Компрометация US-ключа не затрагивает EU-данные.
Ротация раз в 90 дней с автоматическим перешифрованием всех DEK данного региона.

### Write-to-Read Routing (Session Stickiness)

Проблема: после POST в Primary (US) GET-запрос из EU-реплики не видит изменений
из-за Replication Lag. Результат: пользователь не видит только что созданный Workspace.

**Решение:**
```typescript
// После успешной записи
response.headers.set('Set-Cookie', 'force_primary=1; Max-Age=10; Secure; SameSite=Strict');

// В Middleware
if (request.cookies.get('force_primary')) {
  return routeToPrimary(request);
}
return routeToNearestReplica(request);
```

10 секунд достаточно для гарантированной синхронизации реплики.

### Middleware проверка кросс-региональных запросов

```typescript
// В каждом региональном API handler
const orgRegion = await getOrgRegion(orgId); // из JWT, не из БД
if (orgRegion !== process.env.CURRENT_REGION) {
  await alertCrossRegionLeak({ orgId, orgRegion, currentRegion: process.env.CURRENT_REGION });
  return new Response('Cross-region routing error', { status: 500 });
}
```

Алерт в Slack + запись в `compliance_evidence_snapshots`. Регулярный пентест на эту тему.

### Cloudflare for SaaS (Custom Hostnames)

Вместо Vercel Domains API (жёсткие Rate Limits Let's Encrypt — 50/неделю).
Cloudflare for Platforms берёт на себя:
- Выдачу SSL/TLS сертификатов (через собственные лимиты)
- DDoS защиту на Edge
- Роутинг трафика на наши Vercel Functions

Org добавляет `ai.acme.com` → получает CNAME запись → Cloudflare верифицирует →
SSL активен. Статус в UI: `pending_dns` → `pending_validation` → `active` → `error`.

## Что входит в v3.0

### 1. Data Residency — EU / US

```sql
CREATE TABLE organizations (
  -- существующие поля...
  data_region       TEXT NOT NULL DEFAULT 'us' CHECK (data_region IN ('eu', 'us')),
  -- После создания организации регион нельзя менять (только через v3.1 migration API)
);

-- Для v3.1 (схему закладываем сейчас)
CREATE TABLE organization_regions (
  org_id       UUID NOT NULL REFERENCES organizations(id),
  region       TEXT NOT NULL CHECK (region IN ('eu', 'us')),
  is_primary   BOOLEAN DEFAULT TRUE,
  activated_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (org_id, region)
);
```

**Data Export с региональной меткой:**
ZIP экспорта (`gdpr_requests.type = 'export'`) генерируется только
из Data Plane региона пользователя. `manifest.json` содержит:
```json
{ "data_region": "eu", "exported_at": "2026-07-01T10:00:00Z", "regional_plane": "eu-frankfurt" }
```

**Cross-Region DR:**
Асинхронная репликация критических таблиц (`organizations`, `billing_events`,
`global_accounts`) из Primary-региона в Standby. При полном падении региона:
Read-Only режим (`platform_readonly = true`) поднимается из Standby за минуты.

### 2. Custom Domains (Cloudflare for SaaS)

```sql
CREATE TABLE organization_custom_domains (
  id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id                UUID NOT NULL UNIQUE REFERENCES organizations(id),
  domain                TEXT NOT NULL UNIQUE,
  cloudflare_hostname_id TEXT,
  status                TEXT NOT NULL DEFAULT 'pending_dns'
                        CHECK (status IN ('pending_dns', 'pending_validation', 'active', 'error')),
  ssl_status            TEXT,
  ssl_expires_at        TIMESTAMPTZ,
  verified_at           TIMESTAMPTZ,
  created_at            TIMESTAMPTZ DEFAULT now()
);
```

**Алгоритм:**
1. Org вводит домен `ai.acme.com`
2. POST к Cloudflare API → создаём Custom Hostname → возвращаем CNAME
3. Отображаем в UI: «Добавьте CNAME запись: `ai.acme.com → custom.cloudflare.net`»
4. Вебхук от Cloudflare (асинхронно) → обновляем `status` → уведомление admin org
5. HSTS enforced: `Strict-Transport-Security: max-age=31536000; includeSubDomains`
6. Enterprise: поддержка загрузки собственного SSL-сертификата (через Cloudflare API)

### 4. AI Marketplace

```sql
CREATE TABLE marketplace_providers (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name            TEXT NOT NULL,
  slug            TEXT NOT NULL UNIQUE,
  api_base_url    TEXT NOT NULL,
  auth_scheme     TEXT CHECK (auth_scheme IN ('bearer', 'api_key', 'oauth2')),
  models          JSONB NOT NULL,   -- [{id, name, price_per_token_in, price_per_token_out}]
  healthcheck_url TEXT NOT NULL,
  status          TEXT DEFAULT 'pending_review'
                  CHECK (status IN ('pending_review', 'active', 'suspended')),
  reviewed_by     UUID,
  reviewed_at     TIMESTAMPTZ,
  created_at      TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE org_marketplace_connections (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES organizations(id),
  provider_id     UUID NOT NULL REFERENCES marketplace_providers(id),
  encrypted_dek   TEXT NOT NULL,    -- KMS-зашифрованный DEK
  encrypted_key   TEXT NOT NULL,    -- AES-256-GCM(api_key, DEK)
  key_fingerprint TEXT NOT NULL,    -- первые 8 символов для отображения
  status          TEXT DEFAULT 'active' CHECK (status IN ('active', 'suspended', 'revoked')),
  elastic_ip_id   TEXT,             -- Elastic IP для Enterprise
  shredded_at     TIMESTAMPTZ,      -- при revoke: Crypto-Shredding DEK
  created_by      UUID NOT NULL,
  approved_by     UUID,             -- Maker-Checker
  approved_at     TIMESTAMPTZ,
  created_at      TIMESTAMPTZ DEFAULT now(),
  UNIQUE (org_id, provider_id)
);

CREATE TABLE marketplace_provider_stats (
  org_id          UUID NOT NULL REFERENCES organizations(id),
  provider_id     UUID NOT NULL REFERENCES marketplace_providers(id),
  date            DATE NOT NULL,
  total_cost_usd  NUMERIC(10,6) DEFAULT 0,
  request_count   INTEGER DEFAULT 0,
  error_count     INTEGER DEFAULT 0,
  p95_latency_ms  NUMERIC,
  PRIMARY KEY (org_id, provider_id, date)
);
```

**Crypto-Shredding для ключей провайдеров:**
Каждое подключение имеет свой DEK (Data Encryption Key) в KMS.
При `revoke`: удаляем DEK → `encrypted_key` становится нечитаемым → `shredded_at = now()`.

**Approval Workflow (Maker-Checker) для критических операций:**
- Подключение нового провайдера
- Смена ключа (`encrypted_key`)
- Просмотр `key_fingerprint` в аудите

Второй admin подтверждает через `POST /api/org/{id}/marketplace/{connectionId}/approve`.
Без `approved_by` — подключение в статусе `pending_approval`, запросы не проходят.

**Изоляция трафика (SSRF защита):**
Все запросы к провайдерам идут через выделенный outbound proxy.
Whitelist: только IP-диапазоны верифицированных провайдеров из `marketplace_providers`.
Прямой коннект из backend к произвольным URL — запрещён.

**Elastic IPs для Enterprise:**
Крупные org получают выделенный NAT Gateway IP.
Провайдер добавляет наш IP в свой whitelist → rate limit изолирован от других org.

**Sandbox для тестирования провайдера:**
Перед подключением реального ключа admin тестирует через sandbox:
`POST /api/org/{id}/marketplace/{providerId}/test` с `X-Test-Key` → mock-ответы.

**Marketplace Review процесс:**
1. Провайдер подаёт заявку через форму
2. Автоматический healthcheck: GET `healthcheck_url` с тестовым токеном
3. При успехе → ручное ревью администратора платформы
4. `status = 'active'` → провайдер появляется в Marketplace
5. Весь процесс в `organization_audit_log (action='marketplace_provider_reviewed')`

**Мониторинг расходов по провайдерам:**
Dashboard `/org/{id}/marketplace` показывает:
- Расходы по каждому провайдеру за период
- P95 latency и error rate
- Сравнение с нашим нативным роутингом через OpenRouter

### 5. GDPR — Restriction of Processing

```sql
ALTER TABLE gdpr_requests DROP CONSTRAINT gdpr_requests_type_check;
ALTER TABLE gdpr_requests ADD CONSTRAINT gdpr_requests_type_check
  CHECK (type IN ('erasure', 'export', 'rectification', 'restriction'));

ALTER TABLE user_settings
  ADD COLUMN processing_restricted BOOLEAN DEFAULT FALSE,
  ADD COLUMN restriction_reason    TEXT,
  ADD COLUMN restriction_lifted_by UUID,
  ADD COLUMN restriction_lifted_at TIMESTAMPTZ;
```

При `processing_restricted = TRUE`:
- Все AI-запросы пользователя → 451 Unavailable For Legal Reasons
- API response: `{ "error": "PROCESSING_RESTRICTED", "detail": "..." }`
- При входе пользователь видит неустранимый баннер с объяснением и контактом support
- Снятие ограничения: только через ручное ревью admin в `/admin/users/{id}`
- Запись в `admin_audit_log` + `gdpr_requests (status='completed')`

**Автоочистка кэша при Right to Erasure:**

Расширяем Inngest воркер из v2.8:
```typescript
// После Crypto-Shredding DEK
await Promise.all([
  redis.del(`user:${userId}:*`),           // Redis кэш
  cloudflare.purgeCache({ tags: [`user-${userId}`] }), // CDN
  supabase.replication.invalidate(userId),  // Read Replicas
]);
await gdprRequests.update({ id, status: 'completed', completed_at: now() });
```

Каждый шаг логируется в `gdpr_requests.metadata` для аудита.

### 6. SLA Tiers с финансовыми гарантиями

```sql
CREATE TABLE sla_tiers (
  plan_type         TEXT PRIMARY KEY,
  uptime_sla        NUMERIC NOT NULL,    -- 99.5, 99.9, 99.99
  support_sla_hours INTEGER NOT NULL,    -- 72, 24, 4
  sla_scope         TEXT NOT NULL,       -- только наши 5xx, не провайдерские 502/504
  credit_thresholds JSONB NOT NULL,
    -- [{ "below": 99.9, "credit_pct": 10 }, { "below": 99.0, "credit_pct": 25 }]
  updated_at        TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE sla_credit_drafts (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES organizations(id),
  period          TEXT NOT NULL,        -- '2026-07'
  region          TEXT NOT NULL,        -- 'eu' | 'us'
  actual_uptime   NUMERIC NOT NULL,
  credit_usd      NUMERIC(10,4) NOT NULL,
  status          TEXT DEFAULT 'draft'
                  CHECK (status IN ('draft', 'approved', 'rejected', 'applied')),
  approved_by     UUID,
  approved_at     TIMESTAMPTZ,
  applied_at      TIMESTAMPTZ,
  created_at      TIMESTAMPTZ DEFAULT now()
);
```

**Maker-Checker для SLA кредитов:**
Система автоматически рассчитывает → создаёт `sla_credit_drafts (status='draft')`.
Finance-менеджер рассматривает → `status = 'approved'` → `INSERT INTO billing_events (type='SLA_CREDIT')`.

**SLA scope (важно для юридических документов):**
- Засчитывается: наши 5xx, таймауты БД, недоступность API endpoint
- НЕ засчитывается: 502/504 от OpenAI/Anthropic/OpenRouter, AWS outage, DDoS

**Раздельный SLA по регионам:**
`hourly_sla_rollups` содержит `region TEXT`. Dashboard показывает EU и US отдельно.
Enterprise может иметь разные SLA для разных регионов.

### 7. Pentest Accounts с RLS изоляцией

```sql
CREATE TABLE pentest_accounts (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id          UUID NOT NULL REFERENCES organizations(id),
  created_by      UUID NOT NULL,
  pentest_user_id UUID NOT NULL,      -- Supabase Auth user для пентестера
  expires_at      TIMESTAMPTZ NOT NULL CHECK (expires_at < created_at + INTERVAL '7 days'),
  scope           TEXT[] NOT NULL,    -- ['read:usage', 'write:compare']
  ip_whitelist    INET[],
  sandbox_tenant_id UUID NOT NULL,    -- изолированный tenant для sandbox данных
  auto_delete_at  TIMESTAMPTZ NOT NULL,
  notes           TEXT,
  created_at      TIMESTAMPTZ DEFAULT now()
);
```

**RLS изоляция:**
```sql
-- Пентест-аккаунты видят ТОЛЬКО свой sandbox_tenant_id
CREATE POLICY pentest_isolation ON usage_events
  USING (
    CASE WHEN current_setting('app.is_pentest', true) = 'true'
    THEN tenant_id = current_setting('app.sandbox_tenant_id')::UUID
    ELSE TRUE
    END
  );
```

**Lifecycle:**
- За 24 часа до `expires_at` → email admin org
- При `auto_delete_at`: Crypto-Shredding KMS ключа sandbox_tenant_id → удаление данных sandbox → `INSERT INTO organization_audit_log`
- Досрочный отзыв: `POST /api/org/{id}/pentest/{accountId}/revoke`

### 11. Dynamic Trace Sampling

Tail-based sampling в OTEL Collector:

```yaml
# otel-collector-config.yaml
processors:
  tail_sampling:
    policies:
      - name: errors-policy
        type: status_code
        status_code: { status_codes: [ERROR] }
        sampling_percentage: 100
      - name: slow-spans-policy
        type: latency
        latency: { threshold_ms: 2000 }
        sampling_percentage: 100
      - name: high-cost-policy
        type: string_attribute
        string_attribute: { key: cost_usd, values: ["[0-9]{2,}\\.[0-9]+"] }
        sampling_percentage: 100
      - name: default-policy
        type: probabilistic
        probabilistic: { sampling_percentage: 1 }
```

Конфигурация порогов через feature flags — без перезапуска коллектора.

## Что НЕ входит в v3.0 (→ v3.1)

- **API региональной миграции org** (EU → US) — сложный процесс с полной очисткой источника
- **Client-Side BYOK с trace_token** — фундаментально меняет модель безопасности
- **Региональный fallback read-only дашборд** — покрывается Cross-Region DR частично
- **Федерация данных** (агрегированный мультирегиональный view) — схема `organization_regions` заложена
