## Ключевые архитектурные изменения относительно первоначального плана

### request_logs → вне PostgreSQL

Хранение сырых HTTP-логов (каждый запрос, IP, latency) в PostgreSQL — антипаттерн.
При 100k RPD таблица вырастет до сотен миллионов строк за месяц.

**Решение:**
```
API → OpenTelemetry Collector → Grafana Loki (логи) + Grafana Tempo (трейсы)
```

В PostgreSQL остаётся только:
```sql
CREATE TABLE hourly_sla_rollups (
  hour            TIMESTAMPTZ NOT NULL,
  endpoint        TEXT NOT NULL,
  org_id          UUID,
  p50_ms          NUMERIC,
  p95_ms          NUMERIC,
  p99_ms          NUMERIC,
  success_count   INTEGER,
  error_count     INTEGER,
  PRIMARY KEY (hour, endpoint, org_id)
);
```

Агрегируется из Loki раз в час через Inngest job.

### Crypto-Shredding для GDPR

Вместо перезаписи `prompt_text = '[DELETED]'` — Envelope Encryption:

```
Для каждого пользователя: user_encryption_key (DEK) хранится в KMS
prompt_text = AES-256-GCM(raw_prompt, DEK)
```

При GDPR Erasure:
1. Удаляем DEK из KMS → все зашифрованные данные пользователя становятся нечитаемыми
2. S3 WORM-бэкапы физически остаются, но данные нечитаемы
3. GDPR "право на удаление" выполнено без перезаписи immutable storage

Таблица `user_encryption_keys`: `user_id`, `key_id` (KMS reference), `created_at`, `shredded_at`.

### Vanta/Drata вместо самописного compliance движка

Интеграция с Vanta или Drata через их API:
- Они автоматически собирают evidence из Supabase, GitHub, Vercel, AWS
- Наш `compliance_evidence_snapshots` — дополнительный источник (Terraform state, restore drill results)
- Аудиторы принимают Vanta/Drata отчёты без лишних вопросов

## Что входит в v2.8

### 1. GDPR Tooling (с Crypto-Shredding)

```sql
CREATE TABLE gdpr_requests (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL REFERENCES auth.users(id),
  type            TEXT NOT NULL CHECK (type IN ('erasure', 'export')),
  status          TEXT NOT NULL DEFAULT 'pending'
                  CHECK (status IN ('pending', 'processing', 'completed', 'failed')),
  deadline_at     TIMESTAMPTZ NOT NULL,  -- +30 дней для erasure, +72 часа для export
  completed_at    TIMESTAMPTZ,
  export_url      TEXT,          -- одноразовая ссылка, TTL 24 часа
  export_expires_at TIMESTAMPTZ,
  created_at      TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE user_encryption_keys (
  user_id     UUID PRIMARY KEY REFERENCES auth.users(id),
  key_id      TEXT NOT NULL,    -- KMS Key ID (не сам ключ)
  created_at  TIMESTAMPTZ DEFAULT now(),
  shredded_at TIMESTAMPTZ       -- NULL = ключ активен
);
```

**Right to Erasure:**
1. `POST /api/user/gdpr/delete-request` → создаёт запись в `gdpr_requests`
2. Inngest воркер следит за дедлайном (30 дней)
3. При выполнении: `DELETE key FROM KMS` → `shredded_at = now()`
4. Сохраняем: `billing_events`, `usage_events` (зашифрованные), `organization_audit_log`
5. Удаляем: профиль, `saved_prompts`, `notifications`

**Напоминания (Inngest воркер):**
- За 7 дней до дедлайна → email admin
- За 1 день → email + in-app алерт
- Просрочено → эскалация суперадмину

**Data Export (Right to Portability):**
ZIP с `manifest.json`, структурированные JSON/CSV:
```
export_2026-07-01/
  manifest.json
  usage_history.csv
  billing_history.csv
  saved_prompts.json
  profile.json
```
Одноразовая ссылка, TTL 24 часа. Повторный запрос — через 30 дней.
ZIP шифруется паролем (пользователь получает его отдельно).

**ZDR у LLM-провайдеров:**
Документируем подписанные Zero Data Retention соглашения с Anthropic, OpenAI, OpenRouter
в `compliance_evidence_snapshots`. Используем ZDR API эндпоинты где доступны.

### 2. SOC 2 Type II Readiness

**Vanta/Drata интеграция:**
Подключаем через их API. Они автоматически:
- Проверяют MFA у всех admin
- Собирают evidence из GitHub (code review, branch protection)
- Мониторят Supabase RLS policy
- Собирают данные об encryption at rest

**Наш вклад в `compliance_evidence_snapshots`:**
```sql
CREATE TABLE compliance_evidence_snapshots (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  period          TEXT NOT NULL,    -- '2026-Q2', '2026-07'
  control         TEXT NOT NULL,    -- 'restore_drill', 'zdr_agreements', 'terraform_audit'
  status          TEXT NOT NULL CHECK (status IN ('complete', 'missing', 'partial')),
  evidence        JSONB NOT NULL,   -- структурированные данные для аудиторов
  encrypted_payload TEXT,           -- KMS-зашифрованные чувствительные детали
  created_at      TIMESTAMPTZ DEFAULT now()
);
```

**Ежемесячный SOC 2 чек-лист (Inngest cron):**
Проверяет наличие evidence по всем контролям.
При `status = 'missing'` → алерт ответственным + запись в snapshot.

**`security.txt`:**
```
GET /.well-known/security.txt
→ contact, policy URL, encryption key, preferred-languages
```
Responsible Disclosure policy: `/security/responsible-disclosure`.

**Dependabot + Trivy в CI/CD:**
- Dependabot: автоматические PR для уязвимых зависимостей
- Trivy: сканирование Docker образов в GitHub Actions
- Блокировка мержа при Critical/High уязвимостях (не WARNING)

### 3. Restore Drills (DR Testing)

Раз в квартал (Vercel Cron или ручной запуск):
1. Разворачиваем staging из последнего production S3-бэкапа
2. Запускаем smoke-test suite против staging
3. Фиксируем: RTO (время восстановления), RPO (точка восстановления), результат тестов
4. Запись в `compliance_evidence_snapshots (control='restore_drill')`

SOC 2 аудиторы запрашивают эти протоколы в первую очередь.

### 4. Terraform Provider

`terraform-provider-new-era` на Terraform Registry.

**Ресурсы:**
```hcl
resource "newera_organization" "acme" { ... }
resource "newera_organization_sso" "acme_okta" { ... }
resource "newera_scim_token" "acme_scim" { ... }
resource "newera_feature_flag" "new_ui" { ... }
resource "newera_api_key" "ci_key" { ... }
```

**`terraform import` для всех ресурсов:**
```bash
terraform import newera_organization.acme org_uuid
terraform import newera_organization_sso.acme_okta org_uuid
```
Без Import нельзя взять под управление существующие ресурсы.

**Audit в `compliance_evidence_snapshots`:**
Каждый `terraform apply` логируется: инициатор (API key), список изменённых ресурсов,
результат. Change management evidence для SOC 2.

**Acceptance тесты в CI:**
Реальные ресурсы создаются в sandbox, проверяются, удаляются.
Запуск при каждом PR в репозиторий провайдера.

### 5. Observability

**OpenTelemetry pipeline:**
```
App → OTEL Collector → Grafana Loki (logs) + Grafana Tempo (traces) + Prometheus (metrics)
```

**`trace_id` сквозной:**
- Генерируется в Middleware при каждом запросе
- Передаётся в заголовках к LLM-провайдерам (`X-Request-Id`) где поддерживается
- Добавляется в Sentry error context
- Хранится в `usage_events.trace_id`

**PII Redaction:**
- Sentry `beforeSend`: удаляем email, IP, токены из error context
- OTEL AttributeProcessor: `redact` для span атрибутов с PII паттернами

**Error Budget / Burn Rate:**
SLO: 99.5% успешных запросов за 30 дней (бюджет = 0.5% = ~3.6 часа).

При Burn Rate > 5x нормы за последний час → алерт команде в Slack.
При Burn Rate > 14.4x → feature freeze, все руки на стабилизацию.

Метрики вычисляются из `hourly_sla_rollups`.

**SLA Dashboard для организаций:**
В `/org/{id}/dashboard` — uptime и latency именно их трафика, не средние по платформе.

**Публичная статус-страница:**
`/status` расширяется: подписка на уведомления (email/Slack webhook).
При автоматическом определении деградации (error rate > порога) → уведомление подписчикам.

- За 30, 14 и 3 дня → email admin организации
- Два сертификата одновременно для zero-downtime ротации

### 7. Read-Only режим для maintenance

Feature flag `platform_readonly` в `feature_flags`.
При включении:
- Блокируются все мутации (POST/PUT/PATCH/DELETE) кроме чтения
- API возвращает `503 Maintenance` с `Retry-After` заголовком
- Frontend показывает maintenance баннер
- Исключение: `/api/health`, `/api/user/logout`

Используется при аудитах и опасных миграциях.
