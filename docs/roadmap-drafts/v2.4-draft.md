## Что входит в v2.4

### 1. Entitlements Engine — production-ready

**Таблица plan_entitlements с версионированием:**
```sql
CREATE TABLE plan_entitlements (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  plan_type    TEXT NOT NULL,
  version      INTEGER NOT NULL DEFAULT 1,
  key          TEXT NOT NULL,     -- 'daily_requests', 'allowed_models', 'cost_per_minute_usd'
  value        JSONB NOT NULL,
  effective_from TIMESTAMPTZ DEFAULT now(),
  created_by   UUID,

  UNIQUE (plan_type, version, key)
);

-- История изменений
CREATE TABLE plan_entitlements_history (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  plan_type    TEXT NOT NULL,
  key          TEXT NOT NULL,
  old_value    JSONB,
  new_value    JSONB,
  changed_by   UUID NOT NULL,
  changed_at   TIMESTAMPTZ DEFAULT now(),
  reason       TEXT
);
```

**Grandfathering:** Пользователь при регистрации получает `entitlements_version`
в `user_settings`. При изменении тарифов новая версия применяется только к
новым пользователям — существующие остаются на своих условиях.

**Кэш в памяти + pg_notify инвалидация:**
```typescript
// При старте приложения — загрузка в память
let entitlementsCache: EntitlementsMap = await loadEntitlements();

// При изменении тарифов администратором → pg_notify 'entitlements_changed'
// Все инстансы приложения перезагружают кэш
supabase.channel('entitlements').on('postgres_changes', ..., () => {
  entitlementsCache = await loadEntitlements();
});
```

Middleware читает из памяти — нулевая нагрузка на БД при каждом запросе.

**Enterprise custom overrides:**
Поле `custom_overrides JSONB` в `user_settings` переопределяет отдельные лимиты
для конкретного пользователя. Приоритет custom > plan. Изменяется только
администратором через `/admin/users/[id]`, пишется в `admin_audit_log`.

### 2. Auto-recharge — idempotent и race-free

Пользователь настраивает: «пополнять на $10 когда баланс < $2».

**Защита от гонок — Advisory Lock:**
```sql
SELECT pg_advisory_xact_lock(user_id::bigint);
-- только после получения блокировки проверяем баланс
SELECT balance_usd FROM user_settings WHERE user_id = $userId FOR UPDATE;
-- если balance < threshold И нет pending auto-recharge → инициируем
```

**Идемпотентность:**
При инициировании пишем `billing_events(type=AUTO_RECHARGE_INITIATED,
stripe_invoice_id=...)`. UNIQUE(stripe_invoice_id) не даёт обработать
одно событие Stripe дважды.

**Пауза auto-recharge:**
Поле `auto_recharge_paused BOOLEAN DEFAULT FALSE` в `user_settings`.
Кнопка «Пауза» в личном кабинете — настройки сохраняются, списание
не происходит. Кнопка «Возобновить» включает обратно.

### 3. Dunning Process — полный цикл

Новая таблица:
```sql
CREATE TABLE dunning_status (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID NOT NULL UNIQUE REFERENCES auth.users(id),
  current_stage   TEXT NOT NULL DEFAULT 'none',
    -- 'none' | 'day1_notified' | 'day3_retried' | 'day7_downgraded' | 'recovered'
  last_action_at  TIMESTAMPTZ,
  payment_failed_at TIMESTAMPTZ,
  stripe_invoice_id TEXT,
  created_at      TIMESTAMPTZ DEFAULT now()
);
```

**Статус past_due (отдельный от free и pro):**
`user_settings.plan_status` TEXT: `'active'` | `'past_due'` | `'suspended'`

В период `past_due` (Day 1–7) пользователь сохраняет Pro-функции.
В UI — неустранимый баннер «Проблема с оплатой → Обновить карту».

**Шаги:**
- Day 0: платёж не прошёл → `plan_status = past_due`, email пользователю
- Day 1: email «Напоминание об оплате»
- Day 3: Stripe повторная попытка + email о попытке
- Day 7: downgrade на `free`, `plan_status = active` (free), email с инструкцией восстановления
- При успешном платеже в любой день: `plan_status = active (pro)`, `current_stage = recovered`

**Идемпотентность шагов:**
Перед каждым шагом проверяем `last_action_at` — если шаг уже выполнен
сегодня, пропускаем. Prevents race condition при параллельных cron-запусках.

**Ручное управление в admin:**
`/admin/users/[id]` → кнопки «Retry Payment», «Reset Dunning», «Force Downgrade».
Все действия пишутся в `admin_audit_log`.

### 4. Reconciliation — только алертинг (изменение подхода)

**Важное архитектурное решение:**
Reconciliation НЕ автоматически корректирует `balance_usd`.
Автокоррекция скрывает баги в транзакционной логике.

Ежедневный cron:
```sql
SELECT user_id,
  SUM(amount_usd) AS ledger_sum,
  us.balance_usd AS cached_balance,
  ABS(SUM(amount_usd) - us.balance_usd) AS discrepancy
FROM billing_events be
JOIN user_settings us USING (user_id)
GROUP BY user_id, us.balance_usd
HAVING ABS(SUM(amount_usd) - us.balance_usd) > 0.001;
```

При обнаружении расхождения:
1. `INSERT INTO billing_events (type='RECONCILIATION_ALERT', metadata={discrepancy, ledger_sum, cached_balance})`
2. `INSERT INTO notifications` для всех admin
3. НЕ трогаем `balance_usd` — ручной разбор администратором

### 5. Email Jobs — Dead Letter Queue

Расширяем `email_jobs`:
```sql
ALTER TABLE email_jobs ADD COLUMN
  attempt_count  INTEGER DEFAULT 0,
  last_error     TEXT,
  dlq_at         TIMESTAMPTZ;  -- NULL = не в DLQ
```

После 3 неудачных попыток (exponential backoff: 1 мин → 5 мин → 30 мин):
- Запись переходит в Dead Letter: `dlq_at = now()`
- `INSERT INTO notifications` для admin: «Письмо не доставлено: [тип, user_id]»
- В `/admin` — раздел «Failed Emails» с кнопкой «Retry»

### 6. Мультивалютность через Stripe API

**Изменение подхода:** таблица `exchange_rates` убирается из плана.

Stripe API возвращает цены в `preferred_currency` пользователя.
В UI показываем: «≈ €9.20 (по курсу Stripe)» с пометкой что итоговая
сумма зависит от курса банка.

Внутри системы (в Ledger, billing_events) — только USD.
`preferred_currency` в `user_settings` используется только для отображения.

### 7. Spending Dashboard

Таблица `daily_usage_rollups`:
```sql
CREATE TABLE daily_usage_rollups (
  user_id     UUID NOT NULL,
  date        DATE NOT NULL,
  model_key   TEXT NOT NULL,
  total_cost  NUMERIC(10,6) DEFAULT 0,
  total_tokens INTEGER DEFAULT 0,
  request_count INTEGER DEFAULT 0,

  PRIMARY KEY (user_id, date, model_key)
);
```

Обновляется инкрементально при каждой записи в `usage_events` через триггер.
Dashboard читает только эту лёгкую таблицу — не `usage_events` напрямую.

Бар-чарт расходов по дням за 30 дней с разбивкой по моделям.
Прогноз на конец месяца (линейная экстраполяция).

### 9. PII Masking на уровне БД

Regex-маскировка `prompt_text` перед записью в `usage_events`:
- Email: `[email]`
- Телефон: `[phone]`
- API-ключи (sk-..., Bearer ...): `[api_key]`
- Номера карт (16 цифр): `[card]`

Количество замен логируется в `usage_events.pii_masked_count INTEGER`.
Содержимое не логируется.
