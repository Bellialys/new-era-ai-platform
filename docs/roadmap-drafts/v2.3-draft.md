## Что входит в v2.3

### 1. Stripe интеграция — pay-as-you-go

Пользователь пополняет баланс через Stripe Checkout.
После успешной оплаты Stripe отправляет вебхук `checkout.session.completed`.

Новые поля в `user_settings`:
```
stripe_customer_id   TEXT    UNIQUE  -- Stripe customer ID
balance_usd          NUMERIC(10,4) DEFAULT 0  -- кэш, источник истины — billing_events
plan_type            TEXT DEFAULT 'free'       -- из v2.2
plan_expires_at      TIMESTAMPTZ               -- для будущих подписок
payment_method_last4 TEXT                      -- последние 4 цифры карты из Stripe
payment_method_brand TEXT                      -- 'visa', 'mastercard' и т.д.
```

**Stripe Customer Portal:**
`GET /api/billing/portal` → редирект на Stripe Customer Portal.
Пользователь обновляет карту, скачивает инвойсы — без нашего кода.

**Тарифные планы (plan_type из v2.2):**
- `free` — 50 запросов/день, только модели с price_label = 'free'
- `pro` — без ограничений по моделям, pay-as-you-go или фикс
- `enterprise` — кастомные лимиты (ручная настройка администратором)

Смена плана: `/api/billing/upgrade` → Stripe Checkout.

### 2. billing_events — append-only ledger

Все изменения баланса — только через эту таблицу. `balance_usd` в `user_settings`
является кэшем и восстанавливается из `billing_events` при расхождении.

```sql
CREATE TABLE billing_events (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id          UUID NOT NULL REFERENCES auth.users(id),
  type             TEXT NOT NULL,
    -- DEPOSIT        пополнение через Stripe
    -- RESERVE        резервирование перед AI-вызовом
    -- CAPTURE        списание фактической стоимости после ответа
    -- RELEASE        освобождение резерва (отмена или истечение)
    -- REFUND         возврат администратором
    -- ADJUSTMENT     ручная корректировка администратором
  amount_usd       NUMERIC(10,6) NOT NULL,  -- положительное = приход, отрицательное = расход
  reference_id     TEXT,                    -- usage_event_id, stripe_payment_intent_id и т.д.
  stripe_event_id  TEXT UNIQUE,             -- идемпотентность Stripe webhooks
  metadata         JSONB DEFAULT '{}',
  created_at       TIMESTAMPTZ DEFAULT now(),
  created_by       UUID                     -- NULL = система, UUID = admin (для ADJUSTMENT/REFUND)
);

CREATE INDEX idx_billing_events_user ON billing_events(user_id, created_at DESC);
CREATE INDEX idx_billing_events_type ON billing_events(user_id, type, created_at DESC);
```

**UNIQUE(stripe_event_id)** гарантирует: даже если Stripe доставит вебхук
дважды, второй INSERT упадёт с ошибкой constraint — двойного начисления не будет.

### 3. Предавторизация (Pre-authorization)

Перед вызовом OpenRouter резервируем estimated cost:
```
estimated_cost = max_tokens × price_per_token_model
```

Шаги:
1. `INSERT INTO billing_events (type=RESERVE, amount=-estimated_cost)`
2. `UPDATE user_settings SET balance_usd = balance_usd - estimated_cost WHERE balance_usd >= estimated_cost`
   — если условие не выполнено → 402 BALANCE_INSUFFICIENT
3. Вызов OpenRouter
4. При успехе: `INSERT (type=CAPTURE, amount=actual_cost - estimated_cost)` + корректировка
5. При ошибке: `INSERT (type=RELEASE, amount=+estimated_cost)`

Всё в рамках одной транзакции с `SELECT ... FOR UPDATE` на строку `user_settings`.

**Orphaned Reservations Reaper:**
Vercel Cron раз в минуту ищет RESERVE-события старше 5 минут без парного CAPTURE/RELEASE
и автоматически создаёт RELEASE. Защита от зависших резервов при падении сервера.

### 4. Email уведомления через Resend

Supabase Edge Function `send-billing-email`:
- Подтверждение пополнения баланса
- Алерт при достижении 80% / 100% бюджета
- Месячный чек (PDF)

PDF генерируется в Serverless Function (Node.js runtime, не Edge) —
Edge имеет ограничения на бинарные зависимости.

Триггеры: через `billing_events` после DEPOSIT, или из Cost Alerts (v2.2).

---

### 5. Weighted rate limiting

Расширяем `rate-limit-config.ts`:
```typescript
export const COST_RATE_LIMITS = {
  // максимальная стоимость за минуту на пользователя (USD)
  free: 0.10,
  pro:  2.00,
  enterprise: null,  // без лимита
};
```

Дорогие модели потребляют «кредиты» быстрее. Дешёвые модели —
пользователь делает много запросов, дорогие — мало. Работает поверх
существующего rate limiter через Upstash.

### 6. Глобальный бюджет платформы

Новая таблица:
```sql
CREATE TABLE platform_settings (
  key    TEXT PRIMARY KEY,
  value  JSONB NOT NULL,
  updated_at TIMESTAMPTZ DEFAULT now(),
  updated_by UUID  -- admin_id
);

-- Пример записи:
-- key = 'global_budget', value = { "monthly_usd": 500, "current_usd": 123.45 }
```

При достижении 100% глобального бюджета: блокируются запросы от `free`
и анонимных пользователей. `pro` и `enterprise` продолжают работать.

**Graceful degradation:**
- API: `503 Service Unavailable` + заголовок `Retry-After: 86400`
- Frontend: страница с объяснением что бесплатные лимиты на сегодня исчерпаны
  + CTA «Перейти на Pro»

Алерты администратору при 75% / 90% / 100% глобального бюджета.


Фильтры: `?type=DEPOSIT,CAPTURE&from=2026-07-01&to=2026-07-31`.

### 8. Reconciliation job

Vercel Cron ежедневно в 02:00 UTC:
```sql
SELECT user_id, SUM(amount_usd) AS ledger_balance
FROM billing_events
WHERE type IN ('DEPOSIT', 'CAPTURE', 'RELEASE', 'REFUND', 'ADJUSTMENT')
GROUP BY user_id
HAVING ABS(SUM(amount_usd) - (
  SELECT balance_usd FROM user_settings WHERE user_id = billing_events.user_id
)) > 0.001;
```

При расхождении:
1. Запись в `admin_audit_log` с деталями
2. Автоматическая корректировка `balance_usd` до значения из ledger
3. Уведомление администратору в `notifications`

---

### 9. Ручная корректировка баланса (Admin)

В `/admin/users/[userId]` → кнопка «Корректировать баланс»:
- Ввод суммы (положительная = пополнение, отрицательная = списание)
- Обязательный комментарий
- `INSERT INTO billing_events (type=ADJUSTMENT, created_by=admin_id)`
- `UPDATE user_settings SET balance_usd = balance_usd + amount`
- Запись в `admin_audit_log`

---

### 10. Алерты по биллинговым аномалиям

Мониторим и пишем в `notifications` для `role=admin`:
- Повторный Stripe вебхук (constraint violation на stripe_event_id)
- Расхождение balance после reconciliation
- Пользователь с balance < 0 (не должно происходить)
- Orphaned reservation освобождена Reaper-ом
