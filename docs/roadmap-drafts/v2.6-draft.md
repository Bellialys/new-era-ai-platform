## Что входит в v2.6

### 1. Team Workspaces — Organizations

```sql
CREATE TABLE organizations (
  id                   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name                 TEXT NOT NULL,
  slug                 TEXT NOT NULL UNIQUE,
  balance_usd          NUMERIC(10,4) DEFAULT 0,    -- кэш, источник истины — billing_events
  plan_type            TEXT DEFAULT 'team',
  monthly_budget_usd   NUMERIC(10,4),
  current_month_cost   NUMERIC(10,4) DEFAULT 0,
  billing_month        DATE,
  allowed_email_domains TEXT[] DEFAULT '{}',       -- автодобавление по домену
  owner_id             UUID NOT NULL REFERENCES auth.users(id),
  created_at           TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE organization_members (
  org_id      UUID NOT NULL REFERENCES organizations(id),
  user_id     UUID NOT NULL REFERENCES auth.users(id),
  role        TEXT NOT NULL CHECK (role IN ('owner', 'admin', 'member', 'viewer')),
  joined_at   TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (org_id, user_id)
);

-- Под-бюджеты на участника или проект
CREATE TABLE organization_member_limits (
  org_id              UUID NOT NULL REFERENCES organizations(id),
  user_id             UUID NOT NULL REFERENCES auth.users(id),
  monthly_budget_usd  NUMERIC(10,4) NOT NULL,
  current_month_cost  NUMERIC(10,4) DEFAULT 0,
  billing_month       DATE,
  PRIMARY KEY (org_id, user_id)
);

-- Audit log организации
CREATE TABLE organization_audit_log (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL REFERENCES organizations(id),
  actor_id    UUID NOT NULL,  -- кто совершил действие
  action      TEXT NOT NULL,
    -- 'member_invited', 'member_removed', 'role_changed', 'prompt_deleted',
    -- 'budget_changed', 'key_revoked', 'ip_whitelist_changed'
  target_id   TEXT,           -- user_id, prompt_id, api_key_id и т.д.
  metadata    JSONB,
  created_at  TIMESTAMPTZ DEFAULT now()
);
```

**Биллинг на организацию:**
Все члены команды тратят из общего `balance_usd` организации.
Advisory Lock при списании:
```sql
SELECT pg_advisory_xact_lock(hashtext(org_id::text));
SELECT balance_usd FROM organizations WHERE id = $orgId FOR UPDATE;
-- если balance < cost → 402 ORG_BALANCE_INSUFFICIENT
```

**Разбивка расходов по участникам:**
`usage_events` получает новые поля: `organization_id UUID`, `member_user_id UUID`.
Owner видит в Dashboard: кто сколько потратил, какими моделями пользовался.

**Уведомления 80% / 100% баланса организации:**
При пересечении порогов → уведомление всем `admin` и `owner` в `notifications`.

**Под-бюджеты на участника:**
Owner устанавливает лимит на члена в `organization_member_limits`.
При превышении — блокировка запросов от этого участника (403 MEMBER_BUDGET_EXCEEDED),
алерт всем admin + owner.

**Автодобавление по домену:**
Если `allowed_email_domains = ['acme.com']`, пользователь с email `user@acme.com`
при регистрации автоматически добавляется как `member`.

**2FA обязательный для owner:**
При создании организации или получении роли `owner` — принудительный
TOTP enrollment. Без 2FA роль `owner` недоступна.

### 2. Shared Prompts & Arenas

```sql
CREATE TABLE saved_prompts (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id        UUID NOT NULL REFERENCES organizations(id),
  created_by    UUID NOT NULL REFERENCES auth.users(id),
  title         TEXT NOT NULL,
  prompt_text   TEXT NOT NULL,
  model_settings JSONB DEFAULT '{}',
  permission    TEXT NOT NULL CHECK (permission IN ('read', 'write', 'publish')),
    -- read:    только автор и owner видят и используют
    -- write:   все члены команды могут изменять
    -- publish: опубликован для всей организации (только owner/admin)
  tags          TEXT[] DEFAULT '{}',
  status        TEXT DEFAULT 'draft' CHECK (status IN ('draft', 'published')),
  created_at    TIMESTAMPTZ DEFAULT now(),
  updated_at    TIMESTAMPTZ DEFAULT now()
);
```

**Сканирование секретов при сохранении:**
Те же regex что PII masking из v2.4 — API-ключи (`sk-...`, `Bearer ...`),
токены, credentials. Блокировка сохранения при обнаружении с сообщением
«Промпт содержит чувствительные данные. Удалите их перед сохранением.»

**SDK методы:**
```typescript
await client.org.prompts.list({ tags: ['support'] });
await client.org.prompts.run(promptId, { variables: { product: 'Pro' } });
```

### 3. IP Whitelisting для Enterprise API Keys

Поле `allowed_ips CIDR[]` в `api_keys` (тип `inet[]` в PostgreSQL).
Поддержка IPv4, IPv6 и CIDR-диапазонов.

In-memory кэш разрешённых IP: загружается при старте, инвалидируется через
`pg_notify 'api_keys_changed'` при изменении whitelist.

Обработка `X-Forwarded-For`: доверяем только известным прокси
(Vercel Edge IP-диапазоны), не принимаем произвольный заголовок.

Все попытки с неразрешённых IP логируются в `api_key_audit`.

### 4. Python SDK

Пакет `new-era-py` на PyPI:
```python
from new_era import NewEra, BudgetExceededError, RateLimitError

client = NewEra(api_key="ne_sk_live_...")

# Сравнение моделей
result = client.compare(
    prompt="Explain quantum entanglement",
    models=["gpt-4o", "claude-3-5-sonnet"],
    idempotency_key="req-123"
)

# Streaming
for chunk in client.compare(prompt="...", stream=True):
    print(chunk.delta, end="")

# Промпты организации
prompts = client.org.prompts.list(tags=["support"])
```

Паритет функций с TypeScript SDK.
Типизированные исключения: `BudgetExceededError`, `RateLimitError`, `AuthError`.
Автогенерация типов из OpenAPI схемы.

### 5. CLI `npx @new-era/cli`

```bash
new-era login          # Device Flow (RFC 8628) — работает по SSH
new-era compare "Prompt" --models gpt-4o,claude-3-5-sonnet
new-era models list
new-era usage --from 2026-06-01 --to 2026-07-01
new-era org members list
new-era org invite user@company.com --role member
new-era prompts upload ./prompt.txt --tags support,v2
new-era webhooks test --endpoint https://my.app/hook
```

**Device Authorization Grant (RFC 8628):**
```
$ new-era login
Open this URL to authenticate: https://app.domain/device?code=ABCD-1234
Waiting for authorization...
✓ Logged in as user@company.com
```
Разработчик открывает ссылку на любом устройстве (включая телефон).
Работает в headless SSH средах без браузера на сервере.

### 6. Streaming SSE — Production-ready

**Heartbeats:**
Сервер отправляет `: ping\n\n` каждые 15 секунд.
Защита от агрессивного закрытия соединений прокси (Nginx, Cloudflare,
AWS ALB обычно рвут соединения через 60-100 сек без активности).

**Last-Event-ID — возобновление потока:**
Каждый SSE-чанк получает `id: chunk_N`.
При разрыве клиент переподключается с заголовком `Last-Event-ID: chunk_N`.
Сервер продолжает поток с нужного места (буфер в Redis на 60 сек).

**Async Iterator в SDK:**
```typescript
for await (const chunk of client.compare({ prompt: "...", stream: true })) {
  process.stdout.write(chunk.delta);
}
```

### 7. Batch Webhooks — Safe Partial Success

Опциональный `batch_size INTEGER DEFAULT 1` в `webhook_endpoints`.
При `batch_size > 1` события агрегируются в один запрос (до 50 в батче).

**Каждое событие имеет уникальный `event_id`.**

Клиент может вернуть частичный успех:
```json
{
  "processed_event_ids": ["evt_1", "evt_2", "evt_5"]
}
```
Сервер помечает эти события как доставленные, остальные уходят на retry.
Если клиент вернул HTTP 200 без тела — весь батч считается доставленным.

Webhook события организации: `member.invited`, `member.removed`,
`organization.balance.low`, `organization.budget.exceeded`.

### 8. Sandbox — расширенная эмуляция

Заголовки для эмуляции:
```
X-Mock-Latency: 3000        # искусственная задержка в мс
X-Mock-Error-Rate: 0.3      # 30% запросов вернут случайную ошибку
X-Mock-Error-Type: timeout  # конкретный тип ошибки (timeout, rate_limit, budget)
```

Разработчики тестируют circuit breakers, retry логику и обработку ошибок
без реальных LLM и без расходов.

Строгая изоляция sandbox от production: отдельная Supabase schema,
RLS не позволяет sandbox-запросам видеть production-данные.
