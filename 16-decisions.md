# DEC-001 - Вести документацию в Markdown в корне репозитория

## Статус

```text
Accepted
# решение принято
```

## Решение

Документацию проекта вести в формате Markdown `.md`.

Основное расположение документации на текущем этапе:

```text
root repository
# документация лежит в корне репозитория
```

Пример:

```text
00-readme.md
# входной файл проектной документации

14-roadmap.md
# главный порядок версий

16-decisions.md
# журнал решений

archive/23-documentation-audit-deep.md
# углублённый аудит документации (перенесён в archive/)
```

Папку `docs/` не использовать как обязательную структуру на текущем этапе.

## Контекст

Проект связан с кодом, GitHub, Next.js, Supabase и OpenRouter. Для такой документации удобнее использовать обычные Markdown-файлы, а не Word или PDF.

Сейчас вся проектная документация уже лежит в корне репозитория с числовыми префиксами.

## Причина

Markdown выбран потому что:

- хорошо открывается в VS Code;
- красиво отображается на GitHub;
- удобно хранится в Git;
- изменения легко сравнивать;
- подходит для технической документации;
- не создаёт тяжёлые бинарные файлы.

Корень репозитория выбран потому что:

- проект пока небольшой;
- файлы `00-23` удобно видеть сразу;
- пользователю проще заменять и открывать файлы;
- не нужно переносить уже созданную структуру.

## Последствия

Все ключевые документы проекта ведутся в `.md` файлах в корне репозитория.

Папку `docs/` можно создать позже, если документация станет слишком большой.

## Когда пересмотреть

Можно пересмотреть, если позже понадобится:

- отдельный сайт документации;
- автоматическая генерация docs;
- разделение пользовательской и developer-документации;
- слишком много файлов в корне репозитория.

Возможные варианты на будущее:

```text
Docusaurus
# сайт документации

Nextra
# документация на Next.js

Mintlify
# developer documentation

docs/
# отдельная папка документации, если корень станет перегружен
```

# DEC-002 - Делать проект поэтапно через MVP

## Статус

```text
Accepted
# решение принято
```

## Решение

Проект разрабатывать маленькими рабочими этапами.

Сначала делается документация, потом Next.js основа, потом Static UI MVP, потом OpenRouter, Supabase, Voting, History и Deploy.

## Контекст

Проект большой. В нём планируются:

- Prompt Arena;
- Code Arena;
- Multi Model Battle;
- Judge Mode;
- Leaderboard;
- Accounts and Profiles;
- Admin Panel and Limits;
- Code Arena Runner;
- AI Team Mode.

Если начать всё сразу, проект станет хаотичным и может не дойти до рабочей версии.

## Причина

MVP позволяет быстро получить работающий результат:

```text
Пользователь вводит задачу.
# основной сценарий

Выбирает 2-3 модели.
# сравнение моделей

Получает ответы рядом.
# главная ценность проекта

Выбирает лучший ответ.
# появляется основа для статистики
```

## Последствия

Сложные функции не добавляются в первый MVP.

Сначала делаем:

```text
Prompt Arena
# основа продукта

OpenRouter
# реальные AI-ответы

Supabase
# сохранение задач и ответов

Voting
# сохранение выбора лучшего ответа

History
# просмотр прошлых сравнений
```

Потом добавляем остальные режимы.

## Когда пересмотреть

Не пересматривать до версии `v1.0 Stable Prompt Arena`.

```text
# решение принято
```

## Контекст


- проще Code Arena;
- проще Judge Mode;

- подходит для первого рабочего MVP.

## Последствия

Первые основные страницы и API:

```text

# главная страница

/arena

# список доступных моделей

/api/compare
# сравнение ответов моделей

/api/vote
# сохранение голоса за лучший ответ
```

Важно:

```text
v0.3
# только локальный UI-выбор победителя

v0.6
# сохранение голоса через /api/vote
```

## Когда пересмотреть

Не пересматривать до завершения `v1.0`.

# DEC-004 - Использовать Next.js

## Статус

```text
Accepted
# решение принято
```

## Решение

Использовать **Next.js** как основной frontend и backend layer проекта.

## Контекст

Проекту нужны:

- страницы сайта;
- интерактивный UI;
- server-side API routes;
- безопасная работа с OpenRouter API key;
- подключение к Supabase;
- деплой на Vercel;
- TypeScript.

## Причина

Next.js подходит для MVP потому что:

- frontend и backend routes находятся в одном проекте;
- хорошо работает с Vercel;
- поддерживает TypeScript;
- удобно развивать проект постепенно;
- можно не создавать отдельный backend на старте.

## Последствия

Базовая структура проекта:

```text
src/app/
# страницы и маршруты Next.js

src/app/api/
# server-side API routes

src/components/
# UI компоненты

src/lib/
# серверная логика, клиенты API и helpers

src/types/
# TypeScript-типы
```

## Когда пересмотреть

Если проект вырастет и API routes перестанут справляться, можно вынести backend отдельно.

Возможные варианты:

```text
NestJS
# отдельный backend на Node.js

FastAPI
# отдельный backend на Python

Queue worker
# отдельная обработка тяжёлых задач
```

# DEC-007 - Использовать OpenRouter для доступа к AI-моделям

## Статус

```text
Accepted
# решение принято
```

## Решение

Использовать **OpenRouter API** как единый слой доступа к нескольким AI-моделям.

## Контекст

Главная идея проекта - сравнивать разные AI-модели. Для этого нужен один API-провайдер, через который можно быстро подключать разные модели.

## Причина

OpenRouter подходит потому что:

- даёт доступ к нескольким AI-моделям через один API;

## Последствия

OpenRouter API key хранится только server-side:

```text
.env.local

Vercel Environment Variables
# production
```

Frontend не должен напрямую обращаться к OpenRouter.

## Когда пересмотреть

Если нужна будет собственная маршрутизация моделей или прямые контракты с провайдерами.

# DEC-008 - Единый стиль имён API и базы

## Статус

```text
Accepted
# решение принято
```

## Решение

Использовать разные стили имён для разных слоёв:

```text
API JSON и TypeScript = camelCase
# modelIds, taskId, responseId, modeSlug

Supabase PostgreSQL = snake_case
# model_id, task_id, model_response_id, mode_slug

Mode slug = kebab-case
# prompt-arena, code-arena, multi-model-battle
```

## Контекст

Проект использует TypeScript, API JSON и PostgreSQL. Если смешивать стили, route handlers станут путаными.

## Причина

Такой подход проще:

- TypeScript и React код читаются в `camelCase`;
- SQL остаётся в привычном `snake_case`;
- slug-и URL-friendly и единообразные в `kebab-case`.

## Последствия

Документация и код должны использовать:

```text
modelIds
# API JSON

taskId
# API JSON

responseId
# API JSON

modeSlug
# API JSON

prompt-arena
# значение режима
```

А база должна использовать:

```text
model_id
task_id
model_response_id
```

Для таблицы `votes` актуальная связь с ответом хранится именно как:

```text
model_response_id
# внешний ключ на model_responses.id
```

Старое имя для голосов не используется:

```text
response_id
# не применять в votes

winner
# не применять как vote_type
```

Актуальный лучший голос:

```text
best
# текущее значение vote_type для выбора лучшего ответа
```

База также использует:

```text
mode_slug
vote_type
```

## Когда пересмотреть

Не пересматривать до `v1.0`, иначе легко сломать API и SQL-схему.

# DEC-010 - Разделить Supabase model UUID и fallback OpenRouter model keys

## Статус

```text
Accepted
# решение принято для v0.5
```

## Решение

В основном `v0.5` режиме поле `modelIds` содержит UUID из таблицы `public.models`.

Пример:

```json
{
  "modelIds": [
    "11111111-1111-4111-8111-111111111111",
    "22222222-2222-4222-8222-222222222222"
  ]
}
```

```text
models.id
# публичный UUID, который можно отправлять с frontend

models.model_key
# технический OpenRouter ID, который должен оставаться на backend
```

Если Supabase catalog не настроен, временно недоступен или вернул пустой список, `/api/models` использует hardcoded fallback. В fallback mode `modelIds` равны OpenRouter model keys из server-side allowlist.

Fallback допустим только для устойчивости MVP и локальной разработки. Он не даёт пользователю право отправлять произвольный provider key.

## Контекст

В текущем MVP Supabase уже подключён как основной источник `models`, `tasks` и `model_responses`.

Однако проект должен продолжать работать локально и во время временных проблем базы, поэтому hardcoded allowlist остаётся аварийным fallback.

Для production-архитектуры нельзя позволять frontend напрямую управлять произвольным OpenRouter model key.

## Причина

Такой переход нужен для:

- контроля расходов;
- безопасного отключения моделей;
- будущей админ-панели;
- нормальной связи ответов с таблицей `models`;
- защиты от произвольной подстановки модели.

## Последствия

В `v0.5` контракт `/api/compare` поддерживает два безопасных источника selection IDs:

```text
Supabase mode: modelIds = public.models.id UUID
# основной режим

Fallback mode: modelIds = OpenRouter model keys из hardcoded allowlist
# только если Supabase catalog недоступен
```

## Когда пересмотреть

После `v0.5.3 Voting MVP stabilization` и перед public production deploy проверить, можно ли сделать Supabase catalog обязательным, оставив fallback только для local/dev.

# DEC-011 - Проверять modeSlug уже в v0.4.1

## Статус

```text
Accepted
# решение принято
```

## Решение

Backend route `/api/compare` должен валидировать `modeSlug`.

В текущей версии разрешён только:

```text
prompt-arena
# единственный активный режим в MVP
```

Если `modeSlug` отсутствует, backend использует `prompt-arena` по умолчанию.

Если передан другой `modeSlug`, backend возвращает ошибку `INVALID_MODE_SLUG`.

## Причина

Проект будет иметь несколько режимов. Лучше сразу контролировать режим на backend, чтобы позже не смешать Prompt Arena, Code Arena и Team Mode.

## Последствия

Сейчас API становится строже, но безопаснее.

Будущие режимы нужно будет добавлять явно в `ALLOWED_MODE_SLUGS`.

# DEC-012 - Image Arena откладывается до стабильной Prompt Arena

## Статус

```text
Deferred
# решение зафиксировано, реализация отложена
```

## Решение

Image Arena / Visual Arena не реализуется в текущем MVP и откладывается до стабильной Prompt Arena, настроенного Supabase Storage, лимитов стоимости и safety-контролей.

## Контекст

Режим Image Arena позволяет пользователю ввести одну визуальную идею, получить изображения от нескольких image-capable моделей, сравнить их и выбрать лучший результат.

Этот режим требует больше инфраструктуры, чем Prompt Arena:

- image-capable models;
- Supabase Storage;
- metadata в PostgreSQL;
- лимиты генераций;
- контроль стоимости;
- moderation/safety rules;
- отдельный UI сравнения изображений.

## Причина

Image generation дороже и сложнее текстовых ответов. Если добавить режим слишком рано, он может сломать roadmap MVP, увеличить расходы и создать риски безопасности.

## Последствия

До этапа `v1.8` Image Arena остаётся только в документации.

Нельзя добавлять:

- страницу `/image-arena`;
- route `/api/image-arena/generate`;
- таблицы `image_generations` или `artifacts` как обязательные для текущего MVP;
- image generation calls из frontend;
- произвольный выбор image model key пользователем.

## Когда пересмотреть

После Stable Prompt Arena и после того, как будут готовы:

- Supabase Storage;
- лимиты генераций;
- allowlist model capabilities;
- безопасное backend-хранение ключей;
- понятные правила хранения image metadata.
