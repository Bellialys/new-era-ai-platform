# 16 - Decisions проекта

## Назначение файла

Этот файл фиксирует важные решения проекта **Новая эпоха**.

Файл нужен, чтобы при разработке было понятно:

- какие решения уже приняты;
- почему выбран именно такой стек;
- почему проект делается через MVP;
- почему первым режимом является Prompt Arena;
- почему сложные функции вынесены на поздние версии;
- какие решения можно пересмотреть позже;
- какие правила нельзя нарушать при разработке.

`16-decisions.md` работает как ADR - Architecture Decision Record, то есть журнал архитектурных и продуктовых решений.

## Главный источник порядка версий

Канонический порядок версий находится в файле:

```text
14-roadmap.md
# главный источник порядка разработки
```

Если этот файл конфликтует с `14-roadmap.md`, главным считается `14-roadmap.md`.

Краткий порядок версий:

```text
v0.1 - Project Documentation
# документация, идея, структура и план

v0.2 - Next.js Base
# базовый Next.js проект

v0.3 - UI MVP
# интерфейс Prompt Arena без реального AI

v0.4 - OpenRouter Integration
# реальные ответы моделей через backend API

v0.5 - Supabase Integration
# сохранение задач, ответов и моделей

v0.6 - Voting MVP
# выбор лучшего ответа

v0.7 - History MVP
# история сравнений

v0.8 - First Deploy
# публикация MVP на Vercel

v0.9 - MVP Stabilization
# лимиты, ошибки, UX и безопасность

v1.0 - Stable Prompt Arena
# первая стабильная версия

v1.1 - Code Arena Lite
# сравнение моделей по коду без запуска кода

v1.2 - Multi Model Battle
# формальные бои моделей

v1.3 - Judge Mode
# оценка ответов моделью-судьёй

v1.4 - Leaderboard
# рейтинг моделей

v1.5 - Accounts and Profiles
# авторизация, профили, личная история

v1.6 - Admin Panel and Limits
# управление моделями, режимами, лимитами и стоимостью

v1.7 - Code Arena Runner
# sandbox, тесты и безопасный запуск кода

v2.0 - AI Team Mode
# командная работа AI-моделей с ролями
```

## Формат решений

Каждое решение описывается по единому принципу:

```text
Решение
# что выбрали

Статус
# Accepted, Proposed, Deferred, Rejected или Needs Review

Контекст
# почему возник вопрос

Причина
# почему выбран именно этот вариант

Последствия
# что это даёт и какие ограничения создаёт

Когда пересмотреть
# при каких условиях решение можно изменить
```

## Статусы

```text
Accepted
# решение принято и используется

Proposed
# решение предложено, но ещё не внедрено

Deferred
# решение отложено на будущее

Rejected
# решение отклонено

Needs Review
# решение нужно пересмотреть позже
```


Документацию проекта вести в формате Markdown `.md`.

Проект связан с кодом, GitHub, Next.js, Supabase и OpenRouter. Для такой документации удобнее использовать обычные текстовые файлы, а не Word.

# DEC-002 - Делать проект поэтапно через MVP

## Статус

```text
Accepted
# решение принято
```

## Решение

Проект разрабатывать маленькими рабочими этапами.

Сначала делается простая рабочая основа, потом стабильный Prompt Arena, потом остальные режимы.

## Контекст

Проект большой. В нём планируются:

- Prompt Arena;
- Code Arena;
- Multi Model Battle;
- Judge Mode;
- Leaderboard;
- Accounts and Profiles;
- Admin Panel and Limits;
- Code Arena Runner;
- AI Team Mode.

Если начать всё сразу, проект станет хаотичным и может не дойти до рабочей версии.

## Причина

MVP позволяет быстро получить работающий результат:

```text
Пользователь вводит задачу.
# основной сценарий

Выбирает 2-3 модели.
# сравнение моделей

Получает ответы рядом.
# главная ценность проекта

Выбирает лучший ответ.
# появляется основа для статистики
```

## Последствия

Сложные функции не добавляются в первый MVP.

Сначала делаем:

```text
Prompt Arena
# основа продукта

OpenRouter
# реальные AI-ответы

Supabase
# сохранение задач и ответов

Voting
# выбор лучшего ответа

History
# просмотр прошлых сравнений
```

Потом добавляем остальные режимы.

## Когда пересмотреть

Не пересматривать до версии `v1.0 Stable Prompt Arena`.

# DEC-004 - Использовать Next.js

## Статус

```text
Accepted
# решение принято
```

## Решение

Использовать **Next.js** как основной frontend и backend layer проекта.

## Контекст

Проекту нужны:

- страницы сайта;
- интерактивный UI;
- server-side API routes;
- безопасная работа с OpenRouter API key;
- подключение к Supabase;
- деплой на Vercel;
- TypeScript.

## Причина

Next.js подходит для MVP потому что:

- frontend и backend routes находятся в одном проекте;
- хорошо работает с Vercel;
- поддерживает TypeScript;
- удобно развивать проект постепенно;
- можно не создавать отдельный backend на старте.

## Последствия

Базовая структура проекта:

```text
src/app/
# страницы и маршруты Next.js

src/app/api/
# server-side API routes

src/components/
# UI компоненты

src/lib/
# серверная логика, клиенты API и helpers

src/types/
# TypeScript-типы
```

## Когда пересмотреть

Если проект вырастет и API routes перестанут справляться, можно вынести backend отдельно.

Возможные варианты:

```text
NestJS
# отдельный backend на Node.js

FastAPI
# отдельный backend на Python

Queue worker
# отдельная обработка тяжёлых задач
```

# DEC-007 - Использовать OpenRouter

## Статус

```text
Accepted
# решение принято
```

## Решение

Использовать **OpenRouter API** для доступа к нескольким AI-моделям через один API.

## Контекст

Главная идея проекта - сравнение нескольких моделей. Подключать каждую модель через отдельный API на старте слишком сложно.

## Причина

OpenRouter выбран потому что:

- даёт доступ к разным AI-моделям через один формат API;
- упрощает сравнение моделей;
- ускоряет MVP;
- позволяет менять модели без полной переделки backend;
- подходит для Prompt Arena, Code Arena, Judge Mode и AI Team Mode.

## Последствия

Запросы к OpenRouter выполняются только на server-side.

```text
Frontend
# отправляет prompt в наш API

Next.js API route
# проверяет входные данные и выбранные модели

OpenRouter API
# получает ответы моделей

Supabase
# сохраняет задачу и ответы
```

## Когда пересмотреть

Можно пересмотреть, если:

- OpenRouter станет слишком дорогим;
- потребуются прямые контракты с провайдерами;
- появится своя локальная инфраструктура моделей;
- понадобится отдельная оптимизация под конкретного провайдера.

```text
.env.local
# локальные секреты разработчика
```

```text

# должен исключать .env.local
```

Не пересматривать. Это базовое правило безопасности.

# DEC-009 - Использовать allowlist моделей

## Статус

```text
Accepted
# решение принято
```

## Решение

Использовать server-side allowlist моделей.

Пользователь не должен иметь возможность отправить любой произвольный `model_id` напрямую в OpenRouter.

## Контекст

Если разрешить любой model ID, пользователь сможет выбрать слишком дорогую или неподходящую модель.

## Причина

Allowlist нужен для:

- контроля расходов;
- безопасности;
- стабильного UI;
- отключения проблемных моделей;
- ограничения MVP максимум 2-3 моделями;
- будущего управления через Admin Panel.

## Последствия

На раннем этапе модели можно хранить в server config.

Позже источник моделей переносится в Supabase:

```text
models
# таблица доступных моделей

is_active
# модель включена

is_public
# модель видна обычным пользователям

supports_code
# модель подходит для Code Arena

supports_judge
# модель подходит для Judge Mode

price_label
# free, cheap, balanced или expensive
```

## Когда пересмотреть

Allowlist не отменять. Можно менять только место хранения allowlist.

# DEC-010 - Ограничить MVP максимум 3 моделями

## Статус

```text
Accepted
# решение принято
```

## Решение

В MVP разрешить выбирать максимум 3 модели за один запрос.

## Контекст

Каждая выбранная модель создаёт отдельный AI-запрос. Чем больше моделей, тем выше стоимость и риск ошибок.

## Причина

3 модели достаточно для сравнения:

```text
2 модели
# минимальное сравнение

3 модели
# нормальный максимум для MVP

4+ модели
# дороже, медленнее и сложнее для старта
```

## Последствия

Backend должен проверять количество моделей.

Frontend должен показывать понятное ограничение.

## Когда пересмотреть

Можно пересмотреть после появления:

- аккаунтов;
- лимитов;
- статистики расходов;
- админ-панели.

# DEC-011 - Ограничить prompt до 4000 символов в MVP

## Статус

```text
Accepted
# решение принято
```

## Решение

В MVP ограничить длину prompt до 4000 символов.

## Контекст

Очень длинные запросы увеличивают стоимость, время ответа и риск ошибок.

## Причина

Лимит 4000 символов:

- достаточен для большинства обычных задач;
- защищает бюджет MVP;
- упрощает UX;
- уменьшает риск timeout;
- позволяет позже расширить лимиты по тарифам или ролям.

## Последствия

Проверка должна быть и на frontend, и на backend.

```text
Frontend validation
# пользователь сразу видит ограничение

Backend validation
# настоящая защита
```

## Когда пересмотреть

После появления аккаунтов, тарифов или режима Code Arena.

# DEC-013 - Не делать AI Team Mode в MVP

## Статус

```text
Accepted
# решение принято
```

## Решение

AI Team Mode не входит в MVP и переносится на `v2.0`.

## Контекст

AI Team Mode - сложный режим, где несколько моделей работают как команда с ролями, шагами и раундами.

## Причина

AI Team Mode нельзя делать первым, потому что он требует:

- несколько AI-запросов на одну задачу;
- роли моделей;
- сохранение промежуточных шагов;
- сложный UI;
- больше расходов;
- логику раундов;
- контроль качества итогового ответа.

## Последствия

AI Team Mode делается только после:

```text
Stable Prompt Arena
# основа проекта работает

Code Arena Lite
# есть кодовый режим без Runner

Multi Model Battle
# есть формальное сравнение моделей

Judge Mode
# есть автоматическая оценка

Leaderboard
# есть рейтинг

Accounts and Profiles
# есть пользователи

Admin Panel and Limits
# есть контроль расходов и доступа
```

## Когда пересмотреть

Не пересматривать до `v2.0`.

# DEC-016 - Сначала server config моделей, потом Supabase models

## Статус

```text
Accepted
# решение принято
```

## Решение

На раннем MVP можно хранить список разрешённых моделей в server config. Позже перенести управление моделями в таблицу `models` в Supabase.

## Контекст

Для первого подключения OpenRouter не нужна сложная админка. Но в будущем модели нужно будет включать, отключать, сортировать и распределять по режимам.

## Причина

Пошаговый подход проще:

```text
server config
# быстрый старт для v0.4

Supabase models
# нормальная система для v0.5+

Admin Panel
# управление моделями в v1.6
```

## Последствия

В коде нужно сразу отделить:

```text
model_key
# внутренний ключ модели в проекте

provider_model_id
# точный ID модели у OpenRouter

display_name
# название для UI
```

## Когда пересмотреть

На этапе `v0.5 Supabase Integration` и затем на `v1.6 Admin Panel and Limits`.

# DEC-017 - Использовать TypeScript

## Статус

```text
Accepted
# решение принято
```

## Решение

Использовать TypeScript во всём проекте.

## Контекст

Проект будет работать с API, базой данных, режимами, ответами моделей и разными типами данных. Без типов легко допустить ошибку.

## Причина

TypeScript нужен для:

- типов API requests;
- типов API responses;
- типов моделей;
- типов задач;
- типов голосов;
- типов режимов;
- безопасного рефакторинга.

## Последствия

Основные типы хранить отдельно:

```text
src/types/
# общие TypeScript-типы

src/lib/
# серверная логика с типами
```

## Когда пересмотреть

Не пересматривать.

# DEC-021 - Code Arena сначала Lite, потом Runner

## Статус

```text
Accepted
# решение принято
```

## Решение

Code Arena разделить на два этапа:

```text
v1.1 - Code Arena Lite
# модели пишут код, пользователь сравнивает ответы вручную

v1.7 - Code Arena Runner
# sandbox, тесты и объективные результаты запуска кода
```

## Контекст

Code Arena Lite можно сделать сравнительно рано, потому что она не запускает чужой код.

Code Arena Runner опаснее и сложнее, потому что он должен безопасно запускать код.

## Причина

Runner требует:

- sandbox;
- timeout;
- memory limit;
- no network mode;
- изоляцию файловой системы;
- очистку после запуска;
- защиту секретов;
- логи запусков;
- лимиты пользователей;
- админ-контроль.

Поэтому Runner нельзя ставить раньше аккаунтов, лимитов и админ-панели.

## Последствия

Правильный порядок:

```text
v1.1 - Code Arena Lite
# ручное сравнение кода без запуска

v1.5 - Accounts and Profiles
# пользователи и личные лимиты

v1.6 - Admin Panel and Limits
# управление доступом и стоимостью

v1.7 - Code Arena Runner
# безопасный запуск кода
```

## Когда пересмотреть

Runner можно приблизить только если безопасный sandbox и лимиты будут готовы раньше. Но Runner не должен блокировать выпуск Code Arena Lite.

# DEC-022 - Код моделей запускать только в sandbox

## Статус

```text
Accepted
# решение принято
```

## Решение

Любой код, сгенерированный AI-моделью или пользователем, можно запускать только в изолированном sandbox.

## Контекст

Code Arena Runner предполагает выполнение кода. Такой код может быть вредным, бесконечным или случайно опасным.

## Причина

Нельзя запускать чужой код:

```text
на сервере приложения;
# может повредить проект

рядом с .env;
# может украсть секреты

без timeout;
# может зависнуть

без memory limit;
# может перегрузить систему

с доступом к network;
# может делать внешние запросы
```

## Последствия

Для Runner нужен отдельный безопасный подход:

```text
E2B
# sandbox для запуска кода

Judge0
# code execution API

Docker sandbox
# свой вариант, сложнее для поддержки

WebAssembly sandbox
# ограниченный вариант для некоторых языков
```

До выбора sandbox Runner не включается.

## Когда пересмотреть

Не пересматривать правило sandbox. Можно менять только реализацию sandbox.

# DEC-023 - AI Team Mode использует 8 канонических ролей

## Статус

```text
Accepted
# решение принято
```

## Решение

AI Team Mode использует 8 канонических ролей:

```text
Planner
# планирует задачу

Architect
# проектирует структуру решения

Implementer
# предлагает реализацию или код

Tester
# проверяет ошибки и тестовые сценарии

Security Reviewer
# проверяет риски безопасности, секреты, опасные команды и API

Critic
# ищет слабые места решения

Synthesizer
# собирает финальный ответ

Judge
# оценивает итоговый результат
```

## Контекст

Team Mode должен быть не просто параллельной отправкой запроса разным моделям, а управляемой командной работой.

## Причина

8 ролей покрывают полный цикл работы:

- планирование;
- архитектура;
- реализация;
- тестирование;
- безопасность;
- критика;
- сборка ответа;
- финальная оценка.

## Последствия

В `18-team-mode-spec.md` нужно использовать этот набор как основной.

Некоторые роли можно отключать через настройки режима, но канонический список остаётся единым.

## Когда пересмотреть

Только если логика Team Mode радикально изменится.

# DEC-024 - В Team Mode использовать flow_type вместо schema

## Статус

```text
Accepted
# решение принято
```

## Решение

В таблицах и API AI Team Mode использовать поле `flow_type`, а не `schema`.

## Контекст

Слово `schema` может путаться с PostgreSQL schema, JSON schema или общей схемой базы данных.

## Причина

`flow_type` точнее описывает назначение поля:

```text
sequential
# роли выполняются по очереди

parallel
# роли работают параллельно

debate
# модели спорят и улучшают ответ

review
# отдельный режим проверки
```

## Последствия

В базе и API использовать:

```text
flow_type
# тип командного процесса
```

## Когда пересмотреть

Не пересматривать, если структура Team Mode не меняется радикально.

# DEC-026 - Точные OpenRouter model ID проверять перед production

## Статус

```text
Accepted
# решение принято
```

## Решение

Точные `provider_model_id` для OpenRouter проверять перед добавлением в production-конфиг или базу данных.

## Контекст

Список моделей у OpenRouter может меняться. Некоторые модели могут получать новые версии, preview-суффиксы или временно исчезать.

## Причина

Нельзя строить production-код только на старом списке моделей из документации.

Перед использованием нужно проверить модели через OpenRouter Models API или актуальную страницу модели.

## Последствия

В `11-ai-models.md` можно хранить стратегию выбора моделей, но перед кодом нужно подтвердить точные ID.

Для MVP лучше начать с небольшого набора проверенных моделей.

## Когда пересмотреть

Не пересматривать. Проверка model ID нужна всегда перед изменением production-набора моделей.

# Как обновлять файл decisions

## Когда добавлять новое решение

Новое решение нужно добавить, если оно влияет на:

- архитектуру;
- безопасность;
- базу данных;
- API;
- UI;
- деплой;
- AI-модели;
- стоимость;
- роли пользователей;
- будущую масштабируемость.

## Когда не нужно добавлять решение

Не нужно добавлять мелкие изменения:

- исправление опечатки;
- переименование локальной переменной;
- небольшой UI-отступ;
- временный mock;
- локальная настройка редактора.

## Правило

Если по решению может возникнуть вопрос “почему мы сделали именно так?”, его нужно записать в `16-decisions.md`.
