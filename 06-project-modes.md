# 06 - Режимы проекта

## Назначение файла

Этот файл описывает режимы работы проекта **Новая эпоха**.

Режимы - это продуктовая основа платформы. Они отличают проект от обычного AI-чата, потому что пользователь получает не один ответ от одной модели, а несколько вариантов от разных моделей, может сравнить их, выбрать лучший результат, сохранить историю и постепенно формировать рейтинг качества моделей.

Этот файл должен быть согласован с:

- `00-readme.md` - общий вход в проект;
- `04-mvp-scope.md` - границы MVP;
- `07-architecture.md` - техническая архитектура;
- `08-database.md` - структура базы данных;
- `09-api-structure.md` - API проекта;
- `10-ui-pages.md` - страницы интерфейса;
- `11-ai-models.md` - стратегия выбора AI-моделей;
- `12-security-and-env.md` - безопасность и переменные окружения;
- `14-roadmap.md` - главный порядок версий;
- `16-decisions.md` - зафиксированные архитектурные решения;
- `17-code-arena-spec.md` - подробная спецификация Code Arena;
- `18-team-mode-spec.md` - подробная спецификация AI Team Mode.

## Главный источник порядка версий

Главным источником порядка разработки является файл:

```text
14-roadmap.md
# главный порядок версий проекта
```

Этот файл описывает смысл режимов и их место в продукте, но не должен спорить с `14-roadmap.md`.

Если порядок версий в этом файле и в `14-roadmap.md` отличается, правильным считается порядок из `14-roadmap.md`.

## Основная идея режимов

Пользователь выбирает режим, вводит задачу, выбирает модели или использует рекомендуемый набор, после чего backend отправляет запросы нескольким AI-моделям через OpenRouter API. Ответы отображаются рядом, сравниваются, оцениваются пользователем или моделью-судьёй и сохраняются в Supabase.

Базовая логика:

```text
Пользователь вводит задачу.
# исходный prompt или coding task

Пользователь выбирает режим.
# Prompt Arena, Code Arena, Battle, Judge, Leaderboard или Team Mode

Пользователь выбирает модели.
# вручную или через рекомендованный preset

Frontend отправляет запрос в Next.js API Route.
# секретные ключи не попадают в браузер

Backend вызывает OpenRouter API.
# все AI-запросы проходят только через сервер

Система получает ответы моделей.
# успешные ответы и ошибки обрабатываются отдельно

Результаты сохраняются в Supabase.
# данные нужны для истории, голосования и рейтинга

Пользователь или Judge выбирает лучший результат.
# победитель используется для статистики и Leaderboard
```

## Продуктовые режимы

В продуктовой идее проекта есть шесть основных режимов:

| Номер | Режим | Назначение |
|---:|---|---|
| 1 | Prompt Arena | сравнение обычных ответов моделей на один prompt |
| 2 | Code Arena | сравнение ответов моделей на задачи программирования |
| 3 | Multi Model Battle | формальное соревнование моделей с победителем |
| 4 | AI Team Mode | совместная работа моделей с разными ролями |
| 5 | Judge Mode | оценка ответов отдельной моделью-судьёй |
| 6 | Leaderboard | рейтинг моделей по результатам сравнений |

Важно: продуктовая нумерация показывает идею платформы, но не является порядком разработки.

## Порядок разработки режимов

Технически режимы нужно делать не в продуктовой нумерации, а по сложности и безопасности.

Правильный порядок:

| Версия | Что делаем | Почему именно здесь |
|---|---|---|
| `v0.3` | UI MVP для Prompt Arena | нужен первый видимый интерфейс |
| `v0.4` | OpenRouter Integration | нужны реальные ответы моделей |
| `v0.5` | Supabase Persistence | нужно сохранять задачи и ответы |
| `v0.6` | Voting | нужен выбор лучшего ответа |
| `v0.7` | History | нужна история сравнений |
| `v1.0` | Stable Prompt Arena | первый стабильный рабочий MVP |
| `v1.1` | Code Arena Lite | режим кода без запуска чужого кода |
| `v1.2` | Multi Model Battle | формальное соревнование моделей |
| `v1.3` | Judge Mode | автоматическая оценка ответов |
| `v1.4` | Leaderboard | рейтинг на основе накопленных данных |
| `v1.5` | Accounts and Profiles | нужны пользователи и личная история |
| `v1.6` | Admin Panel and Limits | нужны лимиты, контроль моделей и расходов |
| `v1.7` | Code Arena Runner | запуск кода только после защиты и лимитов |
| `v2.0` | AI Team Mode | самый сложный режим после стабильной базы |

## Почему AI Team Mode не делается сразу

AI Team Mode является важным режимом продукта, но его нельзя делать в MVP.

Причины:

- он требует много запросов к AI-моделям;
- он дороже обычного сравнения;
- он требует таблиц `team_runs` и `team_steps`;
- он требует ролей моделей;
- он требует логирования каждого шага;
- он требует контроля стоимости;
- его сложнее отлаживать;
- он должен опираться на уже работающие Prompt Arena, Judge Mode, историю, лимиты и модели.

Поэтому AI Team Mode остаётся на версии:

```text
v2.0 - AI Team Mode
# полноценный командный режим после стабильной платформы
```

## Почему Code Arena Runner не делается сразу

Code Arena делится на две части:

```text
Code Arena Lite
# сравнение решений по коду без запуска чужого кода

Code Arena Runner
# запуск кода в sandbox и проверка тестами
```

`Code Arena Lite` можно делать на `v1.1`, потому что он не запускает код пользователя и не требует sandbox.

`Code Arena Runner` можно делать только на `v1.7`, потому что он связан с рисками:

- выполнение чужого кода;
- бесконечные циклы;
- попытки доступа к файловой системе;
- сетевые запросы из чужого кода;
- большая нагрузка на сервер;
- потенциальные расходы;
- необходимость timeout;
- необходимость sandbox;
- необходимость лимитов;
- необходимость логирования запусков.

Правило проекта:

```text
До v1.7 проект не запускает чужой код.
# Code Arena Lite может сравнивать код текстом, но не выполняет его
```

# Общие правила для всех режимов

## Backend-only AI calls

Все запросы к AI-моделям выполняются только через backend.

Запрещено:

```text
Frontend -> OpenRouter
# нельзя, потому что API-ключ попадёт в браузер
```

Разрешено:

```text
Frontend -> Next.js API Route -> OpenRouter
# правильно, потому что API-ключ хранится на сервере
```

## Единый формат задачи

Для всех режимов базовая задача должна иметь общую структуру:

```ts
type TaskMode =
  | "prompt_arena"
  | "code_arena_lite"
  | "multi_model_battle"
  | "judge_mode"
  | "leaderboard"
  | "code_arena_runner"
  | "ai_team_mode";
```

Общие данные задачи:

```ts
type BaseTask = {
  id: string;
  mode: TaskMode;
  prompt_text: string;
  selected_model_ids: string[];
  created_at: string;
};
```

## Единый формат ответа модели

Каждый ответ модели должен храниться отдельно.

```ts
type ModelResponse = {
  id: string;
  task_id: string;
  model_id: string;
  content: string | null;
  error_message: string | null;
  latency_ms: number | null;
  input_tokens: number | null;
  output_tokens: number | null;
  total_cost: number | null;
  created_at: string;
};
```

## Частичный успех

Ошибка одной модели не должна ломать весь результат.

Пример:

```text
Model A - ответила успешно
# показываем ответ

Model B - timeout
# показываем ошибку только в её карточке

Model C - ответила успешно
# показываем ответ
```

Правило:

```text
Если хотя бы одна модель ответила успешно, пользователь должен увидеть результат.
# не превращаем всю задачу в failed из-за одной модели
```

## Ограничение количества моделей

Для MVP:

```text
minimum selected models: 2
# меньше двух моделей нет смысла сравнивать

maximum selected models: 3
# ограничение расходов и нагрузки на MVP
```

После MVP лимит можно расширить через Admin Panel and Limits.

## Сохранение данных

На MVP сохраняются:

- задача пользователя;
- выбранный режим;
- выбранные модели;
- ответы моделей;
- ошибки моделей;
- голос пользователя;
- дата создания;
- базовая техническая информация.

Для будущих режимов сохраняются дополнительные данные:

- оценки судьи;
- категории задач;
- история пользователя;
- статистика рейтинга;
- запуск кода;
- шаги AI Team Mode.

# Режим 1 - Prompt Arena

## Версии

Prompt Arena реализуется поэтапно:

```text
v0.3 - UI MVP
# интерфейс без реальных AI-запросов или с mock-ответами

v0.4 - OpenRouter Integration
# реальные ответы моделей

v0.5 - Supabase Persistence
# сохранение задач и ответов

v0.6 - Voting
# выбор лучшего ответа

v0.7 - History
# история сравнений

v1.0 - Stable Prompt Arena
# первый стабильный MVP
```

## Назначение

Prompt Arena нужен, чтобы сравнить обычные ответы нескольких AI-моделей на один запрос пользователя.

Это главный режим MVP.

## Для каких задач подходит

- объяснение сложных тем;
- генерация идей;
- учебные вопросы;
- анализ текста;
- написание обычных ответов;
- сравнение формулировок;
- проверка качества ответов разных моделей;
- выбор лучшего ответа для дальнейшей работы.

## Что входит в MVP

В Stable Prompt Arena должны быть:

- страница `/arena`;
- поле ввода задачи;
- выбор 2-3 моделей;
- отправка запроса через backend;
- вызов OpenRouter API;
- параллельное получение ответов;
- карточки ответов;
- обработка ошибок отдельных моделей;
- кнопка выбора лучшего ответа;
- сохранение задачи в Supabase;
- сохранение ответов в Supabase;
- сохранение голоса в Supabase;
- страница истории `/history`;
- страница просмотра одного сравнения `/history/[taskId]`.

## Что не входит в MVP

В MVP не входят:

- аккаунты пользователей;
- платные тарифы;
- публичные профили;
- сложная аналитика;
- Team Mode;
- запуск кода;
- автоматический Judge;
- публичный Leaderboard;
- админ-панель;
- payments;
- browser automation;
- file uploads.

## Минимальный пользовательский сценарий

1. Пользователь открывает `/arena`.
2. Вводит задачу.
3. Выбирает минимум 2 модели.
4. Нажимает кнопку сравнения.
5. Frontend отправляет запрос в `/api/compare`.
6. Backend вызывает OpenRouter.
7. Система получает ответы моделей.
8. Ответы отображаются в карточках рядом.
9. Пользователь выбирает лучший ответ.
10. Frontend отправляет голос в `/api/vote`.
11. Supabase сохраняет результат.
12. Пользователь может открыть историю.

## API для режима

```text
GET /api/models
# получить список доступных моделей

POST /api/compare
# создать сравнение и получить ответы моделей

POST /api/vote
# сохранить выбор лучшего ответа

GET /api/history
# получить историю сравнений

GET /api/history/[taskId]
# получить одно сохранённое сравнение
```

## Таблицы Supabase

Для Prompt Arena нужны:

```text
models
# список доступных моделей

tasks
# задачи пользователя

model_responses
# ответы моделей

votes
# голоса за лучший ответ
```

## Проверки качества

- нельзя отправить пустой prompt;
- нельзя выбрать меньше 2 моделей;
- нельзя выбрать больше 3 моделей в MVP;
- ошибка одной модели не ломает остальные ответы;
- все ответы привязаны к одной задаче;
- выбранный победитель должен существовать среди ответов задачи;
- история показывает сохранённые сравнения;
- секретные ключи не видны в браузере.

## Критерии готовности

Prompt Arena считается готовой для `v1.0`, если:

- пользователь может создать сравнение;
- минимум две модели отвечают через OpenRouter;
- ответы сохраняются в Supabase;
- пользователь может выбрать победителя;
- голос сохраняется;
- историю можно открыть после обновления страницы;
- production deploy работает на Vercel;
- `.env.local` не попадает в Git;
- ошибки моделей отображаются понятно.

# Режим 2 - Code Arena Lite

## Версия

```text
v1.1 - Code Arena Lite
# сравнение решений по программированию без запуска кода
```

## Назначение

Code Arena Lite нужен, чтобы сравнить, как разные AI-модели решают задачи программирования.

В этой версии система не запускает код. Она только отправляет coding prompt моделям и показывает ответы рядом.

## Почему это отдельный режим

Кодовые задачи отличаются от обычных prompt-задач:

- нужен язык программирования;
- нужен формат ответа;
- часто нужно объяснение решения;
- иногда нужен анализ сложности;
- пользователь может сравнивать стиль кода;
- нужны отдельные критерии оценки.

## Что входит в Code Arena Lite

- страница `/code`;
- выбор языка программирования;
- поле для задачи;
- выбор моделей, подходящих для кода;
- отправка через backend;
- карточки решений;
- отображение кода;
- отображение объяснения;
- голос за лучший вариант;
- сохранение результата как task с режимом `code_arena_lite`.

## Что запрещено в Code Arena Lite

До `v1.7` запрещено:

- запускать код пользователя;
- запускать код модели;
- выполнять тесты;
- создавать sandbox;
- давать доступ к файловой системе;
- давать сетевой доступ выполняемому коду;
- создавать таблицу `code_results` как обязательную часть MVP;
- показывать статус выполнения кода.

## Минимальный пользовательский сценарий

1. Пользователь открывает `/code`.
2. Выбирает язык программирования.
3. Вводит задачу.
4. Выбирает 2-3 code-модели.
5. Нажимает кнопку сравнения.
6. Backend отправляет задачу моделям.
7. Ответы отображаются рядом.
8. Пользователь выбирает лучшее решение.
9. Результат сохраняется в Supabase.

## Дополнительные поля задачи

Для Code Arena Lite можно добавить в `tasks.metadata`:

```json
{
  "language": "typescript",
  "difficulty": "easy",
  "requires_explanation": true,
  "requires_complexity_analysis": true
}
```

## Критерии оценки ответа

Пользователь или будущий Judge может смотреть на:

- корректность алгоритма;
- читаемость кода;
- простоту решения;
- обработку крайних случаев;
- объяснение;
- сложность по времени;
- сложность по памяти;
- соответствие выбранному языку;
- отсутствие лишних зависимостей.

## Связь с Code Arena Runner

Code Arena Lite является подготовительным этапом для Runner.

```text
v1.1 - Code Arena Lite
# модели пишут код, но код не запускается

v1.7 - Code Arena Runner
# код запускается в sandbox и проверяется тестами
```

Подробная спецификация находится в:

```text
17-code-arena-spec.md
# подробные правила Code Arena Lite и Runner
```

# Режим 3 - Multi Model Battle

## Версия

```text
v1.2 - Multi Model Battle
# формальное соревнование моделей
```

## Назначение

Multi Model Battle нужен, чтобы превратить обычное сравнение ответов в формальное соревнование.

В Prompt Arena пользователь просто сравнивает ответы. В Battle система дополнительно фиксирует участников, правила, победителя и результат боя.

## Отличие от Prompt Arena

Prompt Arena:

```text
Пользователь сравнивает ответы.
# мягкий режим сравнения
```

Multi Model Battle:

```text
Модели соревнуются по заданным правилам.
# результат используется для статистики и рейтинга
```

## Что входит в первую версию

- страница `/battle`;
- выбор категории боя;
- выбор 2-4 моделей;
- одинаковая задача для всех моделей;
- карточки ответов;
- выбор победителя пользователем;
- запись результата как battle;
- использование результата для будущего Leaderboard.

## Возможные категории боя

- general reasoning;
- writing;
- summarization;
- coding explanation;
- study help;
- business answer;
- technical answer;
- creative idea;
- Ukrainian language answer;
- Russian language answer;
- English answer.

## Минимальный пользовательский сценарий

1. Пользователь открывает `/battle`.
2. Выбирает категорию.
3. Вводит задачу.
4. Выбирает модели.
5. Запускает battle.
6. Система получает ответы.
7. Пользователь выбирает победителя.
8. Результат сохраняется.
9. В будущем результат учитывается в Leaderboard.

## Данные результата

Для первого этапа можно использовать `tasks.metadata`:

```json
{
  "battle_category": "general_reasoning",
  "battle_type": "manual_vote",
  "winner_response_id": "response-id"
}
```

Позже можно добавить отдельные таблицы, если battle-логика станет сложной.

## Критерии готовности

Multi Model Battle готов, если:

- можно выбрать категорию;
- можно выбрать участников;
- все модели получают одинаковую задачу;
- ответы отображаются рядом;
- пользователь может выбрать победителя;
- победитель сохраняется;
- результат можно использовать для рейтинга.

# Режим 4 - Judge Mode

## Версия

```text
v1.3 - Judge Mode
# оценка ответов отдельной моделью-судьёй
```

## Назначение

Judge Mode нужен, чтобы отдельная AI-модель оценивала ответы других моделей по заданным критериям.

Это полезно, когда пользователь не хочет вручную долго сравнивать ответы или хочет получить дополнительную оценку качества.

## Главная идея

```text
Модели A, B, C дают ответы.
# участники сравнения

Модель Judge получает исходный prompt и ответы A, B, C.
# судья анализирует ответы

Judge возвращает оценки и объяснение.
# система показывает победителя и причину
```

## Что входит в первую версию

- выбор judge-модели;
- автоматическая оценка ответов после сравнения;
- оценка по нескольким критериям;
- общий победитель;
- объяснение выбора;
- сохранение judge result в metadata;
- отображение результата в UI.

## Пример критериев

- correctness;
- completeness;
- clarity;
- usefulness;
- safety;
- formatting;
- reasoning quality;
- instruction following.

## Пример структуры оценки

```json
{
  "winner_response_id": "response-id-1",
  "scores": [
    {
      "response_id": "response-id-1",
      "correctness": 9,
      "completeness": 8,
      "clarity": 9,
      "usefulness": 9,
      "total": 35
    },
    {
      "response_id": "response-id-2",
      "correctness": 7,
      "completeness": 8,
      "clarity": 8,
      "usefulness": 7,
      "total": 30
    }
  ],
  "reason": "Первый ответ точнее, полнее и лучше следует задаче пользователя."
}
```

## Важное ограничение

Judge Mode не должен полностью заменять выбор пользователя.

Правильная логика:

```text
User vote
# мнение пользователя

Judge result
# автоматическая оценка
```

Оба результата можно хранить отдельно.

## Риски

- судья может ошибаться;
- дорогая judge-модель увеличивает стоимость;
- слишком длинные ответы могут превышать context length;
- judge prompt должен быть стабильным;
- нельзя давать судье знать имена моделей, если нужен blind judging.

## Критерии готовности

Judge Mode готов, если:

- можно выбрать judge-модель;
- judge получает исходный prompt и ответы;
- judge возвращает структурированный результат;
- результат сохраняется;
- UI показывает победителя и объяснение;
- ошибка judge не ломает ответы моделей.

# Режим 5 - Leaderboard

## Версия

```text
v1.4 - Leaderboard
# рейтинг моделей на основе накопленных результатов
```

## Назначение

Leaderboard нужен, чтобы показывать рейтинг моделей по результатам сравнений, голосов, battle-результатов и judge-оценок.

Главная ценность Leaderboard - помочь пользователю понять, какие модели лучше подходят для разных типов задач.

## Что показывает Leaderboard

- модель;
- количество участий;
- количество побед;
- win rate;
- средняя оценка Judge;
- категория задач;
- средняя скорость ответа;
- примерная стоимость;
- статус модели;
- дата последнего участия.

## Первую версию нужно делать простой

В `v1.4` не нужно строить сложную рейтинговую систему.

Достаточно:

- считать победы по голосам;
- считать количество участий;
- считать простой win rate;
- фильтровать по категории;
- показывать таблицу.

Простая формула:

```text
win_rate = wins / total_matches
# базовый процент побед
```

## Почему Leaderboard нельзя делать раньше

Leaderboard зависит от накопленных данных.

До него должны быть:

- сохранение задач;
- сохранение ответов;
- голоса пользователей;
- история;
- battle-результаты или judge-результаты.

Без этих данных рейтинг будет пустым или фальшивым.

## Источники данных

Для первой версии можно считать рейтинг из:

```text
model_responses
# какие модели участвовали

votes
# какие ответы победили

tasks
# категории и режимы задач
```

Позже можно добавить отдельную таблицу:

```text
leaderboard_snapshots
# сохранённые агрегированные рейтинги
```

## Минимальный пользовательский сценарий

1. Пользователь открывает `/leaderboard`.
2. Видит таблицу моделей.
3. Фильтрует рейтинг по категории.
4. Смотрит win rate, количество побед и участий.
5. Понимает, какие модели лучше подходят под разные задачи.

## Критерии готовности

Leaderboard готов, если:

- таблица показывает модели;
- есть wins;
- есть total matches;
- есть win rate;
- данные считаются из сохранённых результатов;
- можно фильтровать хотя бы по режиму или категории;
- рейтинг не показывает модели без достаточного количества данных как абсолютных лидеров.

# Режим 6 - Code Arena Runner

## Версия

```text
v1.7 - Code Arena Runner
# запуск кода в sandbox и проверка тестами
```

## Назначение

Code Arena Runner нужен, чтобы не только сравнивать текстовые решения моделей, но и запускать код в безопасной изолированной среде, проверять тестами и сохранять результаты выполнения.

## Почему Runner отдельно от Lite

Lite оценивает текст и код визуально.

Runner проверяет код фактически:

```text
Code Arena Lite
# модель написала решение

Code Arena Runner
# система запустила решение и проверила тестами
```

## Что входит в Runner

- sandbox для запуска кода;
- timeout;
- memory limit;
- запрет сетевого доступа;
- ограничение файловой системы;
- тест-кейсы;
- статус выполнения;
- stdout;
- stderr;
- результат тестов;
- таблица `code_results`;
- логирование запусков;
- защита от злоупотреблений.

## Что должно быть готово до Runner

До `v1.7` должны быть готовы:

- Prompt Arena;
- Code Arena Lite;
- история;
- аккаунты;
- лимиты;
- админ-панель;
- usage logs;
- security review;
- отдельное решение по sandbox.

## Минимальный пользовательский сценарий

1. Пользователь открывает `/code`.
2. Вводит coding task.
3. Выбирает модели.
4. Модели генерируют решения.
5. Пользователь запускает проверку.
6. Runner выполняет код в sandbox.
7. Система показывает passed/failed tests.
8. Результаты сохраняются в `code_results`.
9. Пользователь выбирает лучшее решение.

## Критерии готовности

Runner готов, если:

- код запускается только в sandbox;
- есть timeout;
- есть memory limit;
- нет сетевого доступа из запускаемого кода;
- ошибки выполнения отображаются безопасно;
- результаты тестов сохраняются;
- пользовательские лимиты работают;
- админ может выключить Runner;
- Runner не влияет на работу Prompt Arena.

# Режим 7 - AI Team Mode

## Версия

```text
v2.0 - AI Team Mode
# командная работа нескольких AI-моделей с ролями
```

## Назначение

AI Team Mode нужен, чтобы несколько моделей работали не как соперники, а как команда.

Каждая модель получает роль и выполняет отдельную часть работы.

Пример ролей:

- Planner;
- Researcher;
- Developer;
- Critic;
- Security Reviewer;
- Tester;
- Writer;
- Final Answer Builder.

## Главная идея

```text
Одна модель планирует.
# Planner

Другая модель предлагает решение.
# Developer или Writer

Третья модель критикует.
# Critic

Четвёртая проверяет безопасность.
# Security Reviewer

Финальная модель собирает итог.
# Final Answer Builder
```

## Почему это v2.0

AI Team Mode требует:

- нескольких последовательных AI-запросов;
- ролей моделей;
- состояния выполнения;
- истории шагов;
- контроля стоимости;
- лимитов;
- хорошего UI;
- стабильной базы;
- логирования каждого шага;
- возможности остановить выполнение;
- возможности повторить отдельный шаг.

Поэтому он делается после стабильной платформы.

## Что входит в первую версию Team Mode

- страница `/team`;
- выбор шаблона команды;
- выбор задачи;
- последовательное выполнение шагов;
- отображение каждого шага;
- сохранение `team_runs`;
- сохранение `team_steps`;
- финальный ответ;
- обработка ошибок отдельных шагов;

# Итоговое правило

Проект нужно развивать поэтапно.

Главное правило:

```text
Сначала стабильный Prompt Arena.

Потом Code Arena Lite, Battle, Judge и Leaderboard.
```
