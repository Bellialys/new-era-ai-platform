# 06 - Режимы проекта

## Назначение файла

Этот файл описывает режимы работы проекта **Новая эпоха**.

Режимы - продуктовая основа платформы. Пользователь получает не один ответ от одной модели, а несколько вариантов от разных моделей, сравнивает их, выбирает лучший результат, сохраняет историю и постепенно формирует основу для рейтинга качества моделей.

Главный источник порядка версий - `14-roadmap.md`.

Если этот файл конфликтует с `14-roadmap.md`, правильным считается `14-roadmap.md`.

---

# Основная идея режимов

Пользователь выбирает режим, вводит задачу, выбирает модели или использует рекомендованный набор, после чего backend отправляет запросы нескольким AI-моделям через OpenRouter API.

Ответы отображаются рядом, сравниваются, оцениваются пользователем или моделью-судьёй и сохраняются в Supabase.

Базовая логика:

```text
Пользователь вводит задачу.
# исходный prompt или coding task

Пользователь выбирает режим.
# Prompt Arena, Code Arena, Image Arena, Battle, Judge, Leaderboard или Team Mode

Пользователь выбирает модели.
# вручную или через рекомендованный preset

Frontend отправляет запрос в Next.js API Route.
# секретные ключи не попадают в браузер

Backend вызывает OpenRouter API.
# все AI-запросы проходят только через сервер

Система получает ответы моделей.
# успешные ответы и ошибки обрабатываются отдельно

Результаты сохраняются в Supabase.
# данные нужны для истории, голосования и рейтинга

Пользователь или Judge выбирает лучший результат.
# победитель используется для статистики и Leaderboard
```

# Продуктовые режимы

| Номер | Режим | Назначение |
|---:|---|---|
| 1 | Prompt Arena | Сравнение обычных ответов моделей на один prompt |
| 2 | Code Arena | Сравнение ответов моделей на задачи программирования |
| 3 | Multi Model Battle | Формальное соревнование моделей с победителем |
| 4 | AI Team Mode | Совместная работа моделей с разными ролями |
| 5 | Judge Mode | Оценка ответов отдельной моделью-судьёй |
| 6 | Leaderboard | Рейтинг моделей по результатам сравнений |
| 7 | Image Arena / Visual Arena | Сравнение изображений, сгенерированных разными image-моделями |

Важно: продуктовая нумерация показывает идею платформы, но не является порядком разработки.

| `v0.7` | History | Нужна история сравнений |
| `v1.0` | Stable Prompt Arena | Первый стабильный рабочий MVP |

# Единый стиль имён

## Mode slug

Для режимов используется `kebab-case`.

```ts
type ModeSlug =
  | "prompt-arena"
  | "code-arena"
  | "multi-model-battle"
  | "judge-mode"
  | "leaderboard"
  | "code-arena-runner"
  | "image-arena"
  | "ai-team-mode";
```

## API и TypeScript

Frontend, API JSON и TypeScript используют `camelCase`.

```ts
type BaseTask = {
  id: string;
  modeSlug: ModeSlug;
  promptText: string;
  selectedModelIds: string[];
  createdAt: string;
};
```

```ts
type ModelResponse = {
  id: string;
  taskId: string;
  modelId: string;
  content: string | null;
  errorMessage: string | null;
  latencyMs: number | null;
  inputTokens: number | null;
  outputTokens: number | null;
  totalCost: number | null;
  createdAt: string;
};
```

## Database

Supabase PostgreSQL использует `snake_case`.

```text
mode_slug
# режим задачи в базе

task_text
# текст задачи в базе

selected_models
# выбранные модели в jsonb

task_id
# связь ответа или голоса с задачей

model_response_id
# связь голоса с ответом
```

Правило:

```text
API/TypeScript = camelCase.
# удобно для Next.js и React

Database/PostgreSQL = snake_case.
# стандартно для SQL и Supabase
```

# Ограничение количества моделей

Для MVP:

```text
minimum selected models: 2
# меньше двух моделей нет смысла сравнивать

maximum selected models: 3
# ограничение расходов и нагрузки на MVP
```

После MVP лимит можно расширить через Admin Panel and Limits.

---

# Частичный успех

Ошибка одной модели не должна ломать весь результат.

Пример:

```text
Model A - ответила успешно.
# показываем ответ

Model B - timeout.
# показываем ошибку только в её карточке

Model C - ответила успешно.
# показываем ответ
```

Правило:

```text
Если хотя бы одна модель ответила успешно, пользователь должен увидеть результат.
# не превращаем всю задачу в failed из-за одной модели
```

# Режим 1 - Prompt Arena

## Версии

Prompt Arena реализуется поэтапно:

```text
v0.3 - UI MVP
# интерфейс без реальных AI-запросов, с mock-ответами

v0.4 - OpenRouter Integration
# реальные ответы моделей

v0.5.3 - Voting MVP stabilization
# сохранение задач, ответов и Winner vote, health-check, smoke-check и выровненные migrations

v0.6 - Auth, Guest Mode and Profile
# владелец задачи: пользователь или anonymous session

v0.7 - History
# история сравнений

v1.0 - Stable Prompt Arena
# первый стабильный MVP
```

## Назначение

Prompt Arena нужен, чтобы сравнить обычные ответы нескольких AI-моделей на один запрос пользователя.

Это главный режим MVP.

## Что входит в UI MVP v0.3

- страница `/arena`;
- поле ввода задачи;
- выбор 2-3 mock-моделей;
- mock-генерация ответов;
- loading, empty, error и success states;
- карточки ответов;
- локальный UI-выбор победителя.

Важно:

```text
v0.3 - локальный UI-выбор победителя.
# выбор не сохраняется после обновления страницы

v0.6 - saved vote.
# выбор сохраняется через /api/vote в Supabase
```

## Что входит в Stable Prompt Arena v1.0

- страница `/arena`;
- поле ввода задачи;
- выбор 2-3 моделей;
- отправка запроса через backend;
- вызов OpenRouter API;
- отображение ответов рядом;
- обработка ошибок отдельных моделей;
- сохранение задач в `tasks`;
- сохранение ответов в `model_responses`;
- сохранение голосов в `votes`;
- история через `/history` и `/history/[taskId]`.

# Режим 2 - Code Arena

Code Arena делится на две части.

```text
Code Arena Lite
# сравнение решений по коду без запуска чужого кода

Code Arena Runner
# запуск кода в sandbox и проверка тестами
```

## Code Arena Lite - v1.1

Можно делать после Stable Prompt Arena.

Входит:

- задача по коду;
- выбор языка;
- выбор 2-3 моделей с `supports_code = true`;
- ответы моделей с кодом;
- ручное сравнение;
- saved vote.

Не входит:

- запуск пользовательского кода;
- sandbox;
- тест-кейсы;
- автоматическое выполнение.

## Code Arena Runner - v1.7

Runner можно делать только после аккаунтов, лимитов, админ-панели и security review.

Запрещено до `v1.7`:

```text
Запускать пользовательский код.
# опасно без sandbox и лимитов
```

# Режим 6 - Image Arena / Visual Arena

Версия: `v1.8`.

Назначение: сравнивать изображения, которые разные image-capable модели генерируют по одной визуальной идее пользователя.

Сценарий:

- пользователь вводит одну визуальную идею;
- выбирает 2-3 модели с `image` output capability;
- backend вызывает image-capable модели через OpenRouter;
- изображения сохраняются в Supabase Storage в стабильном режиме; alpha backend может вернуть provider URL, если Storage upload/fetch недоступен;
- metadata и storage path сохраняются в Supabase PostgreSQL после выделенной persistence-задачи;
- UI показывает сетку результатов;
- пользователь выбирает лучший вариант.

Важно:

```text
Image Arena не входит в первый MVP.
# сначала нужна стабильная Prompt Arena, Storage, лимиты стоимости и safety-контроли
```

# Режим 7 - AI Team Mode

Версия: `v2.0`.

AI Team Mode является важным режимом продукта, но его нельзя делать в MVP.

Причины:

- много AI-запросов;
- высокая стоимость;
- нужна история шагов;
- нужны роли моделей;
- нужен контроль стоимости;
- нужна обработка ошибок цепочки;
- нужен стабильный Prompt Arena;
- нужны лимиты и админ-контроль.

Правило:

```text
AI Team Mode начинается только с v2.0.
# до этого режим остаётся только в документации
```
