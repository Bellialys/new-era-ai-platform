# 10 - UI Pages

## Назначение файла

Этот файл описывает страницы, экраны, компоненты и правила интерфейса проекта **Новая эпоха**.

Интерфейс проекта должен помогать пользователю быстро выполнить главный сценарий:

```text
Ввести задачу
# пользователь формулирует запрос

Выбрать несколько AI-моделей
# пользователь выбирает участников сравнения

Получить ответы рядом
# система показывает результаты в удобном виде

Выбрать лучший ответ
# пользователь голосует за лучший результат

Сохранить сравнение
# система сохраняет задачу, ответы и голос
```

Главная цель интерфейса - не создать обычный чат, а сделать удобную платформу для сравнения AI-моделей.

## Главный источник порядка версий

Порядок появления страниц должен соответствовать файлу:

```text
14-roadmap.md
# главный источник порядка версий проекта
```

Если этот файл конфликтует с `14-roadmap.md`, правильным считается порядок из `14-roadmap.md`.

## Общий принцип интерфейса

Проект **Новая эпоха** не должен выглядеть как один обычный AI-чат.

Обычный AI-чат:

```text
Один запрос -> одна модель -> один ответ
```

Проект **Новая эпоха**:

```text
Один запрос -> несколько моделей -> несколько ответов -> сравнение -> голос -> история -> рейтинг
```

Поэтому основной экран должен быть построен вокруг сравнения ответов, а не вокруг длинной переписки с одной моделью.

# Страница Prompt Arena

## Путь

```text
/arena
# главная рабочая страница MVP
```

## Версии развития

```text
v0.3
# UI с mock-ответами

v0.4
# реальные ответы OpenRouter

v0.5
# сохранение в Supabase

v0.6
# голосование за лучший ответ

v1.0
# стабильная Prompt Arena
```

## Назначение

Prompt Arena - главный экран MVP. Здесь пользователь вводит задачу, выбирает несколько моделей, получает ответы и выбирает лучший вариант.

## Основной сценарий

```text
Пользователь вводит prompt
# текст задачи

Пользователь выбирает 2-4 модели
# модели берутся из allowlist backend

Пользователь нажимает Run Arena
# запрос отправляется на backend

Backend отправляет запросы в OpenRouter
# ключ OpenRouter остаётся только на сервере

UI показывает ответы рядом
# каждая модель получает отдельную карточку

Пользователь выбирает победителя
# голос сохраняется в Supabase
```

## Структура страницы

```text
ArenaPage
# страница /arena

PromptInputPanel
# поле ввода задачи

ModelSelector
# выбор моделей

RunArenaButton
# запуск сравнения

ResponseGrid
# сетка ответов моделей

ResponseCard
# карточка одного ответа

VoteButton
# кнопка выбора победителя

ArenaStatusBar
# статус генерации и ошибок
```

## PromptInputPanel

Поле ввода задачи должно быть большим и удобным.

Минимальные требования:

- многострочный ввод;
- placeholder с примером;
- счётчик символов или токенов можно добавить позже;
- кнопка очистки не обязательна в MVP;
- нельзя отправлять пустой prompt.

Пример placeholder:

```text
Введите задачу для сравнения моделей. Например: объясни архитектуру MVP для AI-платформы или проверь этот код.
```

## ModelSelector

В MVP пользователь должен выбрать минимум 2 модели.

Правила:

```text
Минимум 2 модели
# иначе сравнение не имеет смысла

Максимум 4 модели в MVP
# чтобы не тратить слишком много токенов

Только активные модели
# is_active = true

Только публичные модели
# is_public = true
```

Карточка модели должна показывать:

- название;
- provider;
- price label;
- краткое назначение;
- отметку `free`, `cheap`, `balanced` или `expensive`.

## RunArenaButton

Кнопка запуска должна быть неактивной, если:

- prompt пустой;
- выбрано меньше 2 моделей;
- уже идёт генерация;
- backend временно недоступен.

Текст кнопки:

```text
Run Arena
# запустить сравнение
```

Во время загрузки:

```text
Generating responses...
# ответы генерируются
```

## ResponseGrid

Ответы моделей должны отображаться рядом.

На desktop:

```text
2 модели -> 2 колонки
3 модели -> 3 колонки
4 модели -> 2x2 сетка или 4 колонки, если хватает ширины
```

На mobile:

```text
1 колонка
# карточки идут одна под другой
```

## ResponseCard

Каждая карточка ответа должна содержать:

- название модели;
- provider;
- статус ответа;
- текст ответа;
- время генерации, если доступно;
- ошибку, если модель не ответила;
- кнопку выбора победителя после завершения генерации.

Статусы:

```text
Waiting
# запрос ещё не отправлен

Loading
# модель отвечает

Success
# ответ получен

Error
# модель не ответила или вернула ошибку
```

## VoteButton

Голосование появляется в `v0.6`.

Правила:

```text
Один task -> один выбранный победитель от пользователя
# для MVP достаточно одного голоса

Нельзя голосовать до завершения всех ответов
# чтобы сравнение было честным

Можно изменить голос до ухода со страницы
# удобно для пользователя
```

Текст кнопки:

```text
Choose as best
# выбрать этот ответ лучшим
```

После выбора:

```text
Best answer selected
# лучший ответ выбран
```

## Ошибки на Arena

UI должен показывать понятные ошибки:

```text
Prompt is empty
# пользователь не ввёл задачу

Select at least 2 models
# выбрано меньше двух моделей

Model did not respond
# одна модель не вернула ответ

Server error
# ошибка backend

OpenRouter request failed
# ошибка запроса к OpenRouter

Rate limit exceeded
# превышен лимит запросов
```

## Что нельзя делать в Prompt Arena MVP

- не делать полноценный чат с историей сообщений;
- не добавлять сложные цепочки агентов;
- не запускать код;
- не делать AI Team Mode внутри Arena;
- не добавлять оплату;
- не давать пользователю вводить произвольный OpenRouter model ID;
- не показывать backend secrets в UI.

# Страница Judge Mode

## Путь

```text
/judge
# страница Judge Mode
```

## Версия

```text
v1.3
# после Multi Model Battle
```

## Назначение

Judge Mode позволяет отдельной модели оценить ответы других моделей.

## Основной сценарий

```text
Пользователь вводит задачу
# исходный prompt

Система получает ответы моделей
# участники дают ответы

Judge model оценивает ответы
# отдельная модель анализирует качество

UI показывает оценку
# пользователь видит аргументы судьи
```

## Что показывать

- исходную задачу;
- ответы участников;
- модель-судью;
- оценку по критериям;
- объяснение выбора;
- итогового победителя.

## Важное правило

Judge Mode не должен заменять голос пользователя. В MVP Judge Mode лучше считать рекомендацией, а не абсолютной истиной.

## Назначение

Admin Panel нужна для управления проектом.

## Что можно делать в первой версии

- включать и отключать модели;
- менять `sort_order`;
- менять `price_label`;
- включать и отключать режимы;
- смотреть usage logs;
- смотреть ошибки OpenRouter;
- управлять базовыми лимитами.

## Что нельзя делать

- нельзя открывать админку обычным пользователям;
- нельзя показывать API keys;
- нельзя редактировать secrets из UI;
- нельзя удалять данные без подтверждения;
- нельзя запускать Code Runner без ограничений.

# Страница Code Runner Results

## Путь

```text
/code/runs/[runId]
# результаты запуска кода
```

## Версия

```text
v1.7
# Code Arena Runner
```

## Назначение

Эта страница появляется только после реализации безопасного sandbox.

Она показывает:

- исходную задачу;
- код модели;
- язык программирования;
- тесты;
- stdout;
- stderr;
- exit code;
- статус timeout;
- результат выполнения.

## Главное ограничение

До `v1.7` этой страницы быть не должно.

```text
Code Arena Lite
# только анализ кода без запуска

Code Arena Runner
# запуск кода только после sandbox, лимитов и security review
```

## Назначение

AI Team Mode - сложный режим, где модели работают как команда с ролями.

## Возможные роли

```text
Planner
# планирует решение

Coder
# пишет код

Reviewer
# проверяет качество

Security Reviewer
# ищет риски безопасности

Judge
# оценивает итог
```

## Почему не делать раньше

AI Team Mode нельзя делать в MVP, потому что он требует:

- стабильной Prompt Arena;
- понятной работы моделей;
- истории запусков;
- контроля стоимости;
- лимитов;
- логирования;
- хорошего UI для цепочек шагов.

# Адаптивность интерфейса

## Desktop

На desktop основной режим должен использовать ширину экрана для сравнения ответов.

```text
Prompt слева или сверху
# зависит от выбранной верстки

Ответы моделей рядом
# главное преимущество проекта
```

## Mobile

На mobile карточки ответов должны идти одна под другой.

Требования:

- текст не должен выходить за границы карточки;
- кнопки должны быть достаточно крупными;
- длинные ответы должны нормально прокручиваться;
- таблицы не должны ломать экран;
- модель и статус должны быть видны в начале карточки.

## Минимальные размеры

```text
Mobile
# от 360 px ширины

Tablet
# от 768 px ширины

Desktop
# от 1024 px ширины
```

# UI-состояния

Каждая важная страница должна иметь состояния.

## Loading

```text
Loading...
# данные загружаются
```

Используется при:

- загрузке моделей;
- генерации ответов;
- загрузке истории;
- сохранении голоса.

## Empty

```text
No results yet
# пока нет данных
```

Используется при:

- пустой истории;
- отсутствии ответов;
- отсутствии доступных моделей.

## Error

```text
Something went wrong
# произошла ошибка
```

Ошибка должна быть понятной, но не должна раскрывать секреты backend.

## Success

```text
Saved
# действие успешно выполнено
```

Используется при:

- сохранении задачи;
- сохранении голоса;
- успешном ответе модели.

# Правила UX

## Главные правила

- пользователь всегда должен понимать, что происходит;
- нельзя оставлять пустой экран без объяснения;
- ошибки должны быть человеческими;
- будущие функции нужно помечать как `Coming soon`;
- основная кнопка должна быть одна на экран;
- нельзя смешивать все режимы в одном интерфейсе;
- сначала простота, потом сложность.

## Как показывать будущие режимы

Будущие режимы можно показывать на главной странице, но с пометкой:

```text
Coming soon
# режим ещё не готов
```

Нельзя делать так, чтобы пользователь думал, что режим уже работает.

## Как показывать модели

Модели нужно показывать понятными карточками.

Пример:

```text
GPT model
Provider: OpenAI
Price: expensive
Role: judge, prompt
```

Не нужно перегружать пользователя внутренними техническими ID модели.

Model ID можно показывать только в админке.

# Чек-лист готовности UI MVP

## v0.3 - UI MVP готов, если

- есть страница `/`;
- есть страница `/arena`;
- можно ввести prompt;
- можно выбрать mock-модели;
- можно нажать кнопку запуска;
- mock-ответы отображаются рядом;
- карточки ответов не ломаются на mobile;
- будущие режимы помечены как `Coming soon`;
- проект проходит `npm run build`.

## v0.6 - Voting MVP готов, если

- ответы моделей сохраняются;
- после ответов можно выбрать лучший вариант;
- выбранный ответ визуально выделяется;
- голос сохраняется в Supabase;
- повторное голосование не ломает данные;
- ошибка сохранения голоса показывается пользователю.

## v0.7 - History MVP готов, если

- есть страница `/history`;
- есть страница `/history/[taskId]`;
- список истории загружается из Supabase;
- можно открыть сохранённое сравнение;
- показываются prompt, ответы и победитель;
- пустая история имеет понятное сообщение.

## v1.0 - Stable Prompt Arena готова, если

- основной сценарий работает от начала до конца;
- пользователь может создать сравнение;
- модели дают ответы;
- ответы сохраняются;
- пользователь выбирает победителя;
- история открывается;
- UI не ломается на desktop и mobile;
- ошибки показываются понятно;
- секреты не попадают в frontend;
- проект опубликован на Vercel.

# Итог

Интерфейс проекта должен развиваться поэтапно.

Правильный порядок:

```text
Home + Prompt Arena UI
# v0.3

Real responses
# v0.4

Saved results
# v0.5

Voting
# v0.6

History
# v0.7

Stable Prompt Arena
# v1.0

Future modes
# после v1.0
```

Главное правило: сначала сделать одну стабильную Prompt Arena, а потом постепенно добавлять Code Arena, Battle, Judge Mode, Leaderboard, аккаунты, админку, Runner и AI Team Mode.
