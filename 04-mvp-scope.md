# 04 - Область MVP

## Назначение файла

Этот файл фиксирует границы MVP проекта **Новая эпоха**.

MVP показывает главную ценность проекта: пользователь отправляет одну задачу нескольким AI-моделям, получает ответы рядом, сравнивает их и выбирает лучший вариант.

Главный источник порядка версий - `14-roadmap.md`.

Если этот файл конфликтует с `14-roadmap.md`, правильным считается `14-roadmap.md`.

---

# Главный принцип MVP

MVP должен быть простым, рабочим и проверяемым.

```text
Сначала делаем стабильную Prompt Arena.
# это главная ценность проекта на старте

Не добавляем сложные режимы до готового MVP.
# иначе проект станет слишком большим и неуправляемым

Каждый этап должен давать рабочий результат.
# после каждого этапа можно проверить сайт руками
```

---

# Два уровня MVP

## 1. UI MVP - v0.3

UI MVP - первая видимая версия интерфейса без реального AI, базы данных и сохранения истории.

Входит:

- главная страница `/`;
- страница `/arena`;
- поле ввода prompt;
- выбор 2-3 mock-моделей;
- кнопка запуска сравнения;
- empty state;
- loading state;
- error state;
- success state;
- mock-ответы моделей;
- карточки ответов;
- локальный UI-выбор победителя;
- кнопка очистки формы.

Не входит:

- OpenRouter API;
- Supabase PostgreSQL;
- реальные API routes;
- сохранение задач;
- сохранение ответов;
- сохранение голосов;
- история сравнений;
- production deploy.

Важно:

```text
v0.3 - UI winner selection
# пользователь может визуально выбрать победителя только в состоянии текущей страницы
# после обновления страницы выбор не сохраняется
```

## 2. Stable Prompt Arena MVP - v1.0

Stable Prompt Arena MVP - первая стабильная рабочая версия продукта.

Входит:

- реальные ответы моделей через OpenRouter;
- backend API routes;
- сохранение задач в Supabase;
- сохранение ответов в Supabase;
- сохранение ошибок моделей;
- сохранение голоса через `/api/vote`;
- история сравнений;
- деплой на Vercel;
- базовые лимиты;
- безопасное хранение ключей;
- стабильная Prompt Arena.

```text
v1.0 - Stable Prompt Arena
# Prompt Arena работает через OpenRouter, Supabase, Voting и History
```

# Главный пользовательский сценарий v1.0

Stable Prompt Arena считается готовой, когда пользователь может:

1. открыть сайт;
2. перейти в Prompt Arena;
3. ввести задачу;
4. выбрать 2 или 3 AI-модели;
5. отправить запрос через backend;
6. получить ответы от выбранных моделей;
7. сравнить ответы рядом;
8. выбрать лучший ответ;
9. сохранить результат в базе данных;
10. открыть результат позже в истории.

# Единые MVP-лимиты

Эти лимиты должны совпадать в документации, frontend, backend, `.env.example` и базе данных.

```text
MIN_PROMPT_LENGTH = 3
# минимальная длина prompt

MAX_PROMPT_LENGTH = 8000
# максимальная длина prompt для MVP

MIN_MODELS_PER_COMPARE = 2
# меньше двух моделей сравнение теряет смысл

MAX_MODELS_PER_COMPARE = 3
# защита от расходов, задержки и перегрузки UI

MODEL_TIMEOUT_MS = 60000
# базовый timeout ответа модели
```

`MAX_PROMPT_LENGTH = 8000` выбран потому, что этот лимит уже согласован с `.env.example` и SQL-проверкой `tasks_prompt_text_length` в `08-database.md`.

# Страница Prompt Arena

Путь:

```text
/arena
# страница сравнения AI-моделей
```

Для `v0.3` на странице должны быть:

- поле ввода задачи;
- выбор mock-моделей;
- кнопка запуска;
- состояние загрузки;
- сообщения об ошибках;
- mock-ответы;
- карточки ответов;
- локальный выбор победителя.

Для `v1.0` дополнительно должны быть:

- реальные модели из allowlist;
- реальные ответы через OpenRouter;
- сохранение задач и ответов;
- сохранение голоса;
- ссылка на историю результата.

---

# Поле ввода задачи

Минимальные требования:

```text
Поле принимает обычный текст.
# пользователь вводит prompt

Пустой запрос нельзя отправить.
# защита от бессмысленных запросов

Слишком длинный запрос нужно ограничить.
# защита от лишних расходов на API
```

Ограничения:

```text
Минимум: 3 символа
# защита от пустого ввода

Максимум: 8000 символов
# единый лимит MVP
```

---

# Выбор AI-моделей

Пользователь должен выбрать минимум две модели.

```text
Минимум: 2 модели
# иначе сравнение не имеет смысла

Максимум: 3 модели
# защита от лишних расходов и перегрузки интерфейса
```

Список моделей должен приходить из allowlist проекта.

Источники правил:

```text
11-ai-models.md
# стратегия выбора моделей

08-database.md
# структура таблицы models

.env.example
# технические лимиты MVP
```

Точные OpenRouter model ID нельзя считать вечными. Перед production их нужно проверять через OpenRouter Models API.

---

# Отправка запроса через backend

Frontend не должен напрямую обращаться к OpenRouter.

Правильная схема:

```text
Frontend -> Next.js API Route -> OpenRouter API
# секретный ключ остаётся на сервере
```

Главный API-маршрут:

```text
POST /api/compare
# отправляет задачу выбранным моделям и возвращает ответы
```

Важное правило:

```text
OPENROUTER_API_KEY хранится только на backend.
# ключ нельзя отдавать в браузер
```

---

# Ответы моделей

Для каждой модели нужно отображать:

- название модели;
- статус ответа;
- текст ответа;
- ошибку, если модель не ответила;
- время выполнения, если это не усложняет интерфейс.

Статусы для MVP:

```text
pending
# ответ ещё загружается

success
# модель успешно ответила

error
# модель вернула ошибку или не ответила
```

Ошибка одной модели не должна ломать всё сравнение.

# Голосование и выбор победителя

Нужно различать два уровня.

## UI-выбор победителя - v0.3

```text
Пользователь нажимает "Выбрать лучшим".
# карточка визуально выделяется

Выбор хранится только в состоянии страницы.
# после обновления страницы выбор пропадает
```

Это нужно, чтобы проверить UX до подключения Supabase.

## Saved vote - v0.6

```text
Frontend отправляет taskId и responseId в /api/vote.
# backend проверяет корректность выбора

Backend сохраняет запись в votes.
# данные пригодятся для истории и Leaderboard

vote_type = user
# обычный голос пользователя
```

Полноценное голосование относится к `v0.6`, а не к `v0.3`.

# Сохранение данных в Supabase

Сохранение появляется с `v0.5` и усиливается в `v0.6-v0.7`.

Минимальные таблицы:

```text
models
# allowlist доступных моделей

tasks
# задачи пользователей

model_responses
# ответы AI-моделей

votes
# выбор лучшего ответа
```

В Stable Prompt Arena нужно сохранять:

- текст задачи;
- выбранный режим;
- список выбранных моделей;
- ответы моделей;
- статусы ответов;
- ошибки моделей;
- выбранный лучший ответ;
- дату создания.

# Что не входит в MVP

В ранний MVP не входят:

- AI Team Mode;
- Code Arena Runner;
- запуск чужого кода;
- сложная админ-панель;
- платёжная система;
- публичные профили;
- сложная аналитика;
- командные цепочки моделей;
- продвинутый Leaderboard;
- полноценная система тарифов.

Эти функции добавляются позже по `14-roadmap.md`.

---

# Почему AI Team Mode не входит в MVP

AI Team Mode требует ролей, шагов, истории командной работы, логирования стоимости и сложной обработки ошибок.

Правильный этап:

```text
v2.0 - AI Team Mode
# командная работа моделей только после стабильной платформы
```

---

# Почему Code Arena Runner не входит в MVP

Запуск чужого кода требует sandbox, timeout, лимитов, логирования и security review.

Правильный этап:

```text
v1.7 - Code Arena Runner
# только после аккаунтов, лимитов и админ-контроля
```

---

# Критерии готовности v1.0

Stable Prompt Arena готова, если:

- `/` открывается;
- `/arena` открывается;
- prompt проходит валидацию;
- пользователь выбирает 2-3 модели;
- backend вызывает OpenRouter;
- ответы отображаются рядом;
- ошибки отдельных моделей не ломают весь запрос;
- задачи сохраняются в `tasks`;
- ответы сохраняются в `model_responses`;
- голос сохраняется в `votes`;
- история открывается через `/history`;
- проект развёрнут на Vercel;
- секреты не попали в GitHub;
- `.env.local` не коммитится;
- базовые лимиты работают.

# Актуализация v0.4.1

Текущий проект уже прошёл стадию чистого Static UI MVP.

Сейчас готово:

- Prompt Arena UI;
- загрузка моделей через `/api/models`;
- отправка prompt через `/api/compare`;
- реальные ответы через OpenRouter на backend;
- server-side allowlist моделей;
- проверка `prompt`, `modelIds`, `modeSlug`;
- безопасные API-ошибки.
- базовый in-memory rate limit на `/api/compare`;
- заготовка Supabase persistence для `tasks` и `model_responses`;
- сохранение token usage в `model_responses`, когда Supabase настроен.

Следующая граница MVP:

```text
v0.5 - Supabase Integration
# завершить Supabase flow: модели из БД, UUID на клиенте, model_key только server-side
```

Всё ещё не входит в ближайший MVP:

- Code Arena Runner;
- AI Team Mode;
- Leaderboard;
- сложная админ-панель;
- платёжная система;
- запуск пользовательского кода.
