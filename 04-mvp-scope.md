# 04 - Область MVP

## Назначение файла

Этот файл фиксирует границы MVP проекта **Новая эпоха**.

MVP - это минимальная рабочая версия продукта, которая уже показывает главную ценность проекта: пользователь отправляет одну задачу нескольким AI-моделям, получает ответы рядом, сравнивает их и выбирает лучший вариант.

Подробный порядок версий находится в файле `14-roadmap.md`.
Этот файл отвечает только за границы MVP: что входит в первую рабочую версию, что можно отложить и что нельзя добавлять слишком рано.

## Главный принцип MVP

MVP должен быть простым, рабочим и проверяемым.

Главное правило:

```text
Сначала делаем стабильную Prompt Arena.
# это главная ценность проекта на старте

Не добавляем сложные режимы до готового MVP.
# иначе проект станет слишком большим и неуправляемым

Каждый этап должен давать рабочий результат.
# после каждого этапа можно проверить сайт руками
```

MVP не должен быть идеальной финальной платформой. Он должен доказать, что основная идея работает.

## Главная цель MVP

Создать рабочую веб-платформу, где пользователь может:

1. Открыть сайт.
2. Перейти в Prompt Arena.
3. Ввести задачу.
4. Выбрать 2 или 3 AI-модели.
5. Отправить запрос через backend.
6. Получить ответы от выбранных моделей.
7. Сравнить ответы рядом.
8. Выбрать лучший ответ.
9. Сохранить результат в базе данных.

Если этот сценарий работает стабильно, MVP считается успешным.

## Как понимать MVP в этом проекте

В проекте есть два уровня MVP.

### 1. UI MVP

UI MVP - это первая видимая версия интерфейса без полноценной базы и без всех серверных функций.

Он соответствует ранним этапам roadmap:

```text
v0.2 - Base Next.js Project
# создана основа проекта

v0.3 - Static UI MVP
# есть страницы и интерфейс Arena
```

UI MVP нужен, чтобы быстро увидеть внешний вид проекта и проверить удобство интерфейса.

### 2. Stable Prompt Arena MVP

Stable Prompt Arena MVP - это полноценная первая рабочая версия.

Она соответствует этапу:

```text
v1.0 - Stable Prompt Arena
# Prompt Arena работает через OpenRouter, Supabase и voting
```

Именно `v1.0` считается настоящим MVP проекта.

## Что входит в MVP

В MVP входит только то, что нужно для проверки основной идеи Prompt Arena.

## 1. Главная страница

Главная страница должна кратко объяснять, что делает проект.

Минимальное содержимое:

- название проекта;
- краткое описание;
- кнопка перехода в Arena;
- список основных возможностей;
- пометка, что это первая версия.

Путь страницы:

```text
/
# главная страница проекта
```

На старте можно сделать главную страницу очень простой. Главное - чтобы пользователь понимал, куда нажимать дальше.

## 2. Страница Prompt Arena

Prompt Arena - главный экран MVP.

Путь страницы:

```text
/arena
# страница сравнения AI-моделей
```

На странице должны быть:

- поле ввода задачи;
- выбор моделей;
- кнопка отправки;
- состояние загрузки;
- карточки ответов;
- кнопка выбора лучшего ответа;
- сообщения об ошибках.

## 3. Поле ввода задачи

Пользователь должен иметь возможность ввести текстовую задачу.

Минимальные требования:

```text
Поле принимает обычный текст.
# пользователь вводит промпт

Пустой запрос нельзя отправить.
# защита от бессмысленных запросов

Слишком длинный запрос нужно ограничить.
# защита от лишних расходов на API
```

Рекомендуемые ограничения для MVP:

```text
Минимум: 3 символа
# защита от пустого ввода

Максимум: 4000 символов
# достаточно для обычных задач и безопаснее по расходам
```

Точное ограничение можно изменить позже после тестирования.

## 4. Выбор AI-моделей

Пользователь должен выбрать минимум две модели.

Для MVP достаточно:

```text
Минимум: 2 модели
# иначе сравнение не имеет смысла

Максимум: 3 модели
# защита от лишних расходов и перегрузки интерфейса
```

Список моделей должен приходить из allowlist проекта.

Источник модели:

```text
11-ai-models.md
# стратегия выбора моделей

08-database.md
# структура таблицы models
```

Точные OpenRouter model ID нельзя считать вечными. Перед production их нужно проверять через OpenRouter Models API.

## 5. Отправка запроса через backend

Фронтенд не должен напрямую обращаться к OpenRouter.

Правильная схема:

```text
Frontend -> Next.js API Route -> OpenRouter API
# секретный ключ остаётся на сервере
```

Главный API-маршрут MVP:

```text
POST /api/compare
# отправляет задачу выбранным моделям и возвращает ответы
```

Важное правило:

```text
OPENROUTER_API_KEY хранится только на backend.
# ключ нельзя отдавать в браузер
```

## 6. Получение ответов от моделей

Система должна получить ответы от выбранных моделей и показать их пользователю.

Для каждой модели нужно отображать:

- название модели;
- технический model key;
- статус ответа;
- текст ответа;
- ошибку, если модель не ответила;
- время выполнения, если это не усложняет MVP.

Статусы для MVP:

```text
pending
# ответ ещё загружается

success
# модель успешно ответила

error
# модель вернула ошибку или не ответила
```

Ошибка одной модели не должна ломать все сравнение.

## 7. Карточки ответов

Ответы моделей нужно показывать рядом в отдельных карточках.

Минимальная структура карточки:

```text
Название модели
# пользователь видит, кто дал ответ

Статус ответа
# success, pending или error

Текст ответа
# основной результат модели

Кнопка выбора лучшего ответа
# пользователь голосует за лучший вариант
```

Для MVP не нужно делать сложную аналитику внутри карточки.

## 8. Голосование за лучший ответ

Пользователь должен выбрать лучший ответ.

Минимальная логика:

```text
Пользователь нажимает "Выбрать лучшим".
# фиксирует лучший ответ по мнению пользователя

Выбранная карточка визуально выделяется.
# пользователь видит свой выбор

Голос сохраняется в Supabase.
# данные пригодятся для истории и Leaderboard
```

Голосование нужно уже в MVP, потому что без него проект не собирает данные для будущего рейтинга моделей.

## 9. Сохранение данных в Supabase

MVP должен сохранять основные данные.

Минимальные таблицы:

```text
models
# allowlist доступных моделей

tasks
# задачи пользователей

model_responses
# ответы AI-моделей

votes
# выбор лучшего ответа
```

Полная структура таблиц описана в `08-database.md`.

В MVP нужно сохранять:

- текст задачи;
- выбранный режим;
- список выбранных моделей;
- ответы моделей;
- статусы ответов;
- ошибки моделей;
- выбранный лучший ответ;
- дату создания.

## 10. Простая история

История относится к этапу:

```text
v0.7 - History MVP
# пользователь может открыть прошлые сравнения
```

Минимальный путь:

```text
/history
# страница истории сравнений
```

История желательна для `v1.0`, но её можно делать после базовой Arena, OpenRouter, Supabase и Voting.

Главное: даже до готовой страницы `/history` данные уже должны сохраняться в Supabase.

## Что не входит в MVP

Чтобы проект не развалился от сложности, в MVP не входят следующие функции.

## 1. AI Team Mode

AI Team Mode не входит в MVP.

Причина:

```text
AI Team Mode требует ролей, шагов, истории командной работы и сложной логики.
# это режим уровня v2.0, а не MVP
```

Файл спецификации: `18-team-mode-spec.md`.

## 2. Code Arena Runner

Запуск чужого кода не входит в MVP.

Причина:

```text
Runner требует sandbox, лимитов, логирования и защиты от вредного кода.
# нельзя запускать чужой код без безопасной архитектуры
```

Правильный этап:

```text
v1.7 - Code Arena Runner
# только после аккаунтов, лимитов и админ-контроля
```

## 3. Полный Leaderboard

Полный рейтинг моделей не входит в MVP.

Причина:

```text
Для рейтинга сначала нужно накопить задачи, ответы и голоса.
# без данных Leaderboard будет искусственным
```

На этапе MVP можно только сохранять данные для будущего Leaderboard.

## 4. Judge Mode

Judge Mode не входит в MVP.

Причина:

```text
Автоматическая оценка ответов требует отдельной логики и более дорогих моделей.
# сначала нужна обычная Prompt Arena
```

Judge Mode можно делать после Multi Model Battle и накопления базовой логики сравнения.

## 5. Полная авторизация пользователей

В MVP можно работать без сложных аккаунтов.

Допустимые варианты:

```text
Без авторизации
# самый простой старт

Анонимная сессия
# можно связать действия пользователя через session id

Позже Supabase Auth
# полноценные аккаунты после стабильного MVP
```

Полные аккаунты лучше добавить после `v1.0`, когда Prompt Arena уже работает.

## 6. Платежи и подписки

Платежи не входят в MVP.

Причина:

```text
Сначала нужно доказать ценность продукта.
# оплата добавляется только после рабочего ядра
```

До платежей должны быть готовы:

- стабильная Arena;
- лимиты;
- учёт расходов;
- аккаунты;
- базовая админка.

## 7. Сложная админ-панель

Полная админ-панель не входит в MVP.

В MVP достаточно технических настроек в базе и env.

Полная админка нужна позже:

```text
v1.6 - Admin Panel and Limits
# управление моделями, лимитами и пользователями
```

## 8. Сложная аналитика

В MVP не нужны:

- графики;
- финансовые отчёты;
- сложная статистика токенов;
- публичные рейтинги;
- детальные отчёты по каждому пользователю.

Можно сохранять базовые поля, чтобы добавить аналитику позже.

## 9. Мобильное приложение

В MVP нужен только веб-сайт.

Мобильное приложение не делаем.

Но сайт должен нормально открываться на телефоне, потому что пользователь может тестировать проект с мобильного устройства.

## Основной пользовательский сценарий MVP

### Сценарий 1 - сравнение ответов моделей

1. Пользователь открывает сайт.
2. Переходит на страницу `/arena`.
3. Вводит задачу.
4. Выбирает 2 или 3 модели.
5. Нажимает кнопку отправки.
6. Система проверяет ввод.
7. Система отправляет запрос на `/api/compare`.
8. Backend отправляет запросы к OpenRouter.
9. Модели возвращают ответы.
10. Ответы сохраняются в Supabase.
11. Ответы отображаются в карточках.
12. Пользователь сравнивает ответы.
13. Пользователь выбирает лучший ответ.
14. Голос сохраняется в Supabase.
15. Пользователь видит подтверждение выбора.

Ожидаемый результат:

```text
Пользователь видит несколько ответов рядом и может выбрать лучший.
# основная ценность проекта доказана
```

## Дополнительные сценарии MVP

### Сценарий 2 - пустой запрос

Шаги:

1. Пользователь открывает `/arena`.
2. Не вводит текст.
3. Нажимает кнопку отправки.

Ожидаемый результат:

```text
Запрос не отправляется.
# пустой ввод не должен уходить в API

Показывается ошибка: "Введите задачу перед отправкой".
# пользователь понимает, что нужно исправить
```

### Сценарий 3 - выбрана только одна модель

Шаги:

1. Пользователь вводит задачу.
2. Выбирает только одну модель.
3. Нажимает кнопку отправки.

Ожидаемый результат:

```text
Запрос не отправляется.
# сравнение невозможно без минимум двух моделей

Показывается ошибка: "Выберите минимум две модели".
# пользователь получает понятное объяснение
```

### Сценарий 4 - выбрано слишком много моделей

Шаги:

1. Пользователь выбирает больше трёх моделей.
2. Пытается отправить запрос.

Ожидаемый результат:

```text
Система ограничивает выбор до трёх моделей.
# защита расходов и интерфейса

Показывается подсказка: "В MVP можно выбрать максимум три модели".
# пользователь понимает ограничение
```

### Сценарий 5 - ошибка одной модели

Шаги:

1. Пользователь отправляет запрос трём моделям.
2. Одна модель возвращает ошибку.
3. Две другие модели отвечают успешно.

Ожидаемый результат:

```text
Успешные ответы отображаются.
# ошибка одной модели не ломает весь результат

В карточке проблемной модели отображается ошибка.
# пользователь видит, какая модель не ответила
```

### Сценарий 6 - ошибка OpenRouter API

Шаги:

1. Пользователь отправляет запрос.
2. OpenRouter недоступен или ключ неправильный.

Ожидаемый результат:

```text
Система показывает понятную ошибку.
# пользователь не видит технический хаос

Секретные ключи не отображаются в браузере.
# безопасность не нарушена
```

### Сценарий 7 - повторное голосование

Шаги:

1. Пользователь выбирает лучший ответ.
2. Потом выбирает другой ответ.

Ожидаемый результат для MVP:

```text
Голос обновляется или сохраняется как последний выбор.
# поведение должно быть предсказуемым

В интерфейсе выделена только одна карточка.
# пользователь видит актуальный выбор
```

Для MVP лучше разрешить один активный голос на одну задачу.

## Минимальные API-маршруты

Подробная структура API описана в `09-api-structure.md`.

Для MVP нужны такие маршруты.

## POST /api/compare

Назначение:

```text
Отправить задачу выбранным моделям и вернуть ответы.
# главный API-маршрут MVP
```

Входные данные:

```text
prompt
# текст задачи пользователя

modelKeys
# список выбранных моделей из allowlist
```

Выходные данные:

```text
taskId
# идентификатор сохранённой задачи

responses
# ответы моделей

errors
# ошибки отдельных моделей
```

## POST /api/vote

Назначение:

```text
Сохранить выбор лучшего ответа.
# основа будущей статистики и Leaderboard
```

Входные данные:

```text
taskId
# идентификатор задачи

responseId
# ответ, который выбран лучшим
```

Выходные данные:

```text
success
# голос сохранён или обновлён
```

## GET /api/models

Назначение:

```text
Получить список доступных моделей.
# интерфейс не должен хранить список моделей вручную
```

На самом раннем этапе `v0.3` можно использовать временный список в коде. Но для `v1.0` лучше получать модели через backend.

## GET /api/history

Назначение:

```text
Получить список прошлых сравнений.
# нужно для страницы /history на v0.7
```

Этот маршрут не обязателен до этапа `v0.7`.

## Требования к безопасности MVP

Даже MVP должен быть безопасным.

## 1. Защита API-ключей

```text
Нельзя вставлять API-ключи прямо в код.
# они попадут в GitHub

Нельзя использовать OPENROUTER_API_KEY на клиенте.
# пользователь сможет увидеть ключ в браузере

Нельзя логировать секретные ключи.
# ключи могут попасть в логи Vercel
```

## 2. Проверка пользовательского ввода

Нужно проверять:

- пустой запрос;
- слишком короткий запрос;
- слишком длинный запрос;
- отсутствие выбранных моделей;
- выбор моделей не из allowlist;
- слишком большое количество моделей.

## 3. Контроль расходов

Для MVP обязательно ограничить:

```text
Максимум 3 модели за запрос.
# защита от резкого роста стоимости

Максимум 4000 символов в prompt.
# защита от слишком дорогих запросов

Только allowlist моделей.
# пользователь не может подставить дорогую модель вручную
```

## 4. Ошибки без утечки данных

Ошибки должны быть понятными, но безопасными.

Нельзя показывать пользователю:

- API-ключи;
- service role key;
- полный stack trace;
- внутренние server logs;
- приватные переменные окружения.

## Ограничения MVP

MVP должен иметь жёсткие ограничения.

```text
Один основной режим - Prompt Arena.
# другие режимы позже

Максимум 3 модели на один запрос.
# контроль расходов

Без запуска пользовательского кода.
# Runner только v1.7

# платежи позже

Без сложной админки.
# админка позже

Без AI Team Mode.
# командный режим только v2.0
```

## Критерии готовности MVP

MVP считается готовым, когда выполнены функциональные, технические и пользовательские критерии.

## Функциональные критерии

- сайт открывается локально;
- страница `/arena` открывается;
- пользователь может ввести задачу;
- пользователь может выбрать минимум две модели;
- нельзя выбрать больше трёх моделей;
- пустой запрос не отправляется;
- запрос отправляется через backend;
- backend обращается к OpenRouter;
- ответы моделей отображаются рядом;
- ошибка одной модели не ломает остальные ответы;
- задача и ответы сохраняются в Supabase;
- пользователь может выбрать лучший ответ;
- голос сохраняется в Supabase.

## Технические критерии

- проект запускается локально;
- проект собирается без критических ошибок;
- `.env.local` не попадает в GitHub;
- OpenRouter key используется только на сервере;
- Supabase service role key используется только на сервере;
- список моделей берётся из allowlist;
- изменённые файлы зафиксированы через Git commit;
- проект можно выложить на Vercel.

## Пользовательские критерии

- понятно, куда вводить задачу;
- понятно, какие модели выбраны;
- понятно, какой ответ от какой модели;
- понятно, что происходит во время загрузки;
- понятно, как выбрать лучший ответ;
- ошибки написаны человеческим языком;
- интерфейс не ломается на телефоне.

## Главный итог MVP

MVP проекта **Новая эпоха** должен доказать одну вещь: пользователю удобно сравнивать ответы разных AI-моделей в одном интерфейсе.

Если пользователь может быстро отправить задачу, увидеть ответы рядом и выбрать лучший вариант, значит первая цель достигнута.

После этого проект можно развивать дальше по roadmap, не ломая уже работающую основу.
