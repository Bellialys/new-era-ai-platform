## Краткое описание проекта

**Новая эпоха** - это веб-платформа, где пользователь вводит одну задачу, выбирает несколько AI-моделей и получает ответы рядом для сравнения.

Пользователь сможет:

- отправить один prompt нескольким моделям;
- сравнить ответы в удобном интерфейсе;
- выбрать лучший ответ;
- сохранить результат сравнения;
- смотреть историю прошлых сравнений;
- позже использовать Code Arena, Multi Model Battle, Judge Mode, Leaderboard и AI Team Mode.

Главная идея проекта - не просто получить один ответ от одной модели, а сравнить несколько моделей и понять, какая из них лучше справляется с конкретной задачей.

---

## Чем проект отличается от обычного AI-чата

Обычный AI-чат работает так:

```text
Пользователь -> одна модель -> один ответ
# пользователь не видит, как ответили бы другие модели
```

Проект **Новая эпоха** работает так:

```text
Пользователь -> несколько моделей -> несколько ответов -> сравнение -> голосование -> история -> рейтинг
# пользователь видит разные варианты и может выбрать лучший
```

Это делает проект полезным для:

- сравнения качества моделей;
- выбора лучшей модели под конкретную задачу;
- тестирования prompt-ов;
- анализа кода;
- оценки ответов;
- будущего рейтинга моделей.

## Главная цель проекта

Создать продвинутую AI-платформу для сравнения, оценки и командной работы нескольких AI-моделей.

Разработка должна идти поэтапно:

```text
Сначала простой рабочий MVP.
# нельзя начинать с огромной сложной системы

Потом стабильная Prompt Arena.
# сначала доказываем главную идею проекта

Потом дополнительные режимы.
# Code Arena Lite, Battle, Judge Mode, Leaderboard

Потом аккаунты и админ-контроль.
# нужны лимиты, история пользователей и управление стоимостью

Потом безопасный запуск кода.
# Code Arena Runner только после security review

Потом AI Team Mode.
# самый сложный режим переносится на v2.0
```

## Первый главный MVP

Первый главный MVP - это **Stable Prompt Arena**.

Он формируется постепенно от `v0.3` до `v1.0`.

В полный MVP входит:

```text
Главная страница.
# пользователь понимает, что делает проект

Prompt Arena.
# основной режим сравнения моделей

Поле ввода задачи.
# пользователь пишет prompt или задачу

Выбор 2-3 моделей.
# пользователь выбирает модели для сравнения

Отправка запроса через backend API.
# OpenRouter API вызывается только на серверной стороне

Карточки ответов.
# ответы моделей отображаются рядом

Голосование.
# пользователь выбирает лучший ответ

Сохранение в Supabase.
# база хранит задачу, ответы и голос

История сравнений.
# пользователь может открыть прошлые сравнения

Деплой на Vercel.
# проект доступен онлайн
```

## Режимы проекта

Проект планирует несколько режимов.

### 1. Prompt Arena

Основной первый режим.

Пользователь вводит задачу, выбирает модели и сравнивает ответы.

Статус:

```text

```

Сравнение моделей по задачам программирования без запуска кода.

Статус:

```text
v1.1

```

Формальный бой моделей по одному prompt.

Статус:

```text
v1.2
# режим соревнования моделей
```

### 4. Judge Mode

Одна модель оценивает ответы других моделей по критериям.

Статус:

```text
v1.3
# автоматическая оценка качества ответов
```

### 5. Leaderboard

Рейтинг моделей на основе голосов и оценок.

Статус:

```text
v1.4
```

Безопасный запуск кода в sandbox.

Статус:

```text
v1.7
```

### 7. AI Team Mode

Командная работа нескольких AI-моделей с ролями.

```text
v2.0
```

## Основные API MVP

Минимальные API routes:

```text
GET /api/models
# вернуть список активных моделей

POST /api/compare
# отправить задачу нескольким моделям и вернуть ответы

POST /api/vote
# сохранить выбор лучшего ответа

GET /api/history
# получить историю сравнений

GET /api/history/[taskId]
# получить одно сравнение по taskId
```

OpenRouter API вызывается только через backend route.

Frontend не должен напрямую обращаться к OpenRouter.

---

## Канонический контракт `/api/compare`

Документ `09-api-structure.md` является источником истины для будущего backend API.

API-ответ модели должен использовать:

```text
answerText
# текст успешного ответа модели

errorCode
# технический код ошибки модели, если status = error

errorMessage
# понятное сообщение об ошибке модели, если status = error
```

Не использовать как API-контракт:

```text
text
# временное поле старого UI/mock-типа

modelRole
# роль модели не должна дублироваться в API-ответе /api/compare
```

Правильное разделение:

```text
ArenaApiResponse
# то, что приходит от /api/compare

ArenaResponseView
# UI-представление, куда frontend добавляет modelRole по modelId
```

## Правила безопасности

Главные правила безопасности:

```text
OPENROUTER_API_KEY хранится только в .env.local и Vercel Environment Variables.
# нельзя вставлять ключ в frontend-код

SUPABASE_SERVICE_ROLE_KEY нельзя использовать на клиенте.
# service role key только для server-side кода

.env.local нельзя коммитить в GitHub.
# секреты не должны попасть в репозиторий

NEXT_PUBLIC нельзя использовать для секретных ключей.
# всё с NEXT_PUBLIC видно в браузере

Supabase RLS должен быть включён там, где есть пользовательские данные.
# защита данных на уровне базы

Code Arena Runner нельзя делать до v1.7.
# запуск кода требует отдельной защиты
```

## Правила разработки

Проект нужно делать поэтапно.

Основные правила:

```text
Не делать всё сразу.
# сначала рабочий простой MVP

Не добавлять новую функцию, пока старая не работает.
# стабильность важнее скорости

Не смешивать MVP и будущие режимы.
# Runner и Team Mode не должны попасть в ранний этап

Не хранить секреты в коде.
# только .env.local и Vercel Environment Variables

Не вызывать OpenRouter из браузера напрямую.
# все AI-запросы проходят через backend API

Перед новой функцией проверять старые сценарии.
# новая функция не должна ломать старую

Каждый важный этап фиксировать через Git commit.
```
