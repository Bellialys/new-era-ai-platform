## Краткое описание проекта

**Новая эпоха** - это веб-платформа, где пользователь вводит одну задачу, выбирает несколько AI-моделей и получает ответы рядом для сравнения.

Пользователь сможет:

- отправить один prompt нескольким моделям;
- сравнить ответы в удобном интерфейсе;
- выбрать лучший ответ;
- сохранить результат сравнения;
- смотреть историю прошлых сравнений;
- позже использовать Code Arena, Multi Model Battle, Judge Mode, Leaderboard и AI Team Mode.

Главная идея проекта - не просто получить один ответ от одной модели, а сравнить несколько моделей и понять, какая из них лучше справляется с конкретной задачей.

## Первый главный MVP

Первый главный MVP - это **Stable Prompt Arena**.

Он формируется постепенно от `v0.3` до `v1.0`.

В полный MVP входит:

```text
Главная страница.
# пользователь понимает, что делает проект

Prompt Arena.
# основной режим сравнения моделей

Поле ввода задачи.
# пользователь пишет prompt или задачу

Выбор 2-3 моделей.
# пользователь выбирает модели для сравнения

Отправка запроса через backend API.
# внешний AI API вызывается только на серверной стороне

Карточки ответов.
# ответы моделей отображаются рядом

Голосование.
# пользователь выбирает лучший ответ

Сохранение в Supabase.
# база хранит задачу, ответы и голос

История сравнений.
# пользователь может открыть прошлые сравнения

Деплой на Vercel.
# проект доступен онлайн
```

## Основные API MVP

Минимальные API routes:

```text
GET /api/models
# вернуть список активных моделей

POST /api/compare
# отправить задачу нескольким моделям и вернуть ответы

POST /api/vote
# сохранить выбор лучшего ответа

GET /api/history
# получить историю сравнений

GET /api/history/[taskId]
# получить одно сравнение по taskId
```

Frontend не должен напрямую обращаться к внешнему AI API.

## Канонический контракт `/api/compare`

Документ `09-api-structure.md` является источником истины для будущего backend API.

API-ответ модели должен использовать:

```text
answerText
# текст успешного ответа модели

errorCode
# технический код ошибки модели, если status = error

errorMessage
# понятное сообщение об ошибке модели, если status = error
```

Не использовать как API-контракт:

```text
text
# старое поле, не должно использоваться в API-контракте

modelRole
# роль модели не должна дублироваться в API-ответе /api/compare
```

Правильное разделение:

```text
ArenaApiResponse
# то, что приходит от /api/compare

ArenaResponseView
# UI-представление, куда frontend добавляет modelRole по modelId
```

## Текущее состояние

Сейчас проект находится на этапе:

```text
v0.3 - Static UI MVP
# интерфейс Prompt Arena работает на mock-данных
```

После исправлений кода закрыто:

```text
ArenaApiResponse и ArenaResponseView разделены.
# API и UI больше не смешиваются

Mock-ответы используют answerText.
# совпадает с 09-api-structure.md

modelRole добавляется на клиенте по modelId.
# не дублируется в API-like response

MAX_PROMPT_LENGTH=8000 применён в UI.
# textarea ограничивает ввод, validateForm проверяет верхнюю границу

Старые responses сбрасываются при изменении prompt или моделей.
# UI не показывает устаревший результат

ResponseCard показывает answerText или errorMessage.
# success/error рендерятся корректно
```

Осталось перед `v0.4`:

```text
Создать package-lock.json локально через npm install и закоммитить.
# lock-файл должен быть настоящим, а не вручную угаданным

Локально прогнать npm run typecheck, npm run lint и npm run build.
# после удалённых правок нужно подтвердить зелёный статус

Подключить backend route /api/compare.
# задача v0.4

Подключить реальные ответы через внешний AI API.
# задача v0.4
```

## Правила разработки

Проект нужно делать поэтапно.

Основные правила:

```text
Не делать всё сразу.
# сначала рабочий простой MVP

Не добавлять новую функцию, пока старая не работает.
# стабильность важнее скорости

Не смешивать MVP и будущие режимы.
# Runner и Team Mode не должны попасть в ранний этап

Не хранить секреты в коде.
# только .env.local и Vercel Environment Variables

Не вызывать внешний AI API из браузера напрямую.
# все AI-запросы проходят через backend API

Перед новой функцией проверять старые сценарии.
# новая функция не должна ломать старую

Каждый важный этап фиксировать через Git commit.
# можно откатиться к стабильному состоянию
```
